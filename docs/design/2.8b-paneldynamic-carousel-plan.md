# Task 2.8b — paneldynamic carousel mode (+ progress)

Status: **DRAFT**. Extends 2.8a (stacked on `feat/2.8a-paneldynamic`). Weight M.
Not CORE — a bounded extension of the 2.8a component reusing its established
patterns. `tab` mode stays 2.8c.

## What changes vs 2.8a

2.8a renders `displayMode: "list"` and shows a `paneldynamic-mode-unsupported`
fallback for anything else. 2.8b adds the `"carousel"` branch; `"tab"` remains
unsupported until 2.8c.

## Verified model surface (survey-core 2.5.33)

- `displayMode === "carousel"` → `isRenderModeList` false, `renderedPanels` is a
  SINGLE-element array `[currentPanel]`, `currentIndex` (0-based), `currentPanel`.
- Nav: `goToNextPanel()` / `goToPrevPanel()` (methods); `isPrevButtonShowing` /
  `isNextButtonShowing` (accessors — gate the buttons). Wraps/clamps in core.
- Progress: `progressText` (e.g. "1 of 3"), `showProgressBar` (accessor).
- Reactivity: `currentIndexChangedCallback` fires on `currentPanel`/`currentIndex`
  change — ALREADY assigned to the shared structural handler in 2.8a's
  `attachCallbacks`, so navigation re-renders with no new wiring.

## Design (extend `PanelDynamicQuestion.renderElement`)

- `isRenderModeList` → the existing list render (unchanged).
- else `displayMode === "carousel"` → carousel:
  - a `paneldynamic-carousel` container;
  - the SINGLE current panel via the existing `PanelDynamicItem`
    (`renderedPanels[0]`, keyed by `panel.id` — a nav swaps which panel that is,
    and `currentIndexChangedCallback` re-renders);
  - a nav row: a `paneldynamic-prev` button when `isPrevButtonShowing`
    (→ `goToPrevPanel()`), a `paneldynamic-next` when `isNextButtonShowing`
    (→ `goToNextPanel()`), each a themed Pressable `role="button"`;
  - a `paneldynamic-progress` Text with `progressText` when `showProgressBar`;
  - the same add button + (via the item) remove button as list mode.
- else (`"tab"`) → keep the `paneldynamic-mode-unsupported` fallback + deferred
  diagnostic (2.8c).

No new reactivity, no new diagnostic. Reuses `PanelDynamicItem`, the add-button
renderer, and the callback plumbing from 2.8a.

## Test plan (TDD, red first)

- carousel renders exactly ONE `paneldynamic-panel-*` (the current), not all.
- next button present (isNextButtonShowing), prev absent at index 0; pressing
  next advances `currentIndex` and renders the next panel (via
  currentIndexChangedCallback); at the last panel next is absent, prev present.
- progress text shows `progressText` when `showProgressBar`.
- add/remove still work in carousel.
- `tab` displayMode still → unsupported fallback + diagnostic (2.8c boundary).
- list mode unchanged (regression).

## Wiring

None beyond the component — paneldynamic is already a supported descriptor (2.8a).
DIFFERENCES: carousel is single-panel with prev/next + a text progress indicator;
tab deferred to 2.8c. kitchen-sink: optionally a second paneldynamic in carousel
mode (or leave the 2.8a list one) — parity regen if added.
