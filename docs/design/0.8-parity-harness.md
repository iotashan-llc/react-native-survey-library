# Task 0.8 design — classified-drift API gate (A12)

Status: **final** — one llm-pair round (codex gpt-5.6-sol@max, independent
parallel-draft; Gemini 3.1 Pro unavailable — individual quota reached,
resets ~17h — proceeded on Codex alone per the fallback rule: one usable
peer is sufficient, no Opus). Codex's independently-drafted plan converged
with the coordinator's own analysis on every material point (runtime
reflection, no doc-generator dependency, the same 15-member watchlist
scope, the same breaking/relevant split); the only divergences are noted
below as "adopted / adjusted / rejected." Owner: task agent (0.8, size M,
not `CORE`). Depends on 0.5 (`factories/manifest.ts` establishes the
committed-classified-table + pure-diff-function + live-harvest-test
pattern this task reuses).

## What 0.5 already covers (not duplicated here)

0.5's `src/factories/manifest.ts` is the **runtime-template manifest**
half of A12/0.4: it classifies every survey-core question-TYPE class name
(`Serializer.getChildrenClasses('question', true)`) and cross-checks our
own factory's registered dispatch keys. That is done. This task covers
the **other** half of A12's "classified-drift API gate": the broader
survey-core API surface — classes/singletons/members outside question
types — that this renderer's source (and its own tests) actually binds
to through the facade.

## Why not literally "via surveyjs-doc-generator"

A12's text names `surveyjs-doc-generator` as the mechanism. Investigated
directly (read the tool's source at
`../surveyjs-doc-generator` and its invocation inside
`../survey-library/packages/survey-core/package.json`'s `doc_gen`
script): it is a TypeScript-AST extractor that requires **raw `.ts`
source with JSDoc** (`ts-node` over `entries/chunks/model.ts`) plus a
**built** `survey.core.js` to seed `Serializer` metadata for JSON
property definitions — a development-time step of the survey-core
monorepo itself. The **installed** `survey-core` npm package (our
peerDependency) ships none of that: `node_modules/survey-core` contains
only `survey.core.js` (compiled bundle), `fesm/*.mjs`, and
`typings/**/*.d.ts` — verified directly against the installed 2.5.33
package. Running doc-generator against it is not possible without either
building survey-core from source in CI (heavy, slow, version-fragile —
the reference checkouts `../survey-library` and `../surveyjs-doc-generator`
are read-only, non-dependency references, not something CI can rely on)
or vendoring a doc-generator dependency chain (`ts-node`, an old pinned
`typescript@4.2.4`) this project doesn't otherwise need.

**Deviation, deliberate:** the gate harvests the renderer-relevant surface
via **plain JS runtime reflection** (`Object.getOwnPropertyDescriptor`
walked up the prototype chain) against the survey-core actually resolved
through `src/core/facade.ts` at Jest test time — no TS Compiler API, no
doc-generator, no building survey-core from source. This is not a new
pattern for this codebase: 0.5's `manifest.ts` already established
"reflect on the live installed package at test time"
(`Serializer.getChildrenClasses(...)` called live) as this project's
drift-detection idiom, rather than any static/AST approach. This gate
extends that same idiom to Base/Question/Model/LocalizableString/
RendererFactory instead of re-deriving a heavier mechanism.

## Why not A12's literal `compatible | native-replacement |
intentionally-unsupported` labels

Those three labels classify **our own eventual `<Survey>` public API's**
relationship to upstream `survey-react-ui`'s DOM `<Survey>` component
(`compatible` = same prop/event; `native-replacement` = an RN-native
substitute for a DOM concept; `intentionally-unsupported` = a documented
won't-support). `<Survey>` doesn't exist yet — it's M1 task 1.1, which
comes *after* this Phase-0 task. Applying that vocabulary to "does
`Base.prototype.addOnPropertyValueChangedCallback` still exist in
survey-core" doesn't fit: every one of this gate's entries would trivially
be `compatible` forever (there is no RN-vs-DOM axis for survey-core's own
internals), making the label decorative rather than load-bearing. Codex's
independent draft reached the identical conclusion unprompted and flagged
it as a risk: reusing A12's dispositions here "describe[s] renderer/
upstream feature parity better than dependency-drift severity" and
recommended reserving that vocabulary for **a later gate built alongside
M1's task 1.1**, once `<Survey>`'s own public surface exists to classify.
This design adopts that recommendation. **This task's operative
classification is `breaking | relevant | irrelevant`** (matching what the
task brief specified); A12 should be read as satisfied by two
complementary gates — 0.5/0.8 covering survey-core's own drift now,
and a `<Survey>`-parity gate covering A12's literal three labels once
M1 lands.

## Scope — the watchlist (18 entries)

**Contract (revised in impl review, round 2 — see the review-rounds
section):** every survey-core member this library's PRODUCTION source
binds to through the facade, plus two explicitly named cross-cutting
design-contract/test-infrastructure observables:

- `Model.getQuestionByName` — the universal fixture lookup every test
  suite builds on;
- `Base.hasActiveUISubscribers` — the ONLY observable of 0.4's
  subscription-leak contract (its leak tests are built entirely on this
  getter; nothing in production reads it, but the design contract it
  observes is production behavior).

Fixture-CHOICE test bindings are deliberately NOT watched (see the
exclusions list below). The inventory was re-run against the merged
0.3/0.4/0.5 codebase during impl review (grep of every member access on
facade-imported survey-core objects across `src/**`, production and
tests), which surfaced two production bindings the first pass missed:
`Question.title` and `QuestionCustomWidget.name`.

| Host | Members | Kind |
|---|---|---|
| `Base.prototype` | `addOnPropertyValueChangedCallback`, `removeOnPropertyValueChangedCallback`, `addOnArrayChangedCallback`, `removeOnArrayChangedCallback` | method ×4 |
| `Base.prototype` | `hasActiveUISubscribers` | accessor ×1 |
| `Question.prototype` | `getTemplate`, `getComponentName`, `isDefaultRendering`, `getType` | method ×4 |
| `Question.prototype` | `customWidget`, `isInputReadOnly`, `name`, `title` | accessor ×4 |
| `QuestionCustomWidget` (probe instance) | `name` | data ×1 |
| `Model.prototype` | `getQuestionByName` | method ×1 |
| `LocalizableString.prototype` | `renderedHtml` | accessor ×1 |
| `RendererFactory.Instance` | `registerRenderer`, `unregisterRenderer` | method ×2 |

`QuestionCustomWidget.name` is a TS constructor parameter-property — an
OWN data property on each INSTANCE, invisible on the prototype — so its
`resolveHost` constructs a probe instance
(`new sc.QuestionCustomWidget('__api-surface-probe__', {})`), verified
side-effect-free against the installed package (the constructor only
assigns instance fields; it does not register into
`CustomWidgetCollection` — probed: collection size stays 0). Its
`expectedKind` is the `data` kind introduced by the round-2 kind
expansion below.

Verified empirically (Node probe against the installed 2.5.33 package)
that every entry above resolves correctly via prototype-chain walk, and
that **ownership is not always where you'd guess** — e.g.
`getTemplate`'s own property lives on `Base.prototype`, not
`Question.prototype`, and `Question.prototype.name` is actually owned by
an intermediate `SurveyElement` class — which is exactly why the harvester
walks the full prototype chain rather than checking only the leaf class's
own properties (a leaf-only check would false-positive "missing" on
inherited members).

**Deliberately excluded:**
- `Serializer.getChildrenClasses`, `ComponentCollection.Instance.add`/
  `.remove` — already exercised live by 0.5's `manifest.test.ts`; adding
  them here would duplicate that gate.
- `IPropertyValueChangedEvent`/`IPropertyArrayValueChangedEvent` — **type-only
  interfaces, erased at runtime; there is no object to reflect on.** Their
  real-world shape (`.name` present on the callback's `options` argument)
  is already exercised behaviorally by 0.4's `SurveyElementBase.test.tsx`
  (real subscribe → real mutate → assert `options.name`). Not duplicated
  here; noted so the gap is visible rather than silently absent.
- **Fixture-choice test bindings** (revised contract, impl review round
  2): members a single test suite reads/writes purely as interchangeable
  exercise props, where the suite itself fails loudly if the member
  drifts — the failing suite IS the detection, and the classification is
  obvious from which suite failed. Enumerated for visibility:
  `Question.indent`/`readOnly` and checkbox `choices` (0.4 reactivity
  fixtures — any serialized property would do), `Model.getValue`/
  `setValue`/`state`/`currentPageNo`/`completeLastPage` (0.3 shim smoke
  test), `Model.dispose` (manifest test cleanup),
  `CustomWidgetCollection.Instance.clear`/`.addCustomWidget` (customWidget
  behavioral-test setup — a CONSUMER-side registration API our production
  code never calls; the production binding those tests exercise,
  `Question.customWidget` + `QuestionCustomWidget.name`, IS watched).
- The future `<Survey>` component's own props/events — doesn't exist yet
  (M1).

**Also documented as mechanism limits** (impl review round 2, minor):
- `WatchedApiMember.member` is a plain `string` — no current renderer
  binding is Symbol-keyed. Widen the type (and the walk) if a future
  binding ever reads a Symbol-keyed member.
- Signature/semantic changes WITHIN a surviving function (arity, argument
  meaning, return shape) are out of scope for this reflection gate — the
  behavioral suites (0.3 shim smoke, 0.4 reactivity, 0.5 registration)
  own those.

**Known, accepted limitation** (flagged independently by both the
coordinator and Codex): this watchlist is **hand-maintained**, not
auto-discovered. If a future milestone starts calling a new survey-core
member without adding a watchlist row, that member's drift goes
undetected by this gate (though a real removal would usually also surface
as a TypeScript compile error against the `.d.ts` typings, which is a
separate, complementary safety net). Extend the table — same convention
0.5 uses for its descriptor table — whenever new code starts binding to a
new survey-core member.

## Mechanism

`src/core/api-surface.ts` (main `src/` tree, not `__tests__/` — mirrors
0.5's `manifest.ts` placement; putting it under `__tests__/` would make
Jest's default `**/__tests__/**` test-file matcher try to execute it as a
test suite with zero `it()` blocks, which fails):

```ts
export interface WatchedApiMember {
  id: string;                 // "Base.addOnPropertyValueChangedCallback"
  member: string;              // "addOnPropertyValueChangedCallback"
  expectedKind: 'method' | 'accessor' | 'data';
  resolveHost: (sc: FacadeModule) => unknown;   // -> prototype, singleton, or probe instance
  reason: string;               // why the renderer depends on it
}

export const API_SURFACE_WATCHLIST: readonly WatchedApiMember[] = [ ...18... ];

export function harvestApiSurface(sc, watchlist?): HarvestedApiMember[]
export function diffApiSurface(watchlist, harvested): { breaking: string[]; relevant: string[] }
```

- `harvestApiSurface` resolves each entry's host via `resolveHost(sc)`,
  **wrapped in try/catch** (Codex's addition, adopted: a removed
  top-level export like `sc.Base` being `undefined` would otherwise throw
  a raw `TypeError` reading `.prototype` instead of cleanly reporting
  `'missing'`), then walks the prototype chain with
  `Object.getOwnPropertyDescriptor` and maps every descriptor shape to a
  **distinct** kind (revised in impl review round 2 — the original
  design's "anything else is `'accessor'`" fallback let an accessor
  silently downgraded to a plain data property or a setter-only stub keep
  the gate green):
  - getter present (with or without setter) -> `'accessor'`
  - setter WITHOUT getter -> `'setter-only'` (every read yields undefined)
  - function-valued data descriptor -> `'method'`
  - any other data descriptor, including `value: undefined` -> `'data'`
  - not found anywhere on the chain -> `'missing'`
- `diffApiSurface` is a **pure function** — no survey-core access — so it
  is unit-tested with synthetic watchlist/harvest fixtures, independent of
  the real package. Classification (matrix revised in round 2):
  - **breaking** — the read/call contract is structurally dead:
    `'missing'` (gone; call/read throws); `'setter-only'` (reads yield
    undefined — dead binding despite a surviving descriptor); `'data'`
    where computed behavior (`method`/`accessor`) was expected — a method
    call site throws, an accessor read silently returns a frozen or
    undefined value with the getter's logic gone (silent misbehavior is
    treated at breaking severity).
  - **relevant**: the member survives WITH computed/callable behavior but
    in a different shape (method <-> accessor swaps, or an expected plain
    `data` member that became computed) — won't structurally die, but the
    call/read site needs human review.
  - **irrelevant**: anything not on the watchlist. Never harvested, never
    surfaced — this is what keeps the gate from becoming "survey-core's
    entire API, diffed" (hundreds of classes) and is the mechanism behind
    "CI fails on unclassified drift": any watched-member diff (breaking
    or relevant) fails the live gate test until a developer reviews it
    and updates `API_SURFACE_WATCHLIST` to the new, deliberately-reviewed
    expectation — same precedent as 0.5's "genuinely new upstream class
    name" gate.

`src/core/__tests__/api-surface.test.ts` (one file, mirroring
`manifest.test.ts`'s single-file pure+live pairing):
1. `diffApiSurface` unit tests against synthetic fixtures — no drift, a
   removed member (breaking), method -> accessor and accessor -> method
   swaps (relevant), accessor -> data / accessor -> setter-only /
   method -> data downgrades (breaking; round 2), data -> accessor
   (relevant; round 2), a harvest missing an id entirely (breaking).
2. `harvestApiSurface` reflection-helper tests — a throwing `resolveHost`
   and a nullish resolved host both report `'missing'`, not a crash;
   distinct kinds for non-function data properties (including
   `value: undefined`), setter-only descriptors, and getter-only vs.
   getter+setter accessors (round 2).
3. **Live-independent end-to-end synthetic drift** (round 2): a watched
   accessor whose synthetic host carries an undefined-valued data
   property, and one whose host carries a setter-only descriptor, each
   run through the FULL harvest+diff pipeline and must fail the gate as
   `breaking` — these pin the exact hole the round-2 review found,
   without touching the installed package.
4. **Source-inventory contract** (round 2): the watchlist must contain
   `Question.title`, `QuestionCustomWidget.name`, and
   `Base.hasActiveUISubscribers` — pins the re-run inventory's findings
   so they cannot be silently dropped.
5. **The live gate**: `harvestApiSurface(facade, API_SURFACE_WATCHLIST)` →
   `diffApiSurface(...)` → assert `{ breaking: [], relevant: [] }` against
   the actually-installed survey-core.
6. **End-to-end simulated-drift proof**: temporarily `delete`s
   `Question.prototype.getComponentName` (confirmed a direct own property
   of `Question.prototype`, not inherited — avoids the inheritance
   ambiguity that ownership varies per member, e.g. `getTemplate` is
   actually owned by `Base.prototype`), re-runs the live harvest+diff
   pipeline, asserts it reports the deletion as `breaking`, restores the
   original descriptor in `finally`. This is the strongest proof the gate
   mechanism (not just the pure diff function) reacts to real drift.

## What Codex caught / what was adjusted / what was rejected

- **Adopted verbatim**: runtime reflection over doc-generator/TS-Compiler-API
  (independently converged); the exact same 15-member watchlist scope
  (independently converged — strong cross-check); breaking vs.
  review-required(relevant) vs. out-of-scope(irrelevant) classification
  criteria (independently converged); wrapping `resolveHost` in try/catch
  so a fully-removed top-level export reports cleanly instead of throwing
  (Codex's addition — adopted); one live-baseline test + one end-to-end
  simulated-drift test with restore-in-`finally`, non-concurrent (Codex's
  addition — adopted); the conclusion that A12's three-label vocabulary
  belongs to a later `<Survey>`-parity gate, not this one (Codex reached
  this independently — strong convergence, adopted).
- **Adjusted**: Codex proposed both the logic module and the test under
  `src/core/__tests__/`; rejected in favor of the established `src/core/`
  (non-`__tests__`) placement 0.5's `manifest.ts` already uses — Jest's
  default `**/__tests__/**` matcher would otherwise try to run a
  zero-`it()` module as a test suite and fail. Codex also proposed a
  per-entry `disposition` field reusing A12's compatible/native-replacement/
  intentionally-unsupported labels alongside a separate computed impact
  axis; simplified away since every entry would always be `'compatible'`
  today (a field with one ever-used value is exactly the kind of
  speculative structure this task's "no speculative framework" directive
  rejects) — kept only the plain `reason` string (matching 0.5's
  `PlannedOrUnsupportedDescriptor.reason` naming).
- **Not resolved, accepted as a known limitation**: watchlist staleness if
  future code binds to a new survey-core member without a table update
  (both the coordinator and Codex flagged this independently; no
  auto-discovery mechanism is being built for an M-sized task — TypeScript
  compilation against `.d.ts` remains a complementary safety net).
- **Gemini**: unavailable (quota exhausted, resets ~17h) — proceeded on
  Codex alone per the availability/fallback rule (one usable peer is
  sufficient; no Opus fallback needed since Codex succeeded).

## Impl review, round 2 (Codex terra@medium on PR #5) — 4 findings, all addressed

1. **[major] Kind-classification hole** — the harvester's "neither
   function-valued-data nor getter -> `'accessor'`" fallback meant a
   watched accessor replaced by a NON-function data property (including
   `value: undefined`) or by a setter-only stub still classified as an
   unchanged accessor, keeping the live gate green through a real
   downgrade. Fixed: every descriptor shape now maps to a distinct kind
   (`method` / `accessor` / `setter-only` / `data` / `missing`;
   getter-only and getter+setter both report `accessor` — the read
   contract production depends on is the getter), and the classification
   matrix routes structurally-dead shapes (`missing`, `setter-only`, and
   `data`-where-computed-expected) to `breaking`. Pinned by
   live-independent synthetic tests in both the pure-diff and
   full-pipeline suites (TDD: 11 tests observed RED against the old
   implementation first).
2. **[major] `Question.title` unwatched** — a production binding
   (UnsupportedQuestion's default presentation) with no row. Added as an
   accessor entry. The demanded inventory re-run (grep of every member
   access on facade-imported survey-core objects across `src/**`) also
   surfaced `QuestionCustomWidget.name` (production:
   QuestionElementBase's custom-widget-ignored diagnostic reads
   `widget.name`) — added with a probe-instance host and the new `data`
   expected-kind, since it is a constructor parameter-property invisible
   on the prototype.
3. **[major] Tests-in-scope claim vs. absent test bindings** — resolved
   by picking the "narrow the documented contract" branch, applied
   consistently: the contract is now production bindings + two named
   cross-cutting observables (`Model.getQuestionByName`,
   `Base.hasActiveUISubscribers` — the latter watched per the review's
   explicit requirement, as 0.4's only leak observable). All
   fixture-choice test bindings — including the review's other two
   candidates (`CustomWidgetCollection.Instance.clear`/`.addCustomWidget`,
   `Model.dispose`) AND the ones the review itself did not enumerate
   (`Question.indent`/`readOnly`, checkbox `choices`, the shim smoke
   test's five Model members) — are documented exclusions with the
   self-detection rationale. Adding only the review's three while
   claiming tests-in-scope would have recreated the same inconsistency
   one level deeper.
4. **[minor] Documented mechanism limits** — string-keyed members only
   (no current binding is Symbol-keyed), and signature/semantic changes
   within a surviving function are out of scope (behavioral suites own
   those). Noted here and in the module header.

## Non-goals

- No re-derivation of 0.5's question-type/dispatch-key manifest.
- No `<Survey>`-vs-upstream-props parity gate (M1, once `<Survey>` exists).
- No automatic watchlist discovery / full survey-core API enumeration.
- No CI workflow changes — this gate is a Jest test, run by the existing
  `yarn test` job.
