# Task 1.2 design — native lifecycle bridge: ref/layout registry + focus/scroll interception (A15)

Status: draft for peer review. Owner: orchestrator. Foundation for 1.1 (`<Survey>` installs the bridge), consumed by every focusable component. Source facts verified against survey-core v2.5.33.

## Verified upstream facts the design rests on

- **`SurveyModel.scrollElementToTop` (survey.ts:5838-5904) is the SINGLE funnel** for every scroll request in the library: invalid-submit error focus (`Question.focus` → it), programmatic `focusQuestion()`, page-change scroll (`PageModel.scrollToTop`), panel expand. It fires `onScrollToTop` — `{element, question, page, elementId, cancel, allow}` — **BEFORE any DOM API or the unguarded `settings.environment` destructure**; `options.allow=false` (or `cancel=true`) fully short-circuits the DOM block. (`onScrollingElementToTop` is a deprecated alias of the SAME EventBase — one subscription covers both names.)
- **Canceling drops the chained callback**: `Question.focus()` passes `onScolledCallback: () => focusInputElement(onError)` — a canceling bridge must drive focus itself. In RN this is moot-but-mandatory anyway: `SurveyElement.FocusElement` returns false immediately when `document` is undefined (DomDocumentHelper gate) — **core can never focus a native input; only the bridge can**.
- The event does NOT expose `onScolledCallback`, but it does carry `question` — question non-null distinguishes focus-intent (question focus) from scroll-only intent (`page.scrollToTop` passes `question: null`). This distinction drives the bridge's focus decision.
- `settings.environment` is `undefined` in RN (built only `if (!!document)`); `scrollElementToTop` destructures it UNGUARDED (survey.ts:5872) — reached only if the event was NOT canceled. Other readers (`dom-utils.ts:10`, dragdrop) are outside the focus/scroll path.
- Cross-page focus is two-phase: `focusQuestionByInstance` sets `focusingQuestionInfo`, changes `currentPage`, and the page-render completion path (`scrollToTopOnPageChange`, survey.ts:3816-3826 — invoked via `setTimeout(…, 1)` from afterRenderPage-adjacent code) re-enters the funnel; `autoFocusFirstQuestion`/`autoFocusFirstError` are plain model properties — no interception needed, they just route through the funnel.
- Public DOM-free helpers the bridge uses as-is: `question.focusIn()` (fires `onFocusInQuestion`, sets `lastActiveQuestion` — in web this is driven by a DOM focus-bubble handler; **the RN bridge must call it** on native focus), `expandAllParents()`, `panel.getFirstQuestionToFocus/focusFirstErrorQuestion`, `survey.getAllQuestions`.
- `survey-react-ui`'s `afterRenderSurvey(el)` (sets `SurveyModel.rootElement` + DOM listeners) is simply **never called** in RN — it early-returns safely and nothing requires it (design decision: don't fabricate a rootElement; the funnel is owned via the event).

## Architecture — three pieces

### 1. `src/lifecycle/registry.ts` — ref/layout registry

Per-survey instance (created by the bridge, provided via `LifecycleContext` from `<Survey>`):

- `registerElement(el: Question | PanelModel | PageModel, handle: ElementHandle): () => void` — returns deregister; keyed by model INSTANCE (Map<Base, ElementHandle>; entries removed on unmount via 0.4's captured-pair mounted hooks — `QuestionElementBase.onQuestionMounted/WillUnmount` are the natural registration points, no new lifecycle surface).
- `ElementHandle = { containerRef: RefObject<View>; focusFirst?: () => boolean; a11yFocus?: () => void }` — `focusFirst` supplied by input-bearing components (TextInput.focus() etc., returns whether focus landed); non-input questions omit it (bridge falls back to a11y focus on the container: `AccessibilityInfo.setAccessibilityFocus`).
- `registerScrollHost(handle: ScrollHostHandle)` — the Survey root's ScrollView: `{ scrollTo(y, animated), measureTarget(containerRef): Promise<{y}> }` (measureLayout against the ScrollView node). One host per survey; nested scroll hosts (paneldynamic internal lists) are M2+ (documented).
- Lookup order for a scroll target: exact model instance → its `page` (when the question isn't registered, e.g. virtualized away) → null (diagnostic `lifecycle-target-unregistered`, dev-only, once per instance).

### 2. `src/lifecycle/bridge.ts` — the interception core

`installLifecycleBridge(survey: SurveyModel, registry): () => void` (returns uninstall; called from `<Survey>` mount/model-swap/unmount — 1.1 owns the call sites):

- Subscribes `survey.onScrollToTop`. Handler: `options.allow = false` ALWAYS (native ownership — never let core reach the environment destructure), then asynchronously:
  1. Resolve target via registry (question → element → page fallback).
  2. `scrollIfVisible` honored: if the handle's container is already fully inside the viewport (measure vs scroll host viewport) and `options.scrollIfVisible !== true`, skip the scroll (mirrors core's semantic).
  3. Scroll: `measureTarget` → `scrollTo(y - topInset, animated: true)`.
  4. **Focus-intent completion** (the dropped `onScolledCallback` replacement): if `options.question` non-null → after scroll settles (scrollTo has no completion callback in RN; use a bounded `setTimeout` of ~300ms, documented approximation) → `handle.focusFirst?.()` — if it returns true, call `question.focusIn()` (event parity); else `handle.a11yFocus?.()` (accessibility focus so screen readers land on the errored question).
- **Re-entrancy/coalescing**: multiple scroll requests in one validation pass (N invalid questions each calling focus? — no: core focuses only the FIRST error via ValidationContext) — still, requests within one tick coalesce to the LAST (matches web behavior where later scrollIntoView wins).
- Consumer parity: because `allow=false` is set by the FIRST subscriber (the bridge), consumer handlers subscribed later still see the event (EventBase fires all handlers) — the bridge subscribes at install time (before consumer props wiring in 1.1), and consumers may re-allow? NO — documented: `onScrollToTop` remains observable for consumers but scroll ownership is the bridge's; a consumer wanting to suppress the native scroll uses the 1.1-level event prop that the bridge consults (`onScrollToElement` RN-level event with `preventDefault()` semantics) rather than fighting the core event. (DIFFERENCES.md entry.)
- **The consult seam (approved amendment)**: `LifecycleBridgeOptions.onScrollRequest?: (info: { element: Base; question: Question | null; page: PageModel | null; viaPageFallback: boolean }) => boolean` — the bridge calls it before every native scroll (after target resolution, before measure/scroll). Return `false` = suppress the NATIVE SCROLL ONLY; **focus-intent completion STILL RUNS on suppression** (focusFirst/a11yFocus + `question.focusIn()` — pinned semantics: web's closest analog is scroll suppressed but focus happens). Return `true`/absent = normal path. A throwing callback is contained (logged) and treated as `true`. 1.1's `onScrollToElement` prop is built directly on this signature.
- **Event-field reality check (implementation note)**: the fired `ScrollToTopEvent` options do NOT include `scrollIfVisible` (survey.ts builds `{element, question, page, elementId, cancel, allow}` only; `scrollIfVisible` stays a local). Every internal core caller of the funnel passes `scrollIfVisible: false` (question focus, panel expand, page scrollToTop), so the bridge implements that semantic unconditionally: skip the scroll when the target is already fully visible in the host viewport (focus still runs).
- Uninstall removes the subscription and clears the registry (model-swap safety — 0.4's D1 pattern at the survey level).

### 3. `src/core/shim.ts` amendment — `settings.environment` stub (defense-in-depth)

The bridge makes the destructure unreachable; the stub makes it non-fatal if any OTHER core path reaches it (current known readers outside focus/scroll: dom-utils shadow-DOM helper, dragdrop adapter — both no-op-safe with undefined fields). Amendment to 0.3's shim: `applySurveyCoreShims()` additionally sets `settings.environment ??= { root: undefined, rootElement: undefined, popupMountContainer: undefined, svgMountContainer: undefined, stylesSheetsMountContainer: undefined } as any` — fields undefined (downstream is optional-chained: `surveyRootElement?.querySelector` skips), object non-undefined (destructure survives). **Never define `document`** (0.3 invariant unchanged). 0.3 design doc gets an amendment note + the shim's packaged tests extend to assert the stub.

Risk note (for review): code that TRUTHINESS-checks `settings.environment` (rather than its fields) will now take the "environment exists" branch — the known readers destructure rather than gate, but the review should sweep for `if (settings.environment)` / `!!settings.environment` patterns.

## What the bridge deliberately does NOT do

- No `survey.rootElement` fabrication (afterRenderSurvey never called; core's rootElement stays undefined; the funnel is owned via the event, not via fake DOM nodes).
- No keyboard-avoidance logic in v1 (scroll target math is plain measureLayout; keyboard-aware offset composition documented as a 1.10 concern where TextInput focus meets the keyboard).
- No nested scroll hosts (M2 paneldynamic).
- No monkeypatching of `FocusElement`/`focusInputElement` (they no-op safely; the bridge REPLACES their role, doesn't fix them).

## Test plan (TDD; real SurveyModel via facade, mocked handles)

1. **Invalid submit** (the A15 headline): survey with required question, `tryComplete()` → bridge receives event with `question`, sets `allow=false` (spied), native scroll called with measured target, `focusFirst` called after settle, `question.focusIn()` fired (assert `onFocusInQuestion`), and — the regression that motivated A15 — **no TypeError** with `settings.environment` undefined (bridge path) AND with the stub (shim path).
2. **Scroll-only intent**: `page.scrollToTop()` → scroll, NO focusFirst call (question null).
3. **Cross-page focus**: `focusQuestion(name)` for a question on another page → currentPage changes; simulate 1.1's render-complete call (`scrollToTopOnPageChange`) → funnel re-entered → target scrolled+focused on the new page. `autoFocusFirstError` variants for multi-page validation (`focusOnFirstError` → page change without focus when autoFocus off — ValidationContext branch).
4. **scrollIfVisible semantics**: visible target + scrollIfVisible false → no scroll, focus still runs.
5. **Registry lifecycle**: unmount deregisters (0.4 captured-pair hooks); post-uninstall events don't touch handles; model-swap installs fresh (old survey's events inert).
6. **Fallbacks**: unregistered question → page fallback scroll + diagnostic once; no focusFirst → a11yFocus; no scroll host → no-op + diagnostic (never throw — invariant 9 spirit).
7. **Coalescing**: two same-tick requests → one scroll to the last target.
8. **Shim amendment**: environment stub idempotent, fields undefined, packaged-entry assertion; sweep test greps installed survey-core bundle for `settings.environment` truthiness-gates (locks the risk note with evidence).
9. **Consumer observability**: a consumer `onScrollToTop` handler still fires (sees `allow` already false).
10. **Consult seam (`onScrollRequest`)**: returns `false` → no native `scrollTo`, focus-intent completion still fires (focusFirst + `focusIn`); returns `true`/absent → normal scroll path; receives `{element, question, page, viaPageFallback}` with the resolved target; a throwing callback is contained and treated as `true`.

## Sequencing

1.2 ships registry + bridge + shim amendment + tests with a minimal test-only host harness. 1.1 wires: bridge install/uninstall on mount/model-swap/unmount, ScrollView host registration, the render-complete call that drives `scrollToTopOnPageChange`, and the consumer-facing `onScrollToElement` event prop. 1.5/1.7+ components register handles via the 0.4 mounted hooks as they land.
