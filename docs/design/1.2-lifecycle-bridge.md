# Task 1.2 design — native lifecycle bridge: ref/layout registry + focus/scroll interception (A15)

Status: draft for peer review. Owner: orchestrator. Foundation for 1.1 (`<Survey>` installs the bridge), consumed by every focusable component. Source facts verified against survey-core v2.5.33.

## Verified upstream facts the design rests on

- **`SurveyModel.scrollElementToTop` (survey.ts:5838-5904) is the SINGLE funnel** for every scroll request in the library: invalid-submit error focus (`Question.focus` → it), programmatic `focusQuestion()`, page-change scroll (`PageModel.scrollToTop`), panel expand. It fires `onScrollToTop` — `{element, question, page, elementId, cancel, allow}` — **BEFORE any DOM API or the unguarded `settings.environment` destructure**; `options.allow=false` (or `cancel=true`) fully short-circuits the DOM block. (`onScrollingElementToTop` is a deprecated alias of the SAME EventBase — one subscription covers both names.)
- **Canceling drops the chained callback**: `Question.focus()` passes `onScolledCallback: () => focusInputElement(onError)` — a canceling bridge must drive focus itself. In RN this is moot-but-mandatory anyway: `SurveyElement.FocusElement` returns false immediately when `document` is undefined (DomDocumentHelper gate) — **core can never focus a native input; only the bridge can**.
- **Focus-intent discrimination (corrected, review round 2 #2)**: the event does NOT expose `onScolledCallback`, and `question` non-null is NOT sufficient — the THREE internal funnel callers (empirically probed against v2.5.33) are:

  | Caller | element | question | id | callback | intent |
  |---|---|---|---|---|---|
  | `Question.focus()` (question.ts:1618) | q | q | `q.id` | `onScolledCallback` → `focusInputElement` | **focus** |
  | `panel.expand()` (panel.ts:2499) | q (first question) | q | `q.inputId` (= `id + "i"`) | none | scroll-only (web never focuses here either) |
  | `page.scrollToTop()` (page.ts:336) | page | null | page.id | none | scroll-only |

  The bridge's discriminator: **focus intent ⇔ `question != null && elementId === question.id`**. Web parity, not a divergence: panel expand scrolls to the first question but never focuses its input (no callback). The `Question.inputId` shape and the two question-carrying caller shapes are pinned by real-caller tests + the api-surface watchlist.
- `settings.environment` is `undefined` in RN (built only `if (!!document)`); `scrollElementToTop` destructures it UNGUARDED (survey.ts:5872) — reached only if the event was NOT canceled. NOTE: cancellation must be IRREVERSIBLE — `EventBase.fire` hands the SAME mutable options object to later subscribers (event.ts:17-23), so a plain `allow = false` could be reassigned by a consumer handler; the bridge locks `allow` via a defineProperty getter (setter no-ops + `allow-override-ignored` diagnostic).
- **Cross-page focus (corrected, review round 2 #3)**: `focusQuestionByInstance` parks `focusingQuestionInfo` and changes `currentPage`; web's `afterRenderPage` (survey.ts:5514-5519) then **SKIPS `scrollToTopOnPageChange` while a focus is parked and calls PRIVATE `focusQuestionInfo()`** (which runs `question.focus(onError)` → the funnel) — `focusQuestionInfo()` is invoked unconditionally, `scrollToTopOnPageChange` only when nothing is parked and not in design mode. 1.1's render-complete seam mirrors that exact logic: `if (survey.focusingQuestionInfo) { (survey as any).focusQuestionInfo(); } else { survey.scrollToTopOnPageChange(); }`. The private-method dependency is deliberate and documented; its existence is pinned by an api-surface watchlist row (`SurveyModel.focusQuestionInfo`) so core drift fails loudly. `autoFocusFirstQuestion`/`autoFocusFirstError` are plain model properties — no interception needed, they route through the funnel.
- Public DOM-free helpers used as-is: `question.focusIn()` (fires `onFocusInQuestion`, sets `lastActiveQuestion` — web drives it from a DOM focus-bubble handler; **in RN it is COMPONENT-owned, fired from the native input's actual `onFocus` event per the `ElementHandle.focusFirst` contract — the bridge never synthesizes it**, review round 2 #5), `expandAllParents()`, `panel.getFirstQuestionToFocus/focusFirstErrorQuestion`, `survey.getAllQuestions`.
- `survey-react-ui`'s `afterRenderSurvey(el)` (sets `SurveyModel.rootElement` + DOM listeners) is simply **never called** in RN — it early-returns safely and nothing requires it (design decision: don't fabricate a rootElement; the funnel is owned via the event).

## Architecture — three pieces

### 1. `src/lifecycle/registry.ts` — ref/layout registry

Per-survey instance (created by the bridge, provided via `LifecycleContext` from `<Survey>`):

- `registerElement(el: Question | PanelModel | PageModel, handle: ElementHandle): () => void` — returns deregister; keyed by model INSTANCE (Map<Base, ElementHandle>; entries removed on unmount via 0.4's captured-pair mounted hooks — `QuestionElementBase.onQuestionMounted/WillUnmount` are the natural registration points, no new lifecycle surface).
- `ElementHandle = { containerRef: RefObject<View>; focusFirst?: () => boolean; a11yFocus?: () => void }` — `focusFirst` supplied by input-bearing components (TextInput.focus() etc., returns whether focus landed); non-input questions omit it (bridge falls back to a11y focus on the container: `AccessibilityInfo.setAccessibilityFocus`).
- `registerScrollHost(handle: ScrollHostHandle)` — the Survey root's ScrollView: `{ scrollTo(y, animated), measureTarget(containerRef): Promise<{y}> }` (measureLayout against the ScrollView node). One host per survey; nested scroll hosts (paneldynamic internal lists) are M2+ (documented).
- Lookup order for a scroll target: exact model instance → its `page` (when the question isn't registered, e.g. virtualized away) → null (diagnostic `lifecycle-target-unregistered`, dev-only, once per instance).

### 2. `src/lifecycle/bridge.ts` — the interception core

`installLifecycleBridge(survey: SurveyModel, registry): () => void` (returns uninstall; called from `<Survey>` mount/model-swap/unmount — 1.1 owns the call sites):

- Subscribes `survey.onScrollToTop`. Handler: LOCK `options.allow` to `false` ALWAYS (defineProperty getter — irreversible for the dispatch, see the verified-facts note; native ownership — never let core reach the environment destructure), then asynchronously:
  1. Resolve target via registry (question → element → page fallback).
  2. Visible-skip: if the handle's container is already fully inside the viewport (measure vs scroll host viewport), skip the scroll (see "Event-field reality check" for the `scrollIfVisible` divergence).
  3. Scroll: `measureTarget` → `scrollTo(y - topInset, animated: true)`.
  4. **Focus-intent completion** (the dropped `onScolledCallback` replacement): if the request is FOCUS INTENT (discriminator above) → after scroll settles (scrollTo has no completion callback in RN; use a bounded `setTimeout` of ~300ms, documented approximation) → `handle.focusFirst?.()`, falling back to `handle.a11yFocus?.()` when absent or `false` (accessibility focus so screen readers land on the errored question). The bridge only INITIATES focus — **`question.focusIn()` is component-owned** (fired from the input's real `onFocus`; the a11y fallback is container focus, no focusIn).
- **Re-entrancy/coalescing/async ordering (review round 2 #4)**: requests within one tick coalesce — the SCROLL TARGET last-wins (matches web behavior where the later scrollIntoView wins), while the FOCUS INTENT coalesces SEPARATELY (a later scroll-only request must not drop a pending focus; a later focus intent replaces an earlier one). A monotonic request generation is captured at flush start and re-checked after every `await`: a stale measurement completion is dropped (never scrolls after a fresher request already handled the funnel). A request arriving during a settle window cancels the pending settle timer and re-parks its focus intent (superseded by the newer request's own focus intent, if any) — no leaked timers, no double-completion.
- Consumer parity: because `allow` is locked false by the FIRST subscriber (the bridge), consumer handlers subscribed later still see the event (EventBase fires all handlers) and read `allow === false` — and consumers CANNOT re-allow: the lock's setter ignores writes (surfacing the `allow-override-ignored` diagnostic once per install). `onScrollToTop` remains observable but scroll ownership is the bridge's; a consumer wanting to suppress the native scroll uses the 1.1-level event prop that the bridge consults (`onScrollToElement` RN-level event with `preventDefault()` semantics) rather than fighting the core event. (DIFFERENCES.md entry.)
- **The consult seam (approved amendment)**: `LifecycleBridgeOptions.onScrollRequest?: (info: { element: Base; question: Question | null; page: PageModel | null; viaPageFallback: boolean }) => boolean` — the bridge calls it before every native scroll (after target resolution, before measure/scroll). Return `false` = suppress the NATIVE SCROLL ONLY; **focus-intent completion STILL RUNS on suppression** (focusFirst/a11yFocus + `question.focusIn()` — pinned semantics: web's closest analog is scroll suppressed but focus happens). Return `true`/absent = normal path. A throwing callback is contained (logged) and treated as `true`. 1.1's `onScrollToElement` prop is built directly on this signature.
- **Event-field reality check (wording corrected, review round 2 #8)**: the fired `ScrollToTopEvent` options do NOT include `scrollIfVisible` (survey.ts builds `{element, question, page, elementId, cancel, allow}` only; `scrollIfVisible` stays a local, verified against survey-events-api.ts:1078-1105). Observed caller patterns: `panel.expand()` and `page.scrollToTop()` pass `scrollIfVisible: false`; `Question.focus(onError, scrollIfVisible?)` FORWARDS its public optional argument, where `true` means force-scroll even when the target is visible (survey-element.ts `IsNeedScrollIntoView`). Because the flag is lost on the event, the bridge always applies the `false` semantic — skip the scroll when the target is already fully visible in the host viewport (focus still runs). **Documented RN divergence: `Question.focus(…, scrollIfVisible: true)` force-scroll is lost** (the focus itself still completes); if a supported way to carry the flag appears upstream, revisit. (DIFFERENCES.md entry.)
- Uninstall removes the subscription and clears the registry (model-swap safety — 0.4's D1 pattern at the survey level).

### 3. `src/core/shim.ts` amendment — `settings.environment` stub (defense-in-depth)

The bridge makes the destructure unreachable; the stub makes the destructure itself non-fatal if any other path reaches it. Amendment to 0.3's shim: `applySurveyCoreShims(settings?)` (the FACADE passes survey-core's `settings` back post-evaluation — shim.ts keeps zero imports, `/shim` stays zero-core-import) sets `settings.environment ??= { root: undefined, rootElement: undefined, popupMountContainer: undefined, svgMountContainer: undefined, stylesSheetsMountContainer: undefined }`. **Never define `document`** (0.3 invariant unchanged). 0.3 design doc gets an amendment note + the shim's packaged tests extend to assert the stub.

**NARROW contract (corrected, review round 2 #6)**: the stub protects exactly the destructures of the environment OBJECT ITSELF — the scroll funnel's `const { rootElement } = settings.environment` (survey.ts:5872; downstream optional-chained) and dom-utils' `const { root } = settings.environment` (untouched for non-string `getElement` arguments). It does NOT make DOM-only paths safe: drag-drop dereferences `settings.environment.root.documentElement` (dragdrop/dom-adapter.ts) and popup mounting calls `getElement(settings.environment.popupMountContainer).appendChild` (popup-view-model.ts) — both still throw on the undefined fields and are explicitly UNSUPPORTED (the renderer never enters them: no DnD, native popups instead of DOM popups). Negative tripwires in environment-stub.test.ts pin that unsupported-ness so a future "fix the stub with fake objects" forces a contract reconsideration.

Risk note (locked by the sweep test): code that TRUTHINESS-checks `settings.environment` (rather than its fields) will now take the "environment exists" branch — the known readers destructure rather than gate; the sweep covers BOTH installed artifacts (CJS `survey.core.js` and ESM `fesm/survey-core.mjs`).

## What the bridge deliberately does NOT do

- No `survey.rootElement` fabrication (afterRenderSurvey never called; core's rootElement stays undefined; the funnel is owned via the event, not via fake DOM nodes).
- No keyboard-avoidance logic in v1 (scroll target math is plain measureLayout; keyboard-aware offset composition documented as a 1.10 concern where TextInput focus meets the keyboard).
- No nested scroll hosts (M2 paneldynamic).
- No monkeypatching of `FocusElement`/`focusInputElement` (they no-op safely; the bridge REPLACES their role, doesn't fix them).

## Test plan (TDD; real SurveyModel via facade, mocked handles)

1. **Invalid submit** (the A15 headline): survey with required question, `tryComplete()` → bridge receives event with `question`, locks `allow=false`, native scroll called with measured target, `focusFirst` called after settle, NO `focusIn` from the bridge (component-owned).
2. **Interception honesty** (bridge.interception.test.ts — `settings.environment` forced back to `undefined` so the facade stub cannot mask it): unbridged baseline THROWS the A15 TypeError; bridged path doesn't; **a later subscriber reassigning `allow = true` cannot re-open core's DOM path** (irreversible-lock regression) and observers still read `false`; reassignment surfaces `allow-override-ignored` once per install.
3. **Scroll-only intents**: `page.scrollToTop()` (question null) → scroll, no focus; **real `panel.expand()`** (question attached, `elementId === inputId`) → scroll, NO focus synthesis; real `question.focus()` (`elementId === question.id`) → focus completes.
4. **Cross-page focus**: `focusQuestion(name)` parks `focusingQuestionInfo` across the page change; the 1.1 render-complete seam (EXACT afterRenderPage mirror: parked → `focusQuestionInfo()`, else → `scrollToTopOnPageChange()`) executes it; both branches tested.
5. **Visible-skip semantics**: fully-visible target → no scroll, focus still runs; host without viewport → scroll unconditionally.
6. **Registry lifecycle**: unmount deregisters (0.4 captured-pair hooks); post-uninstall events don't touch handles; model-swap installs fresh (old survey's events inert); **same-survey `fromJSON()` rebuild** — late old-tree deregisters can't remove new handles, stale instances don't resolve, uninstall clears the rebuilt registry.
7. **Fallbacks**: unregistered question → page fallback scroll + diagnostic once; no focusFirst → a11yFocus; no scroll host → no-op + diagnostic; rejecting `measureTarget` contained (never throw — invariant 9 spirit).
8. **Coalescing + async ordering**: two same-tick requests → one scroll to the last target; focus→page and page→focus same tick (focus intent retained across scroll-target coalescing); superseded mid-measure request never scrolls late (stale drop); focus/scroll-only requests during the settle window (supersede/re-park; exactly-once completion).
9. **Shim amendment**: environment stub idempotent, fields undefined, packaged-entry assertion; truthiness-gate sweep over BOTH installed artifacts (cjs + fesm mjs); **negative tripwires** proving the drag-drop and popup DOM paths still throw (unsupported-ness pinned).
10. **Consumer observability**: a consumer `onScrollToTop` handler still fires (sees `allow` locked false).
11. **Consult seam (`onScrollRequest`)**: returns `false` → no native `scrollTo`, focus initiation still fires (focusFirst/a11yFocus; no bridge focusIn); returns `true`/absent → normal scroll path; receives `{element, question, page, viaPageFallback}` with the resolved target; a throwing callback is contained and treated as `true`.

## Sequencing

1.2 ships registry + bridge + shim amendment + tests with a minimal test-only host harness. 1.1 wires: bridge install/uninstall on mount/model-swap/unmount, ScrollView host registration, the **render-complete seam mirroring `afterRenderPage` exactly** (`if (survey.focusingQuestionInfo) { (survey as any).focusQuestionInfo(); } else { survey.scrollToTopOnPageChange(); }` — private-method dependency pinned by the api-surface watchlist), and the consumer-facing `onScrollToElement` event prop. 1.5/1.7+ components register handles via the 0.4 mounted hooks as they land — their inputs MUST fire `question.focusIn()` from the native `onFocus` handler (the `ElementHandle.focusFirst` ownership contract).
