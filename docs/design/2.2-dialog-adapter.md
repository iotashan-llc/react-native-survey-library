# 2.2 — Confirmation-dialog adapter (CORE)

Status: DRAFT v4 (round-3 three-way findings folded; pair re-review pending)
Owner: orchestrator.
Depends on: 2.1 overlay host (design v3.2 approved; PR #26).

## Problem

Core's delete confirmations (paneldynamic `confirmDelete`, matrixdynamic,
file question) funnel through `confirmActionAsync(options)`
(utils/confirm-dialog.ts:13-27) with precedence:

1. `settings.confirmActionFunc` (SYNC consumer hook) — short-circuits;
2. `settings.confirmActionAsync` (async hook) — core SELF-INSTALLS a
   default wrapper at module eval (confirm-dialog.ts:61-65) that calls
   `showConfirmDialog`;
3. `showConfirmDialog` → **`settings.showDialog`** → DOM
   `PopupBaseViewModel` (RN tripwire; `settings.showDialog` is
   `undefined` until a renderer installs one).

**Round-1 pivot (codex crit #1):** the adapter overrides
`settings.showDialog` — NOT `confirmActionAsync`. Core's self-installed
default already routes through `showDialog`
(confirm-dialog.ts:29-55), so adapting the bottom of the funnel:

- needs NO fragile "was it core's default?" identity capture (the
  round-1 facade-eval heuristic misidentified consumer hooks installed
  before renderer evaluation — model-first import order, HMR epochs);
- leaves BOTH consumer hooks (`confirmActionFunc`,
  `confirmActionAsync`) fully functional at their natural precedence;
- matches upstream renderer practice: survey-react-ui itself installs
  `settings.showDialog` from its modal manager and clears it on
  teardown (popup-modal.tsx:31-54).

## D1 — Scope

- Implement `settings.showDialog(options: IDialogOptions)` returning the
  handle upstream callers dereference. In-scope callers: core's
  `showConfirmDialog` (title/danger mutation + `configConfirmDialog`
  width — confirm-dialog.ts:46-55) and direct consumer `showDialog`
  calls with a registered `componentName`.
- `rootElement` parameter ignored (DOM concept; DIFFERENCES).
- `settings.confirmActionFunc` / consumer-set `confirmActionAsync`:
  untouched, documented (they run BEFORE the default wrapper ever
  reaches us).

## D2 — Host registry (tokens with teardown ownership)

`src/overlay/dialog-adapter.ts`:

- `registerDialogHost(stack): DialogHostToken` — Survey shell calls in
  componentDidMount; `token.dispose()` in detach. Token:
  `{ stack, registrations: Set<PopupRegistration>, disposed: boolean }`.
  Routing: CURRENT host = last non-disposed token (registration order);
  disposal removes the token wherever it sits (out-of-order safe).
- **StrictMode/remount independence (round 2):** token lifecycle is
  self-contained take-and-clear — `dispose()` is idempotent and does
  NOT consult the Survey shell's `detached` flag (which stays true
  after a first teardown; StrictMode double-invokes effects/lifecycles).
  Each mount registers a FRESH token; test
  `<StrictMode><Survey/></StrictMode>`: the simulated first token is
  disposed, the replacement routes, final unmount leaves no live host.
- **Module-epoch anchoring (round 2/3, HMR):** ALL installer state —
  `{ dispatcher, impl, previous, enabled, tokens }` — lives ON the
  settings singleton under `Symbol.for('rnsl.dialogDispatcher')`. The
  TOKEN REGISTRY included (round 3): epoch B must see epoch A's live
  hosts for dispatch AND for last-token restoration decisions.
  `dispatcher` is the ONE stable function ever assigned to
  `settings.showDialog` (assignment + CAS compare exclusively via
  `slot.dispatcher`); it delegates to `impl`, which each epoch
  replaces. D8.8 test: live epoch-A host, dispatch works, epoch B
  installs, epoch-A token disposal still routes/restores correctly.
- INSTALL (0→1 tokens): capture `previous = settings.showDialog`; set
  `settings.showDialog = adapterShowDialog` (stable module ref). If
  `previous` is a consumer function (non-undefined, not ours), it is
  displaced ONLY while hosts are mounted; diagnostic
  `dialog-adapter-displaced-show-dialog` (once). Explicit opt-out:
  `setDialogAdapterEnabled(false)` export — registry still tracks hosts
  but never touches `settings`.
- RESTORE (last token disposed): CAS — restore `previous` ONLY if
  `settings.showDialog === adapterShowDialog` AND `previous` was a
  consumer function. If `previous` was `undefined`, the adapter STAYS
  installed (persistent dispatcher): a post-unmount core call then hits
  the no-host fail-safe instead of core throwing on
  `settings.showDialog(...)` of undefined. (Resolves the codex crit-2
  restoration/persistence tension explicitly: persistence wins; we
  restore only real consumer values.)
- TOKEN TEARDOWN OWNERSHIP (codex crit #2): every dialog opened while
  token T was current adds its `PopupRegistration` to
  `T.registrations` (removed again when the dialog finalizes). On
  `T.dispose()`: (1) mark disposed + remove from routing; (2)
  synchronously `unregister()` each live registration (2.1 semantic
  close → cancel sequence → resolve false); (3) THEN decide restoration.
  **Teardown is a transaction (round 2):** step-2 callbacks run with
  the dispatcher still installed and may re-enter (a cancel handler can
  mount a new Survey or open a new confirmation) — so "was that the
  last token" is RE-READ after step 2, never cached from before it;
  callback exceptions are collected and rethrown only after
  cleanup/restoration completes (finally). If a new token registered
  re-entrantly, no restoration happens. `dispose()` idempotent.

## D3 — adapterShowDialog(options, rootElement?)

1. `const opts = options` (IDialogOptions is required by signature);
   ignore `rootElement`.
2. No current host: diagnostic `dialog-no-host` (deduped) + invoke
   `opts.onCancel?.()` synchronously exactly once, return the DETACHED
   handle. **Concrete detached shape (round 2):** `{ footerToolbar:
   real ActionContainer seeded with two inert Actions (ids 'apply',
   'cancel' — non-null lookups whose title/innerCss are mutable and
   whose action() is a no-op), width: stored value (getter returns the
   exact assignment), popupModel: null }` — `popupModel` is
   EXPLICITLY nullable on the handle type; repeated no-host calls get
   isolated handles. Fail-SAFE: confirm callers treat cancel as "no" —
   never auto-confirm a destructive action.
3. Build `new PopupModel(opts.componentName, opts.data, popupOptions)`
   with the FIELD TABLE (round 3 — forward-if-supplied / force /
   ignore; omitted fields keep PopupModel DEFAULTS):

   | Field | Treatment |
   |---|---|
   | `componentName`, `data` | ctor args (required) |
   | `isModal` | FORCED `true` (dialog shape) |
   | `title`, `displayMode`, `isFocusedContent`, `isFocusedContainer`, `showCloseButton`, `cssClass`, `locale` | forward if supplied |
   | `onApply`, `onCancel`, `onHide`, `onShow`, `onDispose` | composed (state machine / finalizer / timing below) |
   | `verticalPosition`, `horizontalPosition`, `showPointer`, `setWidthByTarget`, `positionMode`, `canShrink` | IGNORED (coordinate positioning: won't-support) → D6 |
   | `rootElement` | IGNORED (DOM) → D6 |

   **onShow timing (round 3):** the 2.1 bridge runs `popup.onShow()`
   BEFORE the entry/record exists (present() sequence), so a consumer
   `onShow` that synchronously hides or unregisters would race an
   unpushed record. The adapter therefore does NOT pass consumer
   `onShow` into the PopupModel; it invokes it on a microtask AFTER
   `popup.show()` returns (entry pushed, record live) — divergence
   (post-present vs during-show) documented in D6; a hide from within
   it is then an ordinary dismissal. `onHide`/`onDispose` compose
   consumer-first, adapter cleanup after, throw-safe (finally).
4. Resolution STATE MACHINE (round 2, both peers): states `open →
   applying → terminal`. `popup.onApply`: if terminal, no-op-false;
   set `applying`; run `opts.onApply?.()`; if it returns `false` AND
   no re-entrant teardown/hide moved the state to terminal, the state
   REOPENS (validation-retry stays possible — a latched-on-entry
   `once` would consume the failed attempt) and the popup stays open;
   otherwise mark terminal(true). **`applying` is a guarded
   transaction (round 3):** a cancellation/hide/unregister arriving
   WHILE `applying` records a PENDING DISMISSAL without invoking the
   false callback; when apply returns: non-false → terminal(true)
   commits once (the yes-action ran); false + pending dismissal →
   terminal(false); false + still open → reopen. `popup.onCancel`:
   terminal(false) unless already terminal or applying (then pending). `popup.onHide` FINALIZER: every dismissal
   path (apply, cancel button, Android back, a11y escape, factory-miss
   Close/`requestHide`, programmatic `hide()`, unregister semantic
   close) funnels through `onHiding()` → `onHide`; the finalizer runs
   terminal(false) if still non-terminal (hide-before-resolution
   resolves `false`, never zero times), removes the registration from
   its token, and schedules bridge `unregister()` on a microtask.
   Terminal transitions mark state BEFORE invoking the consumer
   callback. **Re-entrant unmount during apply (round 3, gemini):**
   `opts.onApply` may synchronously unmount the owning Survey (a
   paneldynamic delete removes the panel → tree collapses) — the token
   teardown then drives hide/onHide WHILE state is `applying`. The
   finalizer therefore treats `applying` like terminal (it does NOT
   resolve false — the consumer's yes-action already ran); the apply
   path completes the resolution when `opts.onApply` returns
   (non-false → terminal(true); false with the popup already torn down
   → terminal(false), no reopen). Tests: apply-false → retry-apply-true;
   apply-false → cancel; apply-true → hide → unregister; apply whose
   callback synchronously disposes the host token (one resolution,
   value true).
5. `registerPopup(popup, token.stack)`; add registration to token;
   `popup.show()`.
6. Return the HANDLE (D4).

Confirm-dialog resolution mapping (`callback` = core's `callbackFunc`):
`onApply → callback(true)`; everything else → `callback(false)`,
exactly once (upstream showConfirmDialog binds these via
options.onApply/onCancel — confirm-dialog.ts:35-43).

## D4 — The returned handle (upstream showConfirmDialog compatibility)

Core does, post-return (confirm-dialog.ts:46-55):

```ts
const toolbar = popupViewModel.footerToolbar;
toolbar.getActionById("apply").title = ...; // + innerCss danger
toolbar.getActionById("cancel").title = ...;
popupViewModel.width = "min-content";       // configConfirmDialog
```

Handle: `{ footerToolbar, get/set width, popupModel }` where
`footerToolbar` IS the 2.1 bridge's real footer `ActionContainer` (the
bridge exposes it on the payload; actions are real `Action` instances,
so post-hoc `title`/`innerCss` mutations re-render reactively through
the row subscriptions — the same mechanism ActionButton already uses).
`width` (round 2, ONE contract): a STORED compatibility value — the
getter returns exactly what was assigned; it has NO visual effect in
v1 (RN dialog panel width comes from the recipe); DIFFERENCES entry.
`popupModel` is the real PopupModel when attached, `null` on the
detached no-host handle (typed nullable). The handle is a plain object
— NOT the PopupModel itself and not a `PopupBaseViewModel` (rejected
round-2 alternative: attaching `width`/`footerToolbar` onto the core
model instance via defineProperty mutates unmodified-core objects;
direct consumers wanting dismissal use `handle.popupModel.hide()`).
No `PopupBaseViewModel` API beyond these members is promised
(DIFFERENCES).

Danger styling (codex maj #5): the overlay footer maps
`action.innerCss` containing `sd-btn--danger` → `ActionButton
variant="danger"` at the render site; test asserts the RENDERED danger
recipe style, not just the model field.

## D5 — Sequences (exactly-once on the confirm callback)

| Path | Resolution |
|---|---|
| Apply | `onApply` → callback(true), popup hides, finalizer no-op |
| Cancel button / Android back / a11y escape | cancel sequence → callback(false) |
| Backdrop tap | nothing (dialog shape) |
| Factory-miss Close / programmatic `hide()` | HIDE only → `onHide` finalizer → callback(false) |
| Host token disposed while open | semantic close → cancel → callback(false) |
| No host at call time | callback(false) sync + detached handle |

## D6 — DIFFERENCES entries

- `showDialog` returns a compatibility handle, not a
  `PopupBaseViewModel` (only `footerToolbar`/`width`/`popupModel`).
- `rootElement`/`cssClass` visual containment ignored; dialog renders in
  the LAST-MOUNTED Survey's overlay.
- Consumer `settings.showDialog` installed BEFORE first Survey mount is
  displaced while Surveys are mounted (restored on last unmount);
  opt-out via `setDialogAdapterEnabled(false)`.

## D8 — Test plan (jest, real core)

1. Registry: install/restore CAS matrix — previous undefined (persist),
   previous consumer fn (restore), consumer replaces ours while mounted
   (leave theirs), opt-out flag, double dispose, A/B mount-unmount
   permutations, register-while-open keeps the dialog on its stack.
2. Token teardown: open dialog on A; mount B; unmount A → dialog
   semantically closed, callback(false) once; new dialog routes to B.
3. REAL core default path: DO NOT set confirmActionAsync; call core's
   `confirmActionAsync({message, funcOnYes, funcOnNo})` util (via a real

   the call; apply removes the panel, cancel keeps it; titles mutated
   post-hoc are reflected (ok/cancel locale, applyTitle), apply button
   renders the DANGER recipe style.
4. Exactly-once: apply→hide→unregister; direct hide (factory miss
   componentName); repeated back; cancel-then-dispose — one resolution
   each; zero-resolution impossible (hide-only path resolves false).
5. No-host: pre-mount call → callback(false) + inert handle usable by
   upstream mutation lines (non-null apply/cancel lookups, mutable
   title/innerCss, no-op action(), width stored, popupModel null);
   post-last-unmount with undefined previous → same (persistent
   dispatcher); repeated no-host calls isolated.
6. `configConfirmDialog` width line doesn't throw; getter returns the
   assigned value; handle exposes the real footer container.
7. StrictMode: `<StrictMode><Survey/></StrictMode>` mount/unmount —

8. Epoch: simulate a second install epoch (fresh module state against

9. Direct showDialog: title + showCloseButton forwarded, consumer
