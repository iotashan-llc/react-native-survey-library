# Task 1.3 design — width-expression resolver: core width grammar → RN numbers

Status: draft for peer review (CORE task — orchestrator llm-pair round
pending per the CORE protocol). Drafted by the 1.3 task agent from
direct upstream source reading + live headless verification. Prereq for
1.4 (composition consumes this); grammar evaluator reusable by 1.1
(`survey.renderedWidth`) and 1.7 (`questionTitleWidth`). Source facts
verified against survey-core v2.5.33 — both the monorepo source
(`../survey-library`) and the installed package through the facade.

## Verified upstream facts the design rests on

- **`element.rootStyle` is the SINGLE width binding point.** The DOM
  renderer binds it verbatim as inline style
  (`survey-react-ui/src/element.tsx:77`); everything else (user `width`,
  `renderWidth` row math, `minWidth`/`maxWidth`, `colSpan`/
  `gridLayoutColumns`) is collapsed INTO it by core
  (`survey-element.ts` `calcRootStyle`, :1119-1153). It is a reactive
  property, recomputed when any of `["minWidth", "maxWidth",
  "renderWidth", "allowRootStyle", "parent"]` change
  (`survey-element.ts:301`) and when panel layout columns change
  (`panel.ts` `updateRootStyle`).
- **Two rootStyle shapes** (empirically confirmed headless):
  - grid branch (survey `gridLayoutEnabled` + parent columns; skipped
    when the spanned width sums to 0 or 100): `{flexGrow: 1, flexShrink:
    0, flexBasis: "<Σ effectiveWidth>%", minWidth: undefined, maxWidth}`
    — e.g. colSpan 1+2 over 3 columns → `"33.33%"` / `"66.66%"`, and
    `effectiveColSpan` written as a side effect of `getColumsForElement`.
  - renderWidth branch (default; `allowRootStyle && renderWidth`):
    `{flexGrow: 1, flexShrink: 1, flexBasis: <renderWidth>, minWidth:
    "min(100%, <minWidth>)" | "auto" | "", maxWidth: <maxWidth>}`. Only
    a `px` minWidth is scaled by `survey.widthScale/100` before the
    `min()` wrap (widthScale = 100 in RN — the DOM responsivity manager
    that changes it never runs); non-px minWidth enters `min()`
    unscaled; `"auto"` bypasses the wrap entirely.
  - otherwise `{}`.
- **The row width math runs synchronously headless.**
  `QuestionRowModel.setWidth` is invoked through
  `visibleElementsAnimation.sync()` (`panel.ts:163-166`), and
  `AnimationProperty.sync` calls `update()` directly when animations are
  disallowed (`utils/animation.ts:283-290`). Animations ARE disallowed
  headless: `getIsAnimationAllowed` (`base.ts:1663`) requires
  `onElementRerendered` non-undefined, and that getter returns undefined
  until a renderer calls `enableOnElementRerenderedEvent()`
  (`base.ts:1684`) — which this renderer never does in M1. Verified by
  probe: all values below populate at `new SurveyModel(json)` with no
  DOM and no render pass.
- **The full `renderWidth` grammar core can emit** (`panel.ts` `setWidth`
  :205-240, `getRenderedCalcWidth` :242-256, `getRenderedWidthFromWidth`
  :262-264):
  - `""` — invisible element (also flowpanel members);
  - user `width` **verbatim** when it isn't a bare number — `"25%"`,
    `"300px"`, but equally `"10em"` or `"banana"` (no validation
    upstream; confirmed `"banana"` passes through);
  - bare-number user width → px-suffixed (`"250"` → `"250px"`,
    `Helpers.isNumber` semantics: plain numerics only, `parseFloat` +
    `isFinite` coercion);
  - no user widths in the row → equal split
    `` `${parseFloat((100/visCount).toFixed(6))}%` `` (3 → `"33.333333%"`);
  - mixed → `calc(100% - W1 - …)` for the single unsized element, or
    `calc((100% - W1 - …)/M)` for M unsized elements, where each `Wi` is
    a preset element's already-rendered width (px, %, or verbatim user
    junk — junk INSIDE calc is possible upstream and must degrade, not
    throw);
  - fixed `"100%"` — question_baseselect/custom/paneldynamic-template/
    matrix-detail panels set it directly.
- **`min()`/`calc()` are the only CSS functions core emits** for widths
  (`grep "calc(\|min(\|max("` over survey-core src: `panel.ts:255`,
  `survey-element.ts:1143`; the slider's calc is a CSS-var write, not a
  width).
- **Defaults**: question `minWidth` = `settings.minWidth` `"300px"`,
  panel/paneldynamic `minWidth` `"auto"`; `maxWidth` =
  `settings.maxWidth` `"100%"` (question.ts:3263-3264, panel.ts:2579-80).
- **Row semantics** (`sd-row.scss`): a row is `display:flex; flex-
  direction:row; width:100%`. A multi-element row (`rowMultiple` css
  when `visibleElements.length > 1`, `getRowCss` panel.ts:270-277) adds
  `flex-wrap: wrap`, `row-gap`, and the **gutter technique**: row
  `margin-left: -g; width: calc(100% + g)`, each element wrapper
  `padding-left: g` (g = `--sd-base-padding`). Consequence: **in DOM, a
  `%` flexBasis resolves against (rowWidth + g), and each element's
  basis INCLUDES its own left gutter.** An RN translation that resolves
  `%` against the bare row width and adds `gap` would wrap rows the DOM
  keeps on one line (Yoga gap is outside the basis; N×(100/N)% + gaps
  > 100%). The resolver must own this subtlety in one place.
- **`startWithNewLine` is row COMPOSITION, not width grammar** — it
  drives `buildRows` (`panel.ts:1629-1642`); by the time widths matter
  the model already exposes `panel.rows[].visibleElements`. 1.4 maps
  rows to Views; 1.3 consumes a row's elements as given.
- **Indents are a separate channel**: `indent`/`rightIndent` →
  `getIndentSize` → always `"<n>px"` (question.ts:1597-1600) → CSS vars
  via `getRootStyle()` (not `rootStyle`). 1.4 composition concern; the
  evaluator here handles px trivially if 1.4 wants it.

## Scope — what 1.3 is (and isn't)

A **pure TS module `src/layout/`** (RN-import-free, the theme-core
precedent): parse + evaluate the documented width grammar, and translate
live `rootStyle` objects into numeric RN styles given a measured
container width. It ships no components and no subscriptions — 0.4's
reactive base already re-renders on `rootStyle` property changes, and
1.4 wires `onLayout` → resolver → child styles.

### D1 — consume `rootStyle`; never re-derive row math

Invariant 6 (never duplicate core's model-state logic). Upstream has ONE
binding point and we use the same one. `colSpan`, `gridLayoutColumns`,
equal-split, and calc composition all stay core's job; the resolver is a
translation layer for the strings core already computed. (Rejected
alternative: re-implementing `setWidth`/`calcRootStyle` from `width`/
`colSpan` inputs — it would drift with every upstream layout fix and
break invariant 6.)

### D2 — documented supported grammar (the "documented supported grammar" deliverable)

```
value   := "" | "auto" | <number> | "<number>" | "<number>px" | "<number>%"
         | "calc(" expr ")" | "min(" value ("," value)+ ")" | "max(" value ("," value)+ ")"
expr    := arithmetic over the same operands with + - * / and nested
           parens; min()/max() allowed inside calc()
```

- Bare numbers = px (core's own convention: `getRenderedWidthFromWidth`
  px-suffixes bare numbers; user-set bare-number `minWidth` reaches
  `min(100%, 250)` unwrapped — CSS-invalid upstream, meaningful here).
- CSS calc type rules enforced (cheap, catches nonsense): `+`/`-` need
  both operands same-type (length±length or number±number); `*` needs ≥1
  plain number; `/` divisor must be a plain non-zero number. A top-level
  plain-number result coerces to px.
- `%` resolves against the caller-provided `percentBase` immediately.
- Any other unit (`em`, `rem`, `vw`, `vh`, `pt`, `fr`, …) → diagnostic
  `layout/unsupported-width-unit`, constraint dropped. Anything else
  unparseable (`"banana"`, unknown functions like `fit-content()`,
  division by zero, mixed-type arithmetic) → `layout/invalid-width`,
  constraint dropped. Dropped means: that ONE property is omitted from
  the output (element falls back to share-the-space `flexGrow`
  behavior); the other properties keep their values; nothing throws
  (invariant 9 spirit). DIFFERENCES.md gets the grammar table + the
  em/rem/vw won't-support entry.

### D3 — evaluation semantics

- `px` → dp 1:1 (CSS reference px ≡ RN dp; both are density-independent).
- `%` base is the caller's `percentBase` (dp): 1.4 passes
  `rowWidth + gutter` for multi-element rows and `rowWidth` for single —
  the row helper below owns that rule so the DOM-parity subtlety lives
  in exactly one place.
- Results are unrounded; negative resolved lengths clamp to 0 (CSS
  used-value clamping for flex-basis/min/max-width). 1.4 applies
  `PixelRatio.roundToNearestPixel` at the style edge; `flexGrow: 1`
  absorbs sub-pixel residue (upstream sets it on every width-bearing
  element, so equal-split rounding gaps self-heal).
- Everything resolves numerically — no RN `%`-string passthrough tier.
  Rationale: the default `minWidth: "min(100%, 300px)"` sits on
  essentially every question and is inexpressible in RN without a
  measured width, so measurement is required anyway; a mixed
  numeric/percent-string output would make one row's elements resolve
  against two different layout passes. Cost: 1.4 defers a row's
  children by one frame until first `onLayout` (standard RN pattern,
  documented there).

### D4 — output contract (pure data, theme-core pattern)

```ts
resolveWidthStyle(rootStyle, { percentBase })
  → { style: { flexGrow?, flexShrink?, flexBasis?, minWidth?, maxWidth? },  // all numbers, dp
      diagnostics: WidthDiagnostic[] }
```

- `flexGrow`/`flexShrink` pass through when numeric (grid branch's
  `flexShrink: 0` preserved — sized grid cells must not shrink).
- `flexBasis`: resolved dp; `"auto"`/`""`/dropped → key omitted.
- `minWidth`: resolved dp; `"auto"`/`""`/`undefined`/dropped → omitted
  (grid branch emits `minWidth: undefined` deliberately — preserved as
  omission, NOT replaced with a default).
- `maxWidth`: same rules (`"100%"` → `percentBase`, faithful to DOM
  where 100% resolves against the same gutter-widened base).
- `{}` in → `{ style: {}, diagnostics: [] }`.
- Diagnostics are **returned as data, never pushed through the
  `src/diagnostics.ts` seam** (observable purity, same contract as
  `resolveTheme`); 1.4's row component forwards them post-commit through
  the seam with a new `layout-diagnostic` payload code added there,
  deduped per (element, offending value) at the forwarding edge.

```ts
resolveRowWidths(row, { rowWidth, gutter = 0 })
  → { isMultiple, percentBase, elements: [{ element, ...WidthResolution }] }
```

- `isMultiple` = `row.visibleElements.length > 1` (the `rowMultiple`
  rule, `getRowCss` parity); `percentBase` = `isMultiple ? rowWidth +
  gutter : rowWidth` (gutter fact above; 1.4 supplies g from theme
  tokens and renders the wrapper `paddingStart: g` + row
  `marginStart: -g` to complete the DOM-equivalent geometry).
- Iterates `row.visibleElements` (already visibility-filtered by core)
  structurally typed (`RowLike`), so tests and 1.4 can pass either a
  live `QuestionRowModel` or a stub.
- `evaluateWidthExpression(raw, percentBase)` is exported for 1.1's
  `survey.renderedWidth` (static px | undefined) and 1.7's
  `questionTitleWidth` reuse.
- Guard rails: non-finite/negative `percentBase`/`rowWidth` clamps to 0
  (pre-measure calls resolve %-terms to 0 rather than NaN-poisoning
  Yoga; 1.4 shouldn't call pre-measure at all).

### D5 — parser implementation

Hand-rolled tokenizer + recursive-descent evaluator (~150 lines, no
dependency, no regex-only shortcut — nested parens and n-ary min/max
need real recursion). Case-insensitive function names; tolerant of
arbitrary whitespace (core emits `" - "`-spaced calc; users may not).
Unary minus supported at factor position. Unknown identifiers/functions
terminate that value's parse with `layout/invalid-width`.

### ESLint purity fence

`src/layout/**` gets the same flat-config block as `src/theme-core/**`
(zero `react-native` imports; survey-core only via facade — and
re-including the survey-core/native-html restrictions since flat config
replaces rather than merges `no-restricted-*` rules per matching block).

## Exclusions (documented, deliberate)

- `questionTitleWidth`/`titleLocation:"left"` column alignment → 1.7
  (evaluator reusable).
- `survey.renderedWidth`/`widthMode`/root width → 1.1 (`<Survey>` root).
- `indent`/`rightIndent` paddings, RTL/direction, row `gap`/wrap
  rendering, onLayout wiring → 1.4 composition.
- `settings.matrix.columnWidthsByType` and matrix column widths → M3.
- `widthScale` ≠ 100 scenarios: core pre-bakes it into the minWidth
  string; no RN responsivity manager exists to change it. Pass-through.
- No live re-resolution subscription machinery: `rootStyle` is a plain
  reactive property; 0.4's base classes cover it (1.4 integration test
  asserts a `width` change re-renders with new numbers).

## Test plan (TDD; grammar units + live-model integration)

1. **Grammar units** (`width-resolver.test.ts`): px/%/bare-number/
   number-typed/auto/empty; both emitted calc shapes at exact probe
   values (`"33.333333%"`@800, `calc(100% - 300px - 20%)`@800→340,
   `calc((100% - 300px)/2)`); nested parens; `min(100%, 300px)` at
   percentBase 800→300 and 250→250; n-ary + nested min/max; whitespace
   variants; unary minus; negative-result clamp to 0; unsupported units
   (`em`/`rem`/`vw`) → `layout/unsupported-width-unit`; `"banana"`,
   unknown fn, `/0`, `len*len`, `len+number` → `layout/invalid-width`;
   junk INSIDE a calc term degrades that value only.
2. **rootStyle translation**: renderWidth-branch full shape (grow 1 /
   shrink 1 / basis / min / max all numeric); grid-branch shape (shrink
   0 preserved, minWidth stays omitted); `{}` → empty; `"auto"` minWidth
   omitted; dropped-constraint keeps siblings.
3. **Live-model integration** (`width-resolver.model.test.ts`, facade,
   headless — locks the "verified upstream facts" as regressions):
   equal-split 3-up row; mixed `300px`/calc/`20%` row; grid
   `colSpan` 1+2 → 33.33/66.66 + `effectiveColSpan`; bare-number
   `"250"` → 250; `"banana"` width → passthrough → diagnostic + dropped
   basis; single-element row `"100%"`; assert synchronous availability
   (no render, no DOM).
4. **Row helper**: multi-row `percentBase = rowWidth + gutter`, single =
   `rowWidth`; `isMultiple` parity with `getRowCss` `rowMultiple` on the
   live model; stub rows accepted.
5. **Purity** (`purity.test.ts`, theme-core pattern): resolver returns
   diagnostics as data and never calls `reportDiagnostic`; type-only
   facade import doesn't evaluate survey-core; ESLint fence exercised by
   the lint run.
