# Task 2.5 — rating-dropdown + buttongroup overflow — FINALIZED design (probe-verified)

Supersedes the DRAFT in `docs/design/2.5-...-plan.md` (on feat/2.7 branch).
All facts below verified by headless probe of survey-core 2.5.33 via the
facade (2026-07-18). Implementation is merge-gated on 2.3-dropdown +
2.4-tagbox landing on main (the OverlayControl extraction needs their code
as its live safety net).

## Probe findings (corrections to the DRAFT)

### Part A — rating `renderAs:"dropdown"`
- getType="rating", getTemplate="rating", isDefaultRendering=true → dispatches
  through the normal **RatingQuestion** (1.14) template route. CONFIRMED.
- `dropdownListModel` present; `dropdownListModel.listModel.actions.length===5`
  (the rate values); `popupModel.contentComponentName === "sv-list"`.
  → rating-dropdown drives the **SAME** dropdownListModel + sv-list popup that
    dropdown/tagbox use. Part A = "reuse the dropdown overlay control with
    rating value semantics."
- `itemComponent === "sv-rating-dropdown-item"` is the COLLAPSED selected-value
  display, NOT an overlay list-row (rows come from sv-list/ListPicker). The 1.14
  deferral comment's "former" implication is wrong. **No `sv-rating-dropdown-item`
  element-route registration is needed** unless a dispatch actually hits it —
  RatingQuestion renders the collapsed display directly.

### Part B — buttongroup overflow
- protoChain: QuestionButtonGroupModel → QuestionCheckboxBase → QuestionSelectBase.
- The DRAFT's "buttongroup has dropdownListModel + needResponsiveWidth +
  supportResponsiveness" is misleading:
  - `dropdownListModel` is **LAZY** — undefined at construction; built
    (`new DropdownListModel(this)`) only when `renderAs === "dropdown"`.
  - `needResponsiveWidth` / `supportResponsiveness` are **methods**, not booleans.
- The overflow decision is core's `processResponsiveness(requiredWidth,
  availableWidth)` (question.ts:3059): if required>available →
  `renderAs = getCompactRenderAs()` ("dropdown"); else → getDesktopRenderAs()
  ("default"). Web drives this from a ResizeObserver in `initResponsiveness`
  (required=scrollWidth, available=clientWidth).
- **Dispatch correction (key finding):** after the flip, `renderAs==="dropdown"`
  but `isDefaultRendering()` stays true, `getTemplate()` stays "buttongroup",
  `getComponentName()` stays "default". So `dispatch-key.ts`
  (isDefaultRendering?getTemplate:getComponentName) would route it to the button
  row regardless. **ButtonGroupQuestion must branch on `question.renderAs`
  itself** — do NOT rely on dispatch-key / RendererFactory. (Web uses
  RendererFactory("buttongroup","dropdown"→"sv-buttongroup-dropdown"); we skip
  that port — one RN component self-branches. ponytail.)
- When compact, `dropdownListModel.listModel.actions.length===3`,
  popup `sv-list` — SAME overlay path as Part A / dropdown.

## Design (unified)

Both parts collapse to: **an existing question component gains a
`renderAs==="dropdown"` branch that renders a shared OverlayControl driven by
`question.dropdownListModel`.**

### Shared work FIRST — extract `OverlayControl`
Dropdown (2.3) + tagbox (2.4) already duplicate: opener Pressable, 2.1
popup-bridge registration/reconcile/teardown, combobox a11y (role + STRING
aria-expanded + label), controlRef/openerHandle focus return, the clear gate.
Extract to a shared base/hook (`OverlayControlBase extends QuestionElementBase`
or `useOverlayControl`) consumed by dropdown, tagbox, rating-dropdown,
buttongroup-overflow. Do it AFTER 2.3/2.4 merge, with their green suites as the
regression net (invariant: keep every dropdown+tagbox test green through the
extraction — that is the safety contract).

### Part A — RatingQuestion
- Detect `renderAs==="dropdown"` (equivalently `dropdownListModel` present).
- Branch: render `OverlayControl(dropdownListModel)` instead of the rating item
  rows. Collapsed display = selected rate value's text.
- Rows resolve through the existing sv-list/ListPicker (no new item component).
- Reactive: `renderAs` is a core property → base subscription re-renders on flip.

### Part B — ButtonGroupQuestion
- Keep the 2.9 horizontal ScrollView for the button row (renderAs==="default").
- **RN measurement seam (replaces ResizeObserver):** ScrollView
  `onContentSizeChange` → contentWidth (= required); `onLayout` → viewportWidth
  (= available). When both known and they cross, call
  `question.processResponsiveness(contentWidth, viewportWidth)` — reuse core's
  decision (invariant 6; do not reimplement the threshold).
- On `renderAs==="dropdown"` → render `OverlayControl(dropdownListModel)`; else
  the button row.
- **Loop guard:** processResponsiveness flips renderAs → re-render → re-measure.
  Only call it when the measured pair would actually change the decision (core's
  own compare is idempotent — verified — but still gate on "numbers known &&
  changed" to avoid onLayout churn / a measure↔setState loop).

### Registration / kitchen-sink / DIFFERENCES
- No new question-type descriptor rows (both are variants of already-registered
  `rating` / `buttongroup`). Update the `sv-rating-dropdown-item` deferral note.
- kitchen-sink: add a `rating renderAs:dropdown` + a `buttongroup` with enough
  choices to force overflow in a narrow container.
- DIFFERENCES: rating-dropdown + buttongroup-overflow both use the 2.1 overlay
  sheet; buttongroup overflow is width-measured via ScrollView content-vs-viewport
  (vs web ResizeObserver); `:focus-within` ring has no RN analog (already noted
  for 2.9).

## Test plan (TDD, red first)
- OverlayControl extraction: all existing dropdown + tagbox tests stay green
  (the net). Add focused tests for the shared base's a11y/teardown contract.
- Part A: rating renderAs:dropdown renders the collapsed control (not item rows);
  press opens overlay with the 5 rate values; select commits the rate value;
  combobox a11y (string aria-expanded).
- Part B: buttongroup that fits → button row (2.9 unchanged); force

  measure↔render loop (measurement called bounded number of times).

# RECONCILED (post three-way review: Claude + Codex gpt-5.6-sol@max + Gemini 3.1 Pro High)

Both peers REJECTED the first-pass "self-branch + unmount" design on the same
CRITICAL flaw. Codex added blocking seam gaps. Revised design below is the one to
implement.

## R1. Dispatch (CONFIRMED sound by Codex)
- rating-dropdown: register a RN RendererFactory route ("rating","dropdown" ->
  our dropdown-style renderer) via the EXISTING descriptor renderer-route
  mechanism (same as boolean 1.13). Once registered, getComponentName() returns
  our key + isDefaultRendering() -> false, so the existing SurveyRowElement
  dispatchKey logic routes correctly with NO new dispatch code. (The earlier
  probe saw isDefaultRendering=true ONLY because no renderer was registered in
  the probe env.)
- buttongroup: NO RendererFactory registration. It self-branches internally
  (see R2) and MUST keep dispatching through the existing "buttongroup" template
  row. Codex confirms skipping sv-rating-dropdown-item + not porting the
  buttongroup RendererFactory route is correct.

## R2. ButtonGroup measurement (CRITICAL fix — both peers)
- Do NOT unmount the button row when compact — that kills onLayout/onContentSize
  and the flip-back can never fire.
- Always-mounted OUTER View measures availableWidth (onLayout) in BOTH modes.
- Cache the intrinsic required width from the default button-row branch
  (onContentSizeChange); keep comparing the CACHED required width against the
  live available width while compact.
- Feed (cachedRequired, availableWidth) to core's processResponsiveness; let CORE
  decide. Do NOT reimplement the threshold.

## R3. processResponsiveness adapter (Codex)
- It is declared `protected` in survey-core typings -> isolate ONE cast in a
  single adapter fn; add an api-surface entry + a behavioral compat test
  (watchlist: lazy dropdownListModel creation, the +/-2 rounding deadband,
  compact/desktop transitions).
- It only does the width transition; web separately gates via needResponsiveness
  incl. a design-mode exclusion. Supply a caller-side gate so RN never compacts
  in design mode (this runtime renderer is not a designer, but assert it).

## R4. OverlayContext wrappers (HIGH, blocking — Codex)
- dropdown/tagbox inject the per-Survey overlay stack through functional
  *QuestionElement wrapper components; rating/buttongroup descriptor rows
  instantiate the classes directly -> onClick toggles PopupModel but NO RN Modal
  registers.
- Add RatingQuestionElement + ButtonGroupQuestionElement context wrappers; repoint
  the EXISTING descriptor rows at them (no new question-type or renderer rows).

## R5. dropdownListModel lifetime (Codex)
- Lazy-created but core RETAINS it after returning to default (until disposal).
- Gate overlay registration on an explicit isOverlayMode hook, NOT VM presence.
  Reconcile to null + semantically unregister/close whenever isOverlayMode goes
  false — including while the popup is open.

## R6. OverlayControlBase extraction — BASE CLASS (Codex; Gemini's "multiple
##     inheritance" objection is wrong here)
- OverlayControlBase extends QuestionElementBase. dropdown, tagbox, rating,
  buttongroup extend OverlayControlBase (single chain — they simply swap their
  base). Do NOT also nest a second QuestionElementBase-based control (Gemini's
  composition would double-nest).
- Base owns: popup-bridge registration/reconcile/teardown, controlRef +
  openerHandle focus-return, combobox a11y (role + STRING aria-expanded + label),
  clear gate. Protected hooks per consumer: isOverlayMode, active mode,
  state-elements, collapsed content, post-reconcile. Respect the real per-consumer
  differences (dropdown excludes its VM in select mode; tagbox retains it + has
  separate chips/opener; rating has diagnostic lifecycles).
- Do the extraction AFTER 2.3/2.4 merge to main, their green suites as the net.

## R7. Collapsed rating display (Codex)
- Use core's selectedItemLocText via the LocString renderer; readOnlyText when
  read-only; core placeholder when empty. NOT a hardcoded value string.

## R8. Rating public API is displayMode (Codex)
- displayMode ('dropdown'/'auto'/'buttons') drives renderAs at load. Tests MUST
  use displayMode:'dropdown' + runtime displayMode changes, not just setting
  renderAs. Explicitly exclude (or define) displayMode:'auto' responsiveness.

## Reconciled TDD list (superset, red first)
- ButtonGroup: widen the always-mounted wrapper while already compact WITHOUT a
  new content-size event -> returns to the row; both callback orders; repeated
  identical callbacks; non-finite/zero/fractional widths; exact +/-2 deadband;
  a NUMERIC max call-count (not just "bounded"); design mode never compacts;
  default->compact->default->compact reuses the retained VM w/ no duplicate
  subscriptions/registrations; flip back while popup open; reset caches +
  retarget popup ownership on question-prop change.
- End-to-end: render rating + buttongroup THROUGH their descriptor routes inside
  Survey/OverlayContext (a manually-supplied stack won't catch the missing
  wrappers).
- Shared base contract parameterized across all 4 consumers: stack swap, question
  swap, unmount-while-open, focus return, live aria-expanded string conversion,
  read-only gate, clear.
- Rating: displayMode:'dropdown', runtime mode change, empty placeholder, clear,
  read-only display, localized/custom captions; REMOVE the old test expecting
  renderAs:'dropdown' -> element-wrapper-missing diagnostic.
- ButtonGroup: a fitting render does NOT instantiate dropdownListModel; compact
  creates it exactly once; collapsed text uses the selected localized caption;
  compact buttongroup still dispatches through the buttongroup row (no
  RendererFactory registration).
- Real-device layout gate: rotation, font scaling, long/localized captions
  (synthetic jest onContentSizeChange can't prove intrinsic width).
