## Probe findings (corrections to the DRAFT)

### Part A — rating `renderAs:"dropdown"`
- getType="rating", getTemplate="rating", isDefaultRendering=true → dispatches
  through the normal **RatingQuestion** (1.14) template route. CONFIRMED.
- `dropdownListModel` present; `dropdownListModel.listModel.actions.length===5`
  (the rate values); `popupModel.contentComponentName === "sv-list"`.
  → rating-dropdown drives the **SAME** dropdownListModel + sv-list popup that
    dropdown/tagbox use. Part A = "reuse the dropdown overlay control with
    rating value semantics."
- `itemComponent === "sv-rating-dropdown-item"` is the COLLAPSED selected-value
  display, NOT an overlay list-row (rows come from sv-list/ListPicker). The 1.14
  deferral comment's "former" implication is wrong. **No `sv-rating-dropdown-item`
  element-route registration is needed** unless a dispatch actually hits it —
  RatingQuestion renders the collapsed display directly.

### Part B — buttongroup overflow
- protoChain: QuestionButtonGroupModel → QuestionCheckboxBase → QuestionSelectBase.
- The DRAFT's "buttongroup has dropdownListModel + needResponsiveWidth +
  supportResponsiveness" is misleading:
  - `dropdownListModel` is **LAZY** — undefined at construction; built
    (`new DropdownListModel(this)`) only when `renderAs === "dropdown"`.
  - `needResponsiveWidth` / `supportResponsiveness` are **methods**, not booleans.
- The overflow decision is core's `processResponsiveness(requiredWidth,
  availableWidth)` (question.ts:3059): if required>available →
  `renderAs = getCompactRenderAs()` ("dropdown"); else → getDesktopRenderAs()
  ("default"). Web drives this from a ResizeObserver in `initResponsiveness`
  (required=scrollWidth, available=clientWidth).
- **Dispatch correction (key finding):** after the flip, `renderAs==="dropdown"`
  but `isDefaultRendering()` stays true, `getTemplate()` stays "buttongroup",
  `getComponentName()` stays "default". So `dispatch-key.ts`
  (isDefaultRendering?getTemplate:getComponentName) would route it to the button
  row regardless. **ButtonGroupQuestion must branch on `question.renderAs`
  itself** — do NOT rely on dispatch-key / RendererFactory. (Web uses
  RendererFactory("buttongroup","dropdown"→"sv-buttongroup-dropdown"); we skip
  that port — one RN component self-branches. ponytail.)
- When compact, `dropdownListModel.listModel.actions.length===3`,
  popup `sv-list` — SAME overlay path as Part A / dropdown.

## Design (unified)

Both parts collapse to: **an existing question component gains a
`renderAs==="dropdown"` branch that renders a shared OverlayControl driven by
`question.dropdownListModel`.**

### Shared work FIRST — extract `OverlayControl`
Dropdown (2.3) + tagbox (2.4) already duplicate: opener Pressable, 2.1
popup-bridge registration/reconcile/teardown, combobox a11y (role + STRING
aria-expanded + label), controlRef/openerHandle focus return, the clear gate.
Extract to a shared base/hook (`OverlayControlBase extends QuestionElementBase`
or `useOverlayControl`) consumed by dropdown, tagbox, rating-dropdown,
buttongroup-overflow. Do it AFTER 2.3/2.4 merge, with their green suites as the
regression net (invariant: keep every dropdown+tagbox test green through the
extraction — that is the safety contract).

### Part A — RatingQuestion
- Detect `renderAs==="dropdown"` (equivalently `dropdownListModel` present).
- Branch: render `OverlayControl(dropdownListModel)` instead of the rating item
  rows. Collapsed display = selected rate value's text.
- Rows resolve through the existing sv-list/ListPicker (no new item component).
- Reactive: `renderAs` is a core property → base subscription re-renders on flip.

### Part B — ButtonGroupQuestion
- Keep the 2.9 horizontal ScrollView for the button row (renderAs==="default").
- **RN measurement seam (replaces ResizeObserver):** ScrollView
  `onContentSizeChange` → contentWidth (= required); `onLayout` → viewportWidth
  (= available). When both known and they cross, call
  `question.processResponsiveness(contentWidth, viewportWidth)` — reuse core's
  decision (invariant 6; do not reimplement the threshold).
- On `renderAs==="dropdown"` → render `OverlayControl(dropdownListModel)`; else
  the button row.
- **Loop guard:** processResponsiveness flips renderAs → re-render → re-measure.
  Only call it when the measured pair would actually change the decision (core's
  own compare is idempotent — verified — but still gate on "numbers known &&
  changed" to avoid onLayout churn / a measure↔setState loop).

### Registration / kitchen-sink / DIFFERENCES
- No new question-type descriptor rows (both are variants of already-registered
  `rating` / `buttongroup`). Update the `sv-rating-dropdown-item` deferral note.
- kitchen-sink: add a `rating renderAs:dropdown` + a `buttongroup` with enough
  choices to force overflow in a narrow container.
- DIFFERENCES: rating-dropdown + buttongroup-overflow both use the 2.1 overlay
  sheet; buttongroup overflow is width-measured via ScrollView content-vs-viewport
  (vs web ResizeObserver); `:focus-within` ring has no RN analog (already noted
  for 2.9).

## R1. Dispatch (CONFIRMED sound by Codex)
- rating-dropdown: register a RN RendererFactory route ("rating","dropdown" ->
  our dropdown-style renderer) via the EXISTING descriptor renderer-route

  row. Codex confirms skipping sv-rating-dropdown-item + not porting the

## R2. ButtonGroup measurement (CRITICAL fix — both peers)
- Do NOT unmount the button row when compact — that kills onLayout/onContentSize
  and the flip-back can never fire.
- Always-mounted OUTER View measures availableWidth (onLayout) in BOTH modes.
- Cache the intrinsic required width from the default button-row branch
  (onContentSizeChange); keep comparing the CACHED required width against the
  live available width while compact.
- Feed (cachedRequired, availableWidth) to core's processResponsiveness; let CORE
  decide. Do NOT reimplement the threshold.

## Implementation amendments (2.5b review-findings fix pass)

- **`liveAvailableWidth` survives a question prop swap** (narrows the
  "reset caches ... on question-prop change" TDD line): only the
  REQUIRED-width cache and the changed-pair dedupe reset on swap. The
  wrapper's measured available width is a property of the HOST GEOMETRY,
  not of the question, and RN re-fires `onLayout` only when layout
  actually CHANGES — resetting it on swap would deadlock measurement for
  the new question (no event would ever repopulate it). The RESET half is
  pinned by the "question prop swap resets the cached required width"
  test; the SURVIVAL half by "the LIVE available width SURVIVES a
  question swap" (v2 review H1 — a content-only event after the swap
  flips compact using the pre-swap width; mutation-verified: nulling
  `liveAvailableWidth` in `syncMeasurementTarget` turns exactly that test
  red).
- **The measuring row stays MOUNTED (hidden) while compact** (strengthens
  R2): `renderAs` is serialized, so a question can MOUNT already compact
  with no cached required width — a collapsed-only tree would never emit
  a content-size event and flip-back would be permanently impossible.
  The row renders inside an always-mounted measure host that, while
  compact, is absolute-positioned, `opacity: 0`, `pointerEvents: 'none'`,
  and a11y-hidden on both platforms. This also keeps the REQUIRED cache
  fresh while compact (content changes re-measure through the hidden
  row). CORRECTION (v2 review, L3): the hiding props gate on VM
  PRESENCE, not on `renderAs` — a question that MOUNTS already compact
  (persisted `renderAs`, VM unbuilt) renders the row VISIBLE,
  interactive, and a11y-exposed for the first frame(s), until the first
  measurement event (or the post-swap deferred ensure below)
  materializes the VM. Deliberate render-purity consequence; kept as-is
  and documented in DIFFERENCES.md (buttongroup).
- **VM construction is render-pure**: render, `getStateElements`, and
  `getOverlayPopup` read only the NON-CREATING `dropdownListModelValue`
  backing field (watchlisted). The CREATING `dropdownListModel` getter is
  touched only outside render — by core's own `processResponsiveness`
  flip path, or by `ensureCompactViewModel` in the measurement handlers
  for the mount-already-compact case (event context: construction fires
  core property notifications, which must land in neither render nor the
  mount-commit window).

## Implementation amendments (2.5 external-review round: Codex gpt-5.6-sol@max adversarial pass)

- **StrictMode lifecycle latches (C1 rating / C2 buttongroup, FIXED)** —
  React 19 StrictMode (dev) replays the class mount lifecycles on the
  SAME instance (componentDidMount → simulated componentWillUnmount →
  componentDidMount). The deferred-ensure machinery's `ensureUnmounted`
  (rating) / `swapEnsureUnmounted` (buttongroup) flags were one-way
  latches set in willUnmount and never reset, so after the replay every
  deferred ensure bailed forever: the rating collapsed control never
  materialized (permanently inert in StrictMode dev) and buttongroup's
  swap-to-already-compact recovery was dead. Fix: clear the latch in
  `componentDidMount` (every (re)mount). The schedule-dedup flag stays
  untouched — a still-pending microtask re-checks all conditions itself,
  so both replay interleavings (microtask before or after the remount
  didMount) converge. Pinned by real `<React.StrictMode>` renders plus
  explicit didMount→willUnmount→didMount sequence tests (both observed
  red pre-fix).
- **Rating overlay item descriptions (C3, CONFIRMED + FIXED; corrects a
  reconciled-round note)** — probe-verified (2026-07-19): in dropdown
  mode core stamps component `'sv-rating-dropdown-item'` on EVERY list
  action, and the min/max actions carry a `description`
  LocalizableString (minRateDescription/maxRateDescription). The earlier
  reconciled note ("that key is web's COLLAPSED selected-value display,
  not an overlay row — register nothing") was WRONG about the overlay
  half: upstream registers `sv-rating-dropdown-item` as the LIST ROW
  content (title + optional description,
  survey-react-ui components/rating/rating-dropdown-item.tsx), so the RN
  title-only ListPicker fallback silently dropped the descriptions. Fix:
  `RatingDropdownItemContent` registered through the descriptor table's
  element route (content-only, same split as `ListItemGroupContent`;
  title-only when no description). The collapsed-display half of the old
  note stands — the RN collapsed control still renders inside
  RatingDropdownQuestion via selectedItemLocText (R7).
- **Buttongroup swap re-measure (C4, FIXED)** — `syncMeasurementTarget`
  clears the required-width cache on question swap, but the retained
  measuring ScrollView emits NO new `onContentSizeChange` when the
  incoming question's content is geometrically identical — a
  swapped-in persisted-compact question could never re-establish its
  required width, so flip-back was permanently dead. Fix: the measure
  host is `key`ed to question identity (`question.id` — instance-unique;
  names can collide), so a swap REMOUNTS the ScrollView and RN's
  fresh-mount contract guarantees exactly one initial content-size
  event. The test models that contract honestly: it fires the post-swap
  content event ONLY IF the measuring ScrollView's composite instance
  actually changed.
- **Invalid measurement samples INVALIDATE their cache (C5, FIXED —
  supersedes the earlier "non-finite/zero widths are ignored, never fed
  to the adapter" framing)** — dropping an invalid sample while KEEPING
  the previous cached value let the OTHER dimension's next event run
  `processResponsiveness` with stale geometry (rotation-style
  transitions report zero widths). Now an invalid sample CLEARS the
  corresponding cache (`liveAvailableWidth` / `cachedRequiredWidth`) AND
  the changed-pair dedupe, pausing processing until a fresh valid sample
  arrives. Deadlock-safe: invalid→valid is a real layout change, so RN
  re-fires `onLayout`/`onContentSizeChange` with the valid width —
  processing always resumes. (Invalid samples are still never fed to the
  adapter; what changed is that they now also invalidate.)
- **Render-purity scope (C6 — backport DONE, 2.5fu)** — the
  non-creating-backing-field discipline (render/`getStateElements`/
  `getOverlayPopup` read only `dropdownListModelValue`) now covers ALL
  FOUR consumers. This pass covered the TWO 2.5 consumers
  (rating-dropdown and buttongroup-compact); the 2.5fu follow-up
  backported it to `DropdownQuestion` and `TagboxQuestion` (2.3/2.4),
  which previously read the CREATING `dropdownListModel` getter during
  render — see the "2.5fu render-purity backport" amendment section
  below.
- **survey-core peer floor verified for the private/protected bindings
  (C7)** — the declared peer range is `>=2.5.32 <2.6.0`; verified
  against the survey-library reference checkout at tag v2.5.32 that both
  watchlisted members exist there: `Question.processResponsiveness`
  (protected, question.ts) and the `dropdownListModelValue` backing
  field (question_rating.ts, question_buttongroup.ts, question_dropdown
  base). The floor is safe as declared; no tightening needed.

## Implementation amendments (2.5fu render-purity backport: dropdown + tagbox)

The C6 follow-up: the non-creating-backing-field discipline now covers
ALL FOUR OverlayControlBase consumers. External review had confirmed
`DropdownQuestion.renderElement` and `TagboxQuestion.renderElement`
read the CREATING `question.dropdownListModel` getter during render
(core constructs + subscribes the VM inside a React render pass), and
the base's DEFAULT `getOverlayPopup` was also creating.

- **Base default `getOverlayPopup` is now NON-creating** — it reads
  `question.dropdownListModelValue?.popupModel ?? null`. With that,
  rating's and buttongroup's formerly-overriding non-creating
  implementations became identical to the default and were deleted
  (behavior unchanged; the base owns the read for all four consumers).
- **Dropdown/tagbox transfer the RatingDropdown pattern verbatim** —
  render + `getStateElements` read only `dropdownListModelValue`; the
  CREATING getter is touched exclusively by the deferred
  (one-microtask) ensure scheduled from componentDidMount/DidUpdate
  (StrictMode-safe: the `ensureUnmounted` latch resets on every
  (re)mount — C1 pattern; unmount-guarded; schedule-deduped;
  forceUpdate only if the getter actually materialized).
- **Pending frames render real question-level content, not an inert
  View** (unlike rating, whose collapsed value needs the VM's
  `placeholderRendered`): probe-verified (2026-07-19, `_setIsTouch`
  env) that `readOnlyText`, `selectedItemLocText`, `value`,
  `locPlaceholder`, `selectedChoices`, `renderedValue`, `isEmpty`,
  `isInputReadOnly`, `isShowingChoiceComment`, `isOtherSelected`,
  `processedTitle`, and `a11y_input_ariaLabel` are all NON-creating —
  but `showSelectedItemLocText` IS creating once a value is set (its
  fold short-circuits on empty, then evaluates
  `dropdownListModel.canShowSelectedItem`), so the pending frames avoid
  it. Dropdown's pending frame renders the question-level R7 fold
  (readOnlyText → selectedItemLocText → raw value → placeholder),
  non-interactive for the single tick; tagbox's renders chips +
  placeholder VM-free, so a committed value never blinks.
- **Select mode never constructs** — dropdown's core getter already
  gates on `useDropdownList` (`renderAs !== 'select'`); tagbox's core
  getter has NO gate, so the renderer discipline is the only thing
  keeping a select-mode mount construction-free (pinned by test). The
  tagbox select fallback's placeholder consequently can no longer ride
  a VM state-element subscription on a select-mode MOUNT — it renders
  through the LocString viewer (self-subscribing), keeping the r2 #2
  staleness pin green without weakening it. A lingering VM after a
  runtime overlay→select flip stays a state element as before.
- **Pins** (all red-first observed where marked): per-consumer
  'constructs the lazy VM exactly once and NEVER during a render pass'
  (D2 reactRendering guard — RED pre-fix: `constructed-during-render`),
  tagbox 'a select-mode MOUNT never constructs the VM' (RED pre-fix),
  StrictMode replay + explicit didMount→willUnmount→didMount sequence
  tests (C1 mirrors), and pending-frame pins (dropdown value text /
  tagbox chips visible before the first microtask, no select fallback,
  no spurious diagnostic). Existing dropdown/tagbox suites already
  flushed a microtask after render, so no assertions were weakened; the
  shared contract + focusReturn suites stayed green unchanged.
