## Part A — rating `renderAs: "dropdown"`

Verified against survey-core 2.5.33: a `rating` with `renderAs:"dropdown"` has
`getTemplate() === "rating"`, `isDefaultRendering() === true` (so it dispatches
through the SAME template route as normal rating — RatingQuestion, task 1.14),
and gains `question.dropdownListModel` + `question.itemComponent ===
"sv-rating-dropdown-item"`.

So RatingQuestion (1.14) already dispatches per-item through `itemComponent`;
in dropdown mode that key becomes `sv-rating-dropdown-item`, which was
**DEFERRED to M2** at 1.14 (see descriptors.ts rating comment + the
`sv-rating-dropdown-item` dispatch-miss fallback). 2.5 lands it.

**Design:** rather than render the rating item row, dropdown-mode rating shows a
collapsed control that opens the overlay picker (the same overlay bridge dropdown
uses). Approach:
- RatingQuestion detects `renderAs:"dropdown"` (or `question.dropdownListModel`
  present) and renders a dropdown-style control (reuse the DropdownQuestion
  control shape: a Pressable opener bound to `dropdownListModel.onClick`, the
  2.1 overlay bridge registration, combobox a11y with STRING ariaExpanded — copy
  the hardened 2.3 patterns). The list rows are the rate values.
- Register the `sv-rating-dropdown-item` element route so the overlay list's
  item dispatch resolves (element-route descriptor row → a small item component,
  OR the shared ListPicker row already handles it — CHECK whether the rating
  dropdown uses `dropdownListModel.listModel` like dropdown/tagbox, in which case
  no custom item component is needed and `sv-rating-dropdown-item` is only the
  in-control selected display).
- Selected display: the chosen rate value's text.

**Open questions to resolve at design time:**
- Does rating-dropdown drive the SAME `dropdownListModel.listModel` +
  `ListPicker` as dropdown/tagbox (then Part A is mostly "reuse DropdownQuestion
  with rating's value semantics")? Probe `q.dropdownListModel.listModel.actions`.
- Is `sv-rating-dropdown-item` an OVERLAY list-row component or the COLLAPSED
  selected-value display? (The 1.14 deferral comment implies the former.)
- Consider extracting the DropdownQuestion control (opener + bridge + a11y) into
  a shared base both dropdown and rating-dropdown use, to avoid a third copy
  (tagbox already copied it). A shared `OverlayControl` mixin/base is the clean
  move if 2.5 would be a 3rd duplication.

## Part B — buttongroup overflow-to-dropdown

Verified: buttongroup has `dropdownListModel` + `needResponsiveWidth` +
`supportResponsiveness`. Today ButtonGroupQuestion (2.9) renders a
horizontal-scroll button row (DIFFERENCES: "No overflow-to-dropdown until task
2.5"). 2.5 adds: when the container is too narrow for the buttons, swap the row
for the overlay dropdown (`dropdownListModel`), mirroring web's shrink-observer
(`buttongroup-dropdown.tsx`).

**Design:**
- Measure the buttongroup row width (onLayout) vs its content; when it can't fit
  (or `needResponsiveWidth` says so), render the collapsed dropdown control +
  overlay instead of the scroll row.
- Reuse the same overlay control/bridge as Part A / dropdown.
- The threshold: web uses a ResizeObserver; RN uses onLayout width. Keep it
  simple — a narrow-mode switch like the 1.4 narrow row stacking, or measure the
  natural button-row width and compare.
