## Scope (this task only)

add-panel button, a per-panel remove button (with core-driven delete

## Verified model surface (survey-core 2.5.33, `question_paneldynamic.ts`)

- `getType()` → `"paneldynamic"`; `isDefaultRendering()` true (no custom
  renderer) → dispatch to a built-in component. Confirm `getTemplate()` return
  during TDD to pick the descriptor route (expected `"paneldynamic"`, template
  route like every other supported type).
- **Iteration source: `renderedPanels`** (`Array<PanelModel>`) — NOT `panels`/
  `visiblePanels`. In LIST it equals `[].concat(visiblePanels)`. Priming
  (`onFirstRendering()`/`locStrsChanged()`) happens in `updateRenderedPanels()`
  BEFORE the setter assigns — the setter itself does not prime (codex r nit).
  Using `renderedPanels` now keeps 2.8b/2.8c (single-panel `[currentPanel]`) a
  pure branch swap.
- **Keys: `panel.id`** (stable across add/remove; index keys corrupt input state
  on mid-array insert via external `addPanel(index)`). The key MUST sit on the
  OUTERMOST mapped node (the per-panel wrapper `View`/`Fragment` that also holds
  the remove button) — NOT on a nested `SurveyPanel`, or React reconciles the
  wrapper by position and still transfers native/draft state between panels
  (codex major #3). Note: list-mode `addPanelUI()` appends (`currentIndex` is
  -1), so the mid-insert regression test must use external `addPanel(index)`.
- Add: `addPanelUI()` (applies `canAddPanel`/`canLeaveCurrentPanel` guards).
  Remove: `removePanelUI(panel)` (applies `canRemovePanel` + confirm decision).
  NEVER the raw `addPanel(index)`/`removePanel(value)`.
- Button gating: visibility `canAddPanel` / `canRemovePanel`; enabled-but-shown
  `enableAddPanel` / `enableRemovePanel` (distinct — disabled ≠ hidden).
- Captions: `locAddPanelText` / `locRemovePanelText` (LocalizableString).
- Empty state: `getShowNoEntriesPlaceholder()` (true when `visiblePanelCount===0`
  and not design mode) → placeholder containing `locNoEntriesText` + the add
  button (otherwise no way to add the first panel).
- `isRenderModeList` = the branch selector (`displayMode==="list" ||
  isSingleInputActive`).

## Delete confirmation (2.2 adapter integration — VERIFIED path)

`removePanelUI(panel)` → `removePanel(panel, isRequireConfirmOnDelete(panel))`.
Confirm only when `confirmDelete` is set AND the panel value is non-empty.
`removePanel(..., true)` calls `confirmActionAsync({ message: confirmDeleteText,
funcOnYes: () => removePanel(), rootElement, cssClass })`. Resolution order
(`utils/confirm-dialog.ts`): `settings.confirmActionFunc` → `confirmActionAsync`
→ fallback `showConfirmDialog` → **`settings.showDialog(...)`**. The merged 2.2
adapter installs exactly `settings.showDialog`, and its code targets this path
(`detachedHandle` comment references upstream `showConfirmDialog`'s
`getActionById('apply'/'cancel')`; no-host fail-safe calls `onCancel()` — never
auto-confirms a destructive delete).

**The renderer does NOT build/show the dialog.** It calls `removePanelUI(panel)`
and lets survey-core dispatch through `settings.showDialog` → the adapter →
OverlayHost. Reduces the renderer's delete responsibility to one method call.

## Nested panel content — reuse the existing composition

Each panel is a full `PanelModel`. Render it via the EXISTING
`SurveyPanel`/`SurveyPanelBase` composition (the same one pages/static panels use):
`SurveyPanelBase.renderRows` → `panel.visibleRows` → `SurveyRow` →
`row.visibleElements` → per-element dispatch. So each dynamic panel maps to one
`<SurveyPanel element={panel} …/>`. No new nested-rendering code; no panel-value
handling in the renderer (each panel's `panel.data` proxies value into the
question's array — the renderer must not read/write panel values).

## Reactivity (class-based SurveyElementBase binding)

Structural changes are delivered by THREE single-assignment callback fields
(overwrite semantics; not multicast):

- `panelCountChangedCallback` — add / remove / `panelCount` set / rebuild.
- `renderModeChangedCallback` — `displayMode` set.
- `currentIndexChangedCallback` — `currentPanel` set (carousel/tab; harmless to
  subscribe in LIST, forward-compatible with 2.8b/2.8c).

**Retarget-safe lifecycle (codex major #2 + gemini #2 — both peers).** A naive
mount-assign / unmount-reset only works for one fixed question instance, but this
project retargets subscriptions when the model prop changes
(`SurveyElementBase.componentDidUpdate` diffs `getStateElements()` — verified
`src/reactivity/SurveyElementBase.tsx:177`). So:
- Use ONE stable bound handler per callback (a class field, not a fresh closure)
  that calls **`this.setState`** (functional updater — bumps a render counter),
  NOT `forceUpdate` (upstream uses `setState`; `forceUpdate` contradicts the
  callback-to-`setState` architecture, codex major #2).
- Store the currently-bound question. Attach in `componentDidMount`; in
  `componentDidUpdate`, if the question identity changed, DETACH from the old
  question and ATTACH to the new one; in `componentWillUnmount`, detach.
- Because the fields are single-assignment, **clear a field only if it still
  equals this instance's handler** (never blindly null a newer owner).
- Call every superclass lifecycle method (`super.componentDidMount/DidUpdate/
  WillUnmount`) so the generic subscription retarget still runs.

The generic per-`Base` subscription (`getStateElement()` → the question) handles
prop/CSS/loc changes. Do NOT rely on an array-changed event on `panels` for
add/remove — survey-core normalizes through `renderedPanels`/the callbacks.
Ignore the `renderedPanels` animation machinery (DOM/`getWrapperElement`-
dependent, inert without a wrapper). Documented limit: one renderer per question
instance (matches upstream); multiplexing only if concurrent renderers ever needed.

## Component shape

- `PanelDynamicQuestion extends QuestionElementBase`:
  - `getStateElement()` → the question.
  - Non-list `displayMode` (`!isRenderModeList`) in 2.8a → render a
    `paneldynamic-mode-unsupported` fallback View + a deferred diagnostic
    (commit-phase, deduped — same pattern as ImageQuestion/imagepicker), so a
    carousel/tab survey degrades gracefully until 2.8b/2.8c.
  - `renderedPanels.length === 0` + `getShowNoEntriesPlaceholder()` → empty
    placeholder (`locNoEntriesText`) + add button.
  - Else: `renderedPanels.map(panel => (` a per-panel wrapper `View`
    **`key={panel.id}`** (the key lives on THIS outer node, codex major #3)
    containing `<SurveyPanel element={panel} …/>` for the nested content + a
    remove button; the remove button renders ONLY when `canRemovePanel` is true
    (ABSENT at `minPanelCount` — not present-and-disabled), and is
    disabled-but-shown only when `enableRemovePanel === false` (codex major #4)).
    Footer add button: rendered only when `canAddPanel` (ABSENT at
    `maxPanelCount`), disabled-but-shown when `enableAddPanel === false`.
  - **Collapse (codex major #1):** `SurveyPanel.renderContent()` returns null
    when `!panel.renderedIsExpanded`, and its header has NO expand affordance —
    so a panel with `panelsState:"collapsed"`/`"firstExpanded"` would show only
    a title with its inputs unreachable. 2.8a MUST NOT leave content unreachable.
    Decision: the paneldynamic panel wrapper adds a minimal press-to-toggle on
    the panel title (calling `panel.toggleState()`/`panel.collapse()/expand()`)
    so collapsed panels can be opened. Test `expanded`, `collapsed`,
    `firstExpanded` (content reachable in each; remove button behavior on a
    collapsed panel). If the toggle proves out-of-scope for 2.8a, the fallback is
    to force-expand in list mode and document `panelsState` collapse as a 2.8b
    item — but "unreachable content" is never acceptable.
  - a11y: add/remove buttons are Pressables with `accessibilityRole="button"` +
    the loc caption; disabled state announced.
- Styling: theme pipeline via `this.themeContext` recipes (reuse `button`/
  `questionChrome`/`row` recipes; no hardcoded colors — invariant 5/6). A
  dedicated `paneldynamic` recipe only if the button/separator states need one
  (defer; start by reusing `recipes.button`).
