## Verified upstream facts (round-1 corrections applied)

- **RN 0.86 New Architecture supports `boxShadow` style arrays with per-layer inset + spread** (round-1 critical: the draft assumed the legacy single-shadow/no-inset model). Shadows map DIRECTLY — no layer dropping, no inset emulation. Android API-level caveats (outer exact ≥28, inset ≥29) get a documented runtime fallback, not a design workaround.
- Model state → class strings via `CssClassBuilder`; select items via `getItemClass(item)` (fires public `onUpdateChoiceItemCss` — preserved by calling it); text via `getControlClass()`; **question outer root via PUBLIC `Question.getRootCss()`** (round-1 fix — `getCssRoot` is protected; `getQuestionRootCss()` is the inner/type-specific root). Item error excludes warnings; root error includes them (`hasCssError(true)`) — locked via `getRootCss()`.
- **Disabled-family reachability is per-getter, not global**: base select/text getters' disabled branch is dead (`getIsDisableAndReadOnlyStyles` second slot hardcoded false, survey-element.ts:1219) — but e.g. Button Group emits disabled classes directly. The bridge documents reachability per component/getter; live disabled variants kept where a getter actually emits them.
- `itemHover` is an affordance gate (`!disabled && !checked && !designMode`) ANDed with live Pressable state; extracted-token precedence honored (`allowHover && !readOnly`).
- `survey.css` is MUTABLE (consumer overrides merge into the active table, survey.ts:1354-1376) and `onUpdateQuestionCssClasses` mutates per-question class objects — hard-coded defaultCss token maps are NOT sufficient (theme-invariant across the 40 presets, yes; consumer-invariant, no).
- Two 1.5× line-height families (editor-font vs base-font); question-title metrics live in sd-element.scss; number gutter = 40dp flex row replacing web negative-margin.
- **Responsive correction**: `default.m600.scss` is NOT a media query — it defines regular + `--mobile` variable overrides; core itself computes `width < 600` and flips `survey.setIsMobile()`. RN mirrors CORE's mechanism, not CSS's.

## Provider

- **Memoization — exact policy**: on every render, build a canonicalized snapshot of the SUPPORTED ITheme fields (sorted-key serialization of `themeName/colorPalette/isPanelless/backgroundImage*/header/headerView/cssVariables`); stable hash as prefilter, full deep-compare on hash match (collision-safe). Cache `{snapshot, resolved, recipes}`; context identity changes iff the snapshot differs. Mutating a same-reference theme object between renders IS detected (snapshot-based, not reference-based). Every recipe build input (spacingMode excluded — see below) participates in the cache key.
- **Context value**: `{ resolved, recipes, mode: {narrow: boolean, rtl: boolean} }`. `narrow`/`rtl` live in a nested object updated via state so recipes re-select without theme re-resolution.
- **Responsive ownership**: the M1 Survey root owns `onLayout`, classifies `width < 600` as `narrow`, calls `survey.setIsMobile(narrow)` (keeps core's own mobile behaviors in sync) AND passes the same flag to the provider. Named `narrow` — NOT `compact` (core's `isCompact` = panelless; collision avoided).
- **RTL**: derived reactively (I18nManager + explicit prop override for testing), in `mode`.
- **Diagnostics**: 0.6 resolver diagnostics emitted in a post-commit effect via the 0.5 seam, deduped `(code,key,rawValue)` across re-resolutions for the provider's lifetime.

## Shadow mapping (`shadows.ts`)

`toBoxShadow(layers: ShadowLayer[]): BoxShadowValue[]` — direct 1:1 (offset, blur, spread, color, inset). Focus-ring composition = append the ring layer to the variant's shadow array (mirrors web `$shadow-*-reset + ring` composition). Android: API ≥29 full fidelity (inset documented from Android 10); 28 outset-only (inset layers dropped + one diagnostic); <28 elevation heuristic (documented last-resort fallback). All 40 themes' 6 shadow tokens × both platforms run through the mapper in tests for validity (no NaN/negative/unparsed), with a curated snapshot subset.

**Border convention (narrowed)**: real borders remain the convention for FOCUS/ERROR rings on interactive surfaces (stable layout, a11y-visible), but inset-shadow tokens now render as actual inset boxShadow — no fake-border substitution. Recipes compose both: token shadows verbatim + semantic border slots.

## Hybrid bridge (`bridge.ts`) — pure extraction, live schemas

1. Component calls the same public getter upstream calls (via facade): `getItemClass(item)` / `getControlClass()` / `getRootCss()`.
2. `extractTokens(classString, schema)` — whitespace tokenization of BOTH sides: the observed string AND each schema entry (multi-class constants like `"sd-item--disabled sd-checkbox--disabled"` expand to their token sets; an entry matches when all its tokens are present).
3. **Schema built from the live `question.cssClasses` object the getter itself used** — cached by a per-object ENTRY SIGNATURE (WeakMap<object, {signature, schema}>; signature = the relevant entries' joined values, recomputed cheaply per lookup and schema rebuilt on mismatch) — round-2 fix: identity-only caching goes stale when `onUpdateQuestionCssClasses` MUTATES the same object. Consumer `survey.css` overrides and mutations both keep working. Canonical default tokens retained as ALIASES so hosts appending stock class names via `onUpdateChoiceItemCss` still flip variants.
4. Returns PURE data: `{ variant: VariantState, unknownTokens: string[] }`. No diagnostics during render: the component queues `unknownTokens` and flushes post-commit (componentDidMount/DidUpdate) through the guarded seam, deduped module-wide via `WeakMap<Question, Set<token>>` (dev-only).
5. Per-getter reachability table shipped with each schema (which variants this getter can actually produce — disabled reachable for buttongroup, unreachable for select items — locked by tests).

Native interaction state (pressed/focused) stays component-owned, combined at selection time; `allowHover && !readOnly` precedence enforced in the selector.

## Recipes — build/select split (round-1 fix)

- `buildRecipes(resolved, buildCtx)` runs at provider (re)build: produces `StyleSheet.create`'d ATOMIC style fragments per component (not blind Cartesian tables) + a discriminated selector map of LEGAL states only. Entry counts published per exemplar in the design fixture (item ≈ 9 fragments/12 legal states; input ≈ 8/9; button ≈ 10/12 incl. action/danger; questionTitle ≈ 5/4) with a build-time budget test (< 5ms per recipe on CI hardware, coarse guard).
- Render path: `selectStyles(recipes.item, variant, mode, pressed, focused)` — array composition of prebuilt fragments; zero object allocation beyond the composed array.
- Metric FORMULAS, not literals: recipes read `resolved` tokens (`3 * t.baseUnit`, editor-font line-height token) so non-default themes flow through. Every metric carries its scss source ref.
- **Metrics fixture is committed BEFORE implementation handoff**: `docs/design/0.7-metrics-fixture.md` — per exemplar: property → formula → scss file:line → default value. (Orchestrator produces it from the research fixture; implementer treats it as spec.)
