# Task 0.7 design — theme-rn: StyleSheet factories + SurveyThemeProvider + hybrid styling bridge (A6 stage 2, A7)

Status: draft for peer review. Owner: orchestrator. Consumes 0.6's `ResolvedTheme` (pure data). Source facts verified against survey-core v2.5.33 default-theme SCSS + css pipeline.

## Verified upstream facts the design rests on

- **Model state → class strings** via `CssClassBuilder.append(name, condition)`; entry points: `getItemClass(item)` (select items), `getControlClass()` (text controls), `getCssRoot`/`getQuestionRootCss` (question roots). Conditions are model predicates: `isItemSelected`, `isReadOnlyStyle` (= readOnly/enableIf, preview excluded), `isPreviewStyle` (`survey.state === 'preview'`), `hasCssError()` — **item form excludes warnings, root form `hasCssError(true)` includes them** (two different predicates, not one boolean).
- **The `disabled` class family is dead code in v2.5.33**: `getIsDisableAndReadOnlyStyles` hardcodes the disabled slot `false` ("useful for the old theme"); every `xxxDisabled` append never fires. ReadOnly is the only live family. (RN variant slot reserved-but-unreachable, documented — not styled.)
- **`itemHover` is an affordance GATE** (`!disabled && !checked && !designMode`), not live hover — must be ANDed with RN's own `Pressable` pressed state, never substituted for it.
- **Host customization hook**: `getItemClass` fires public `survey.onUpdateChoiceItemCss` and returns the possibly-host-mutated string. Bypassing it (re-deriving from raw predicates) silently drops host customization.
- **Web's border/focus mechanism is box-shadow tricks** (inset shadow = fake border; `0 0 0 2px` = focus ring) across decorator/input/button uniformly. RN has no inset shadow.
- **Two distinct 1.5× line-height paths**: `multiply(1.5, --sjs-font-editorfont-size)` (inputs, item labels) vs `calcLineHeight(1.5)` = 1.5 × `--sjs-font-size` (buttons, misc). Identical at defaults (24dp), diverge when themes customize editor font independently — two tokens, never one.
- **Responsive spacing** (`--sd-base-padding` etc.) is set by a `min-width: 600px` media query on the web root (default.m600.scss) — not a theme variable. RN needs a width-based breakpoint primitive.
- Exemplar metrics harvested (scss refs recorded per value; full tables in the research fixture, to be committed as `docs/design/0.7-metrics-fixture.md` by the implementer from this design's appendix refs): checkbox/radio item (24dp decorator, 16dp icon, 12dp row padding, 8dp gaps, 32dp row-mode column gap), input (12×16dp padding, 24dp line-height, 4dp radius, preview = bottom-border-only variant), button (16×48dp padding, weight 600, action/danger variants), question title (16dp/600 in sd-element.scss — NOT sd-question.scss — plus 40dp number gutter replacing the web negative-margin hack).

## Module layout

```
src/theme-rn/
  provider.tsx      — SurveyThemeProvider + useSurveyTheme()/ThemeConsumer (class-component-friendly:
                      contextType consumable — the 0.4 bases are classes)
  shadows.ts        — ShadowLayer[] → RN per-platform mapping
  breakpoints.ts    — width-based spacing modes replacing the m600 media query
  bridge.ts         — class-token extraction (the A7 class-token mapping half)
  recipes/          — per-component style recipes (0.7 ships exemplars: item, input, button,
                      questionTitle; each component task adds its own recipe file alongside its port)
  recipes/types.ts  — Recipe<VariantState> contract
```

## Provider

`SurveyThemeProvider({ theme?: ITheme, children })`: runs 0.6 `resolveTheme` memoized on the `theme` prop reference (+ deep-equal fallback guard for consumers who rebuild the JSON each render — memo key = shallow JSON hash, documented cost), builds the recipe StyleSheet cache, provides `{ resolved, recipes, spacingMode }`. Emits 0.6's `diagnostics` array post-commit (useEffect), deduped across re-resolutions by `(code,key,rawValue)` — the 0.6/0.7 contract. No re-render storm: context value identity stable while inputs unchanged. RTL: provider reads `I18nManager.isRTL` once into context (recipes receive it; A7 makes RTL an M1 foundation — provider carries the bit from day one).

## Shadow mapping (`shadows.ts`)

Input `ShadowLayer[]` (0.6 preserves all layers):
- **Outer layers**: iOS — first outer layer → `shadowColor/shadowOffset/shadowOpacity/shadowRadius` (RN single-shadow model; additional outer layers DROPPED + one diagnostic `theme-shadow-layers-dropped` per distinct input). Android — heuristic `elevation = clamp(round(max(blur/2, offsetY)), 0..24)` from the same first layer; color via `shadowColor` where supported (API 28+; silently best-effort).
- **Inset layers**: DROPPED with diagnostic (`theme-shadow-inset-dropped`) behind the A6 capability flag. The visual role inset shadows play upstream (fake borders) is replaced by the border convention below — recipes never rely on inset shadows.
- Pure function, unit-tested per real theme values (LayeredDark multi-layer, ThreeDimensional offset stacks, Contrast 2px ring).

## The border convention (replaces all three web inset-shadow tricks)

Every focusable surface (choice decorator, input, button) renders a REAL border, always present so layout never shifts: default `borderWidth: 1, borderColor: token or transparent` per recipe; focus → `2dp primary`; error → per-recipe error treatment. One convention, all recipes — no per-component workarounds. Focus ring `0 0 0 2px` shadows in themes map onto this (recipes read the resolved primary, not the shadow string).

## Hybrid bridge (A7 — the load-bearing design)

**Two inputs per styled element, cleanly split:**

1. **Model-derived variant state — via class-token extraction (`bridge.ts`), NOT re-derived predicates.** The component calls the same public getter upstream calls (`question.getItemClass(item)`, `getControlClass()`, `getQuestionRootCss()` — through the facade), then `extractTokens(classString, tokenMap)` splits on whitespace and matches against a per-component token map (`{ 'sd-item--checked': 'checked', 'sd-item--error': 'error', … }` — maps built from the same `defaultCss` block the getter used, source-ref'd). Output: `VariantState` booleans (`checked/readOnly/preview/error/none/selectAll/allowHover…`). Rationale: (a) zero duplication of core predicate logic (invariant 6); (b) `onUpdateChoiceItemCss` host mutations keep working — a host-appended known token flips the variant, unknown tokens are ignored + surfaced once via diagnostic `unknown-style-token` (dev only); (c) the dead `disabled` tokens simply never appear. Handles `undefined` returns (`getControlLabelClass`).
2. **Native interaction state — owned by the component, never by core**: `Pressable` pressed/hovered/focused + RN focus events. Combined in the recipe call: `recipe({ tokens, variant, pressed, focused, rtl, spacingMode })`. `allowHover` gate ANDs with pressed (upstream semantics preserved).

**Recipes** are pure functions `(resolved: ResolvedTheme, ctx: RecipeContext) → named style slots` (e.g. item recipe returns `{container, decorator, icon, label, description}`), pre-composed per variant combination into `StyleSheet.create` tables at provider build time (finite variant space per component; pressed/focused resolved at render by slot selection, not object creation — no per-render allocation). Every metric literal carries its scss source ref comment (A7 rule).

**Breakpoints**: `spacingMode: 'compact' | 'regular'` from container width (`onLayout` at survey root; threshold 600 matching m600), provided via context; recipes consume it where upstream uses the media-query padding vars.

## What 0.7 does NOT do

- No component ports (recipes for item/input/button/questionTitle ship as exemplars + their tests; consumed by M1 ports).
- No renderer changes to 0.4 bases (provider is contextType-compatible; wiring happens in M1's Survey shell).
- No inset-shadow emulation, no per-family shape recipes beyond what tokens carry (family gaps live in recipe metrics per A7).
- No animation (D3 policy from 0.4 stands).

## Test plan (TDD)

1. **Recipe goldens**: for DefaultLight + SharpDark + ContrastLight (spread of families), snapshot each exemplar recipe's full variant matrix output (finite: item ~2⁶ meaningful combos enumerated explicitly, not cartesian-blind).
2. **Bridge extraction**: real class strings from live models via facade (checked/readonly/preview/error/none/selectAll fixtures); host `onUpdateChoiceItemCss` appending a known token flips the variant + unknown token diagnostics once; `undefined` getter return; item-vs-root error predicate difference locked (warning-only question: root error variant on, item error variant off).
3. **Shadow mapper**: real multi-layer/inset/offset theme values → expected RN objects + drop diagnostics; Android elevation heuristic bounds.
4. **Provider**: memoization (same theme ref → same context identity; rebuilt JSON deep-equal → same identity; changed value → new); diagnostics emitted post-commit once across StrictMode double-mount (reuses 0.5 seam contract); RTL bit propagation.
5. **Breakpoint**: threshold behavior + recipes switching spacing values.
6. **ESLint/purity**: recipes import nothing from survey-core (tokens in, styles out); bridge imports only facade getters' TYPES.
