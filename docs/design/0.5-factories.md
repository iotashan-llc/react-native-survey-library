# Task 0.5 design — factories + unsupported-type fallback + runtime-template manifest (A4)

Status: **final** — two llm-pair rounds (codex gpt-5.6-sol; Gemini quota-unavailable). Round 1 `revise` (8 major + 2 minor): all accepted, no held positions. Round 2 found 4 internal defects in the revision itself (descriptor union, element route + split inventories, single-source descriptor ownership, ignored-widget diagnostic path) — all fixed below. Owner: orchestrator. Implements A4. Depends on 0.4 (`QuestionElementBase`).

## Upstream shape (v2.5.33, verified + corrected in review)

- `ReactQuestionFactory` (reactquestion_factory.tsx:3-25) and `ReactElementFactory` (element-factory.tsx:3-27): near-identical `HashTable<string, (props)=>JSX.Element>` singletons — but NOT identical: only ElementFactory exposes `isElementRegistered`. Missing key → `null`.
- **Question dispatch has TWO routes** (reactSurvey.tsx:284-291): key = `question.isDefaultRendering() ? question.getTemplate() : question.getComponentName()`; props `{question, isDisplayMode: question.isInputReadOnly, creator}`. `getComponentName()` is driven by survey-core's global `RendererFactory` (rendererFactory.ts:18-40, question.ts:2910-2918) — upstream components register extra keys like `sv-boolean-checkbox`, `sv-boolean-radio`, `sv-rating-dropdown` for `renderAs` variants. A template-only registry misses this entire route.
- `customWidget` bypasses the factory entirely upstream (reactquestion.tsx:29-38). **RN divergence, explicit**: DOM custom widgets are won't-support; RN never takes that branch — a question with a matched `customWidget` renders via its normal dispatch key (widget ignored) + one diagnostic. Mechanism (round 2: the fallback's diagnostic path can't emit this — a supported question never mounts the fallback): 0.4's `QuestionElementBase` mounted hook checks `question.customWidget` once per question (module-scoped `WeakSet<Question>`, commit phase) and emits `{ code: 'custom-widget-ignored', questionType, name, widgetName }` through the same handler seam. Reading `customWidget` triggers core's widget discovery scan — benign here (diagnostic only, no render behavior keyed on it; the behavior-affecting SCU branch stays unported per 0.4 D3).
- ElementFactory is a separate dispatch system with heterogeneous per-key prop contracts and per-call-site fallback semantics: `survey`/`sv-page` (reactSurvey.tsx:201-210), row fallback to `question` (row.tsx:105-117), `questionErrorComponent` (reactSurvey.tsx:293-300), wrappers (reactsurveymodel.tsx), `LocalizableString.renderAs`, per-question item components (reactquestion_selectbase.tsx:100-124).
- Runtime templates: `QuestionCustomModel.getTemplate() → "custom"`, `QuestionCompositeModel → "composite"` (question_custom.ts:797,1112) — never the ComponentCollection-registered name.
- `SurveyQuestionEmpty` renders `<div />` (not nothing).

## Port

### Registries — `src/factories/QuestionFactory.ts`, `src/factories/ElementFactory.ts`

`RNQuestionFactory` / `RNElementFactory` singletons backed by `Map<string, Creator>` (upstream uses `{}` — prototype-key hazard: `__proto__`/`constructor` registrations would produce false hits or non-function creators; Map kills the class). API mirrors upstream per-factory (`registerQuestion/createQuestion/getAllTypes` vs `registerElement/createElement/isElementRegistered/getAllTypes`); `isQuestionRegistered` added as a documented RN extension (upstream asymmetry noted). Missing key → `null` (API parity — fallback lives OUTSIDE the registry).

### Descriptor table — `src/factories/descriptors.ts`

One declarative table — the SINGLE source of registration truth — as a **discriminated union** on `status`:

```ts
type Descriptor =
  | { status: 'supported'; questionType: string; dispatchKey: string;
      route: 'template' | 'renderer' | 'element'; renderAs?: string;
      component: () => ComponentType<any>; milestone: string }   // component required
  | { status: 'planned' | 'unsupported'; questionType: string; dispatchKey: string;
      route: 'template' | 'renderer' | 'element';
      milestone?: string; reason: string }                        // no component field
```

The registrar walks `supported` rows only: `template` → `RNQuestionFactory`; `renderer` → `RNQuestionFactory` under the renderAs key AND the `renderAs` mapping into survey-core's `RendererFactory` via the facade; `element` → `RNElementFactory` (its keyspace is disjoint from questions — `sv-page`, `questionErrorComponent` values, wrappers, etc.). M0 rows: `empty` (supported/template), `custom`/`composite` (**planned**, M2 task 2.11 — no placeholder components; dispatch miss → fallback + diagnostic; round 1 killed "structural placeholders" as dishonest). Everything else arrives per milestone by adding rows.

### Registration & packaging (A10-compatible, review-corrected)

Component modules export **only components** (side-effect-free, no registration functions — round 2: component-owned `registerX` would compete with the table as a second source of truth). `src/factories/register-all.ts` walks the descriptor table, resolving each supported row's `component` thunk (thunks keep heavy capability imports behind per-component lazy boundaries per A10); `src/index.tsx` imports the registrar. No module-scope self-registration (upstream's pattern + 0.3's `sideEffects` allowlist would let a tree-shaker drop registrations). `sideEffects` array extends to the registrar + root entry (src + lib forms). The packaged-entry suite (0.3's harness) gains an assertion: importing the packed root registers the expected keys.

### Unsupported fallback — `src/components/UnsupportedQuestion.tsx`

Dispatcher contract: `factory.createQuestion(key, props) ?? createUnsupportedQuestion(props, missInfo)`. NOT in the registry — no magic `__unsupported` key (collidable, pollutes `getAllTypes`); customization via a dedicated `setUnsupportedQuestionRenderer()` setter. Extends 0.4 `QuestionElementBase`; renders a neutral `View`/`Text` box (question title + `Unsupported question type: <questionType>`); theme-token styling arrives with 0.7 (review: can't be "theme-styled" before the theme pipeline exists). Never throws (invariant 9).

### Diagnostics — `src/diagnostics.ts`

Payload: `{ code: 'unsupported-question-type', questionType, dispatchKey, template, componentName, name }` (model type ≠ missing key for ComponentCollection/renderAs cases — both included). Once-per-`(question, dispatchKey)` via module-scoped `WeakMap<Question, Set<string>>` — survives StrictMode remount without double-emitting, allows a second emit if the SAME question later misses a DIFFERENT key. Emitted in commit phase (componentDidMount of the fallback), never during render. Consumer handler wrapped in try/catch — a throwing handler logs once and cannot break the non-throwing guarantee (tested). Default handler: `console.warn`.

## Coverage manifest — two classified inventories (review-restructured)

`src/factories/manifest.ts`:

1. **Model-type inventory**: every survey-core question class name → classification `{ status: 'supported' | 'planned' | 'internal-base' | 'unsupported', milestone?, reason? }`. Compared against `Serializer.getChildrenClasses("question", true)` **by name only — no construction** (blind construction is unsafe as an oracle: `canBeCreated` includes creator-bearing internal bases like `textbase`/`nonvalue`; ComponentCollection types share the same global list and constructing them runs consumer callbacks; test order would mutate the set). A new upstream class name = named CI diff, even when it reuses a parent's template key (jsonobject.ts:1191-1201 — the round-0 key-set design could not detect that).
2. **Question-key inventory**: template + renderer-route dispatch keys → owning milestone; exact-checked against `RNQuestionFactory` after importing the package entry (no unexpected, no missing).
3. **Element-key inventory**: element-route keys → owning milestone; exact-checked against `RNElementFactory` separately (disjoint keyspaces — one merged inventory can't exact-match both factories; M0 list may be empty but the gate + ownership ledger exist from day one, per round 1).

Targeted construction still happens where classification says it's safe: entries marked runtime-renderable are instantiated to assert `getTemplate()` matches their descriptor row, disposed in `finally`; `custom`/`composite` exercised via controlled ComponentCollection fixtures added-then-removed in the test.

## Test plan (TDD)

1. Registry API round-trip per factory incl. asymmetric methods; missing key → null; prototype-key registrations (`__proto__`, `constructor`, `toString`) are inert data, never false hits.
2. Dispatch-key honesty: ComponentCollection fixture → `custom`/`composite` from `getTemplate()`, never the registered name; a `renderAs` question dispatches via `getComponentName()` and the renderer-route registration resolves it (both routes).
3. Fallback: miss → `UnsupportedQuestion` rendered, no throw; one supported + one unsupported sibling both render; diagnostic once per (question, key) across StrictMode remount; second key on same question re-emits; throwing consumer handler contained.
4. `empty` renders empty content via its registered component (RN delta vs upstream's `<div/>` documented), not the fallback.
5. Manifest gates: model-type inventory vs live `getChildrenClasses` names; renderer inventory vs both factories; descriptor-table rows classified `supported` all registered; safe-construction template assertions.
6. Registrar: importing package entry registers exactly the renderer inventory; component modules import-side-effect-free (lint or import-probe test).

## Non-goals

- No custom/composite adapters (2.11) — `planned` classification only.
- No theme styling of the fallback (0.7).
- No DOM customWidget path, ever (divergence above).
