## Upstream HTML surface (what must render; verified file:line in research)

Two distinct mechanisms — the design mirrors both:

1. **Markdown-gated LocalizableStrings** (title, description, choice text, labels, error texts…): `hasHtml` is TRUE ONLY if the host app registered `survey.onTextMarkdown` and set `options.html` (survey.ts:7500-7508). Default = always plain escaped text. RN string-viewer (task 1.3, plugs into 0.4's `renderLocString` seam) branches exactly like upstream's `sv-string-viewer`: `hasHtml ? <SanitizedHtml source={renderedHtml}/> : <Text>{renderedHtml}</Text>`. Markdown→HTML conversion itself stays host-side (core ships no parser) — unchanged contract.
2. **Literal-HTML properties** (`html` question via `locHtml.renderedHtml` — its `hasHtml` is always false, use renderedHtml directly; `processedCompletedHtml`/`completedBeforeHtml`/`loadingHtml` incl. the `completedHtmlOnCondition` quirk where item HTML IS markdown-gated while the parent is not): author-supplied real HTML after `processHtml` variable substitution. Rendered through the same `<SanitizedHtml>`.
3. Out of scope, already on won't-support: flowpanel `content`, custom-widget `htmlTemplate` (DOM widgets).

## `<SanitizedHtml>` adapter (component contract in 0.9; component lands with 1.3/M5 tasks)

One internal component wraps `@native-html/render`: props `{ source, baseStyle (theme tokens via 0.7 provider), onLinkPress }`. Lazy-required internally (A10) so apps whose surveys never render HTML don't pay parse cost at startup. All renderer-lib imports confined to this adapter (ESLint boundary like the facade) — swapping the lib later touches one file.

## Sanitizer (A11) — `src/security/sanitize-html.ts`

`@native-html/render` has NO built-in sanitization (its docs say so explicitly). Ours: allowlist sanitizer over **htmlparser2 — already in the dependency tree via `@native-html/transient-render-engine`**; zero new deps (dompurify needs a DOM; sanitize-html drags server-oriented deps).

- Tag allowlist (v1): text/structure (`p div span br hr h1-h6 ul ol li blockquote pre code strong b em i u s sub sup table thead tbody tr td th a img`).
- Attribute allowlist per tag (`href`, `src`, `alt`, `width/height`, `colspan/rowspan`, `style` STRIPPED in v1 — style-attr CSS is a parsing/injection surface; revisit only on demand).
- Always stripped: `script style iframe object embed form input on*` handlers, comments, doctypes, CDATA.
- Every `href`/`src` passed through the URI policy (below); failures remove the attribute + diagnostic.
- Output: sanitized string (parse → filter → serialize), pure function, fuzz-tested against bypass corpus (mXSS-style nesting, entity tricks, uppercase/whitespace scheme obfuscation `jAvAsCrIpT:`, protocol-relative, data: smuggling).
- **Default-on for ALL HTML surfaces including author-supplied completedHtml** — divergence from upstream (which trusts author HTML via dangerouslySetInnerHTML unconditionally). Restrictive-by-default per A11; per-survey opt-out `dangerouslyDisableHtmlSanitization` (name deliberately scary) documented in DIFFERENCES.md with threat rationale (survey JSON often originates server-side/third-party).

## Central URI policy (A11) — `src/security/uri-policy.ts`

- `validateUri(uri, context)` → `{ ok } | { ok: false, reason }`; contexts: `link`, `image`, `choicesByUrl`, `background`, `video`.
- Default scheme allowlist: `https:` always; `http:` allowed but flagged (diagnostic); `mailto:`/`tel:` for `link` only; `data:` for `image` only, restricted to `data:image/(png|jpe?g|gif|webp);base64,` with a size cap (1MB default); everything else (esp. `javascript:`, `file:`, `intent:`, `content:`) denied. Scheme parsing is obfuscation-resistant (trim, lowercase, control-char strip before match).
- Consumer extension: explicit config object (`allowSchemes`, `allowHttp`, `dataImageMaxBytes`, per-context overrides) passed at `<Survey>` level — additive-only for schemes we consider safe-shaped; `javascript:` unoverridable.
- **No auto-navigation anywhere** (A11/invariant 8): link presses surface via `onLinkPress` event with the validated URI; host decides (Linking.openURL is the host's call, not ours). Applies to sanitizer-passed anchors and nav-style core events (`navigateToUrl` surfaces as event — M1 wiring).
- Consumed by: sanitizer (0.9), image question/choices images (M2+), `choicesByUrl` validation BEFORE `new SurveyModel(json)` (module here; wiring in M1 Survey shell — the pre-model timing already recorded in project memory), background image (0.7 owner), expo-video sources (M5).

## Deliverables in 0.9 (implementation task scope)

2. `src/security/sanitize-html.ts` + bypass-corpus tests.
3. `<SanitizedHtml>` adapter + mount smoke under jest (RN preset) proving @native-html/render renders our allowlisted tag set under React 19 — the New-Arch/R19 gate.
4. `@native-html/render` added to peerDependencies + devDependencies; A10 install-line docs updated.
5. ESLint boundary: renderer lib importable only from the adapter.
6. DIFFERENCES.md entries: sanitization-by-default (+ opt-out), no auto-navigation, flowpanel/custom-widget exclusions, WebView escape-hatch note.

## Non-goals

- No markdown parser (host-side, upstream contract).
- No style-attribute CSS support in v1.
- No WebView rendering path.
- No YouTube/iframe embeds (won't-support list).
- String-viewer component itself = task 1.3; html-question component = M5 (both consume this design's adapter).
