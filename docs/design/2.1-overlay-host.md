# 2.1 — Overlay host primitive + list picker (CORE, A9)

Status: DRAFT for llm-pair review. Owner: orchestrator.

## Problem

M2's popup consumers — dropdown, tagbox, rating-dropdown, buttongroup
overflow, paneldynamic tab overflow, and every delete-confirmation flow —
all route through survey-core's popup stack on web:

- `PopupModel` (popup.ts) — pure model: contentComponentName +
  contentComponentData + IPopupOptionsBase (isModal, displayMode,
  onShow/onHide/onCancel/onDispose), `isVisible` @property, `toggleVisibility()`.
- `PopupBaseViewModel` / `PopupDropdownViewModel` / `PopupModalViewModel`
  — DOM view models: coordinate positioning, `initializePopupContainer()`
  which calls `getElement(settings.environment.popupMountContainer)
  .appendChild(...)` — **throws in RN** (documented tripwire,
  environment-stub.test.ts).
- `settings.showDialog(IDialogOptions)` — `undefined` in RN until we
  install one; `confirm-dialog.ts` self-installs a default
  `settings.confirmActionAsync` that routes into `settings.showDialog`
  → throw. paneldynamic/matrixdynamic/file deletes call
  `confirmActionAsync` (question_paneldynamic.ts:1654 etc.).
- `DropdownListModel` (dropdownListModel.ts) — the question-side driver:
  owns a core `ListModel` (list.ts — filtering, focused item, lazy-load
  bookkeeping, selection actions) and a `PopupModel("sv-list",
  { model: listModel })`; web renders the popup content by registry name.

## Decisions

### D1 — RN owns presentation; core keeps ALL popup/list STATE

The DOM view-model layer (`PopupBaseViewModel` family,
`createPopupModalViewModel`) is **never instantiated** in RN — its
container mounting is DOM-fatal and its coordinate positioning is a
documented won't-support. Everything BELOW it is consumed unmodified:

- `PopupModel` — visibility (`isVisible` via the 0.4 reactive base),
  lifecycle callbacks (`onShow`/`onHide`/`onCancel`/`onDispose`),
  content routing (`contentComponentName`/`contentComponentData`),
  modality (`isModal`, `displayMode`).
- `ListModel` — item actions (`renderedActions`/`getVisibleActions`),
  `filterString`, `searchEnabled`(`setSearchEnabled`), `focusedItem`,
  `isAllDataLoaded`, selection via each action's own `action()`.

Invariant 6 applies: never re-derive filter/selection/lazy-load logic.

### D2 — `OverlayHost`: one Modal-based presenter, injectable (A9)

`src/overlay/OverlayHost.tsx` — a class component (0.4 base) rendered by
the Survey shell (inside `LifecycleContext`, after the ScrollView so it
never affects page layout). It presents ONE overlay at a time from a
registration stack (nested popups: last-in wins, exactly one RN `Modal`
mounted — RN nested Modals on Android are unreliable).

- **Presentation**: RN `Modal` (`transparent`, `statusBarTranslucent`,
  `animationType="fade"` for dialogs / `"slide"` for sheets).
- **Two display shapes** mapped from the MODEL (table below):
  - `sheet` — bottom sheet (menu-family modes): full-width bottom-anchored
    panel, drag-free v1 (no gesture dep), backdrop tap = cancel.
  - `dialog` — centered card (modal-family / confirm): backdrop tap does
    NOT cancel when `isModal` (upstream parity: modal popups require an
    explicit button).
- **Keyboard-aware**: `KeyboardAvoidingView` (behavior `padding` iOS /
  `height` Android) wrapping the panel — the list picker's search input
  and dialog content must stay visible.
- **Android back button**: `Modal.onRequestClose` → `model.onCancel` path
  (`PopupModel.cancel()`? — no such method: call our `cancel()` helper =
  `isVisible = false` + fire `onCancel`). Web has no back button —
  RN-specific, documented in DIFFERENCES.
- **a11y**: `accessibilityViewIsModal`, panel `accessibilityRole="none"`
  with labelled title when present; RN `Modal` traps focus natively.
- **Injectable seam (A9)**: `OverlayPresenterContext` — hosts may replace
  the presenter (e.g. gorhom bottom-sheet) by providing
  `{ present(entry), dismiss(entry) }`. Default value = the built-in
  Modal presenter. The registration/stack/bridge layer is presenter-
  agnostic; only the visual container is swappable.

### D3 — display-mode mapping (model → shape)

Upstream `DisplayPopupMode` (popup.ts) collapses to two RN shapes:

| Upstream mode (isModal/displayMode/device) | RN shape |
|---|---|
| `menu-popup` / `menu-overlay` / `menu-popup-overlay` (dropdown lists) | `sheet` |
| `modal-popup` / `modal-overlay` (dialogs, confirm) | `dialog` |

Rule of thumb: `popupModel.isModal ? 'dialog' : 'sheet'`. No coordinate
anchoring ever (won't-support: web popup coordinate models) — a
non-modal dropdown popup that web would anchor under the control renders
as a bottom sheet (the standard mobile pattern; matches what upstream
itself does on `IsTouch` devices via `displayMode: "overlay"`).

### D4 — `PopupBridge`: PopupModel → OverlayHost

`src/overlay/popup-bridge.ts` — `registerPopup(popupModel, host)` returns
an unregister fn. Subscribes `isVisible` (via
`registerFunctionOnPropertiesValueChanged`, the 1.9 pattern — this is a
Base property):

- `isVisible true` → host.present({ shape, title, content }) where
  content = `RNElementFactory.createElement(model.contentComponentName,
  model.contentComponentData)` — same registry-name dispatch as web
  (`"sv-list"` etc.); a miss renders the `element-wrapper-missing`
  diagnostic and an EMPTY panel with a Close action (invariant 9 — never
  a stuck invisible overlay).
- `isVisible false` → host.dismiss(entry) (also fired by backdrop/back:
  those set `isVisible = false` through the model so core's
  `onVisibilityChanged` + question's `processPopupVisiblilityChanged`
  stay authoritative).
- unregister on question unmount: if its entry is presented, dismiss
  FIRST (unmount-with-open contract — same ordering discipline as the
  1.1 two-phase dispose).

### D5 — `RNListPicker` = the `"sv-list"` registration

`src/overlay/ListPicker.tsx`, registered under `sv-list` in
`RNElementFactory` (matching upstream's registry name so
`contentComponentName` dispatch just works). Props: `{ model: ListModel }`.

- `FlatList` over `model.renderedActions` filtered by
  `model.getVisibleActions?.()` semantics — consume core's own visible
  filtering; re-render via 0.4 base subscribed to the ListModel.
- Row = existing `ActionButton`-style pressable but list-item styled
  (new `listItem` recipe; selection state from `action.css`/`selected`
  through the 0.7 bridge — no re-derivation).
- Search: shown iff `model.searchEnabled`; a TextInput bound
  DIRECTLY to `model.filterString` (this is a FILTER, not a question
  value — the 1.9 draft/commit adapter explicitly does not apply; the
  upstream binding is equally immediate).
- Lazy load: `onEndReached` → `model.loadMore?.()` if present, else the
  question-level `choicesLazyLoadEnabled` path stays with 2.3 (the
  dropdown task wires `updateItems`).
- Empty state: `model.emptyMessage` locstring via LocStringViewer.

### D6 — task 2.2 preview (adapter contract this design must not break)

2.2 overrides `settings.confirmActionAsync = (message, callback, options)
=> host.presentConfirm(...)` — the SANCTIONED seam (confirm-dialog.ts
checks it BEFORE falling through to `settings.showDialog`). It never
touches `showDialog`, `PopupBaseViewModel`, or `footerToolbar`.
`settings.showDialog` gets an RN implementation too (componentName
dispatch + Apply/Cancel footer from a core `ActionContainer` we
construct), returning a minimal view-model-shaped object — scoped to
what `IDialogOptions` consumers need, NOT a PopupBaseViewModel clone.
Details in 2.2's own design; 2.1 only guarantees `presentConfirm` /
`presentDialog` primitives on the host.

## Ownership / files

- `src/overlay/OverlayHost.tsx` (+ recipe `overlay` in theme-rn: backdrop,
  sheet, dialog, title fragments — sd-popup SCSS as metric source)
- `src/overlay/OverlayPresenterContext.tsx` (A9 injection seam)
- `src/overlay/popup-bridge.ts`
- `src/overlay/ListPicker.tsx` (+ recipe `listItem`)
- Survey shell mounts `<OverlayHost/>` + provides host via context.
- Watchlist adds: PopupModel.{isVisible, contentComponentName,
  contentComponentData, isModal, onCancel/onHide/onShow, title},
  ListModel.{renderedActions, filterString, searchEnabled, focusedItem,
  emptyMessage}, settings.confirmActionAsync (data).

## Test plan (TDD)

1. OverlayHost: present/dismiss sheet + dialog; one-at-a-time stack;
   backdrop cancels sheet but not modal dialog; onRequestClose (Android
   back) cancels through the MODEL; unmount-with-open dismisses.
2. PopupBridge: isVisible toggles present/dismiss; content dispatched by
   registry name; miss → diagnostic + closable empty panel; unregister
   while open dismisses first; core `onShow`/`onHide`/`onCancel` fire in
   upstream order (pin via spies on the model callbacks).
3. ListPicker: renders ListModel actions; press selects through
   `action.action()` (spy proves core path, no hand-rolled selection);
   filterString typing narrows visible items (core-computed); search
   input rendered iff searchEnabled; focusedItem styling; empty message.
4. Injection: a fake presenter provided via context receives
   present/dismiss instead of the built-in Modal.
5. Recipes: overlay/listItem golden fragments (existing recipe harness).

## Deferred / won't

- Coordinate-anchored popups, pointer arrows (`showPointer`) — never.
- Drag-to-dismiss sheet gesture — needs gesture-handler; v1 backdrop/
  back/cancel only.
- `PopupSurveyModel` (popup surveys) — out of M2 scope.
- Creator-style property-grid dialogs — not runtime.
