## Problem (unchanged)

M2 popup consumers (dropdown, tagbox, rating-dropdown, buttongroup
overflow, paneldynamic tab overflow, delete confirmations) route through
survey-core's popup stack. The DOM view-model layer (`PopupBaseViewModel`
family) mounts DOM containers (`initializePopupContainer` â†’
`settings.environment.popupMountContainer` â€” RN tripwire) and does
coordinate positioning (won't-support). Everything model-level is
consumed unmodified.

**Round-1 verified:** the ONLY model-relevant behaviors living in the
skipped view-model layer are (a) lifecycle sequencing (`onShow` before
present; `model.onHiding()` after hidden â€” which refreshes the inner
ListModel: resets `focusedItem`, and action-dropdown `onHide` clears
`filterString`), and (b) FOOTER construction (Cancel/Apply actions
seeded through `popupModel.updateFooterActions()` â€” the seam tagbox uses
to inject its Done action and revert-on-cancel). Both are ported;
`recalculatePosition` is coordinate-only and safely omitted
(popup.ts:146-148, popup-dropdown-view-model.ts:273-320).

## D2 â€” OverlayHost: stack WITH suspended parents (round-1 critical #2)

`src/overlay/OverlayHost.tsx`. The host owns an ordered **entry stack**;
every entry stays REGISTERED and its content stays MOUNTED while
suspended â€” only the TOP entry is presented inside the single RN `Modal`
(suspended entries render `display:none`-equivalent: kept in the tree,
zero-size wrapper â€” unmounting them would tear down child popup bridges
and ListModel subscriptions).

- **Entry states (round 2): `active` / `suspended` / `dismissing`** with
  GENERATION-scoped transition acknowledgments. `visible` on the
  presenter means "this entry is the ACTIVE one" â€” suspension is NOT
  dismissal: a suspended entry's PopupModel stays `isVisible === true`
  and **suspension never runs `onHiding()`**. Only a `dismissing`
  transition (model-driven hide/cancel, or non-top semantic removal)
  ends with `onHiding()` â€” gated on the presenter's per-entry
  acknowledgment for THAT generation (stale acks from an interrupted
  animation are dropped).
- The DEFAULT presenter issues per-entry acks itself (JS animation/state
  completion) â€” never RN `Modal.onShow`/`onDismiss` (one persistent
  native Modal can't produce per-entry callbacks across top swaps, and
  `onDismiss` is iOS-only).
- Present(child) over an open parent â†’ parent â†’ `suspended` (still
  mounted, model untouched); child â†’ `active`. Hiding the TOP restores
  the parent to `active`. Removing a NON-top entry runs its dismissing
  transition without touching the active entry.
- **Suspended isolation (round 2)**: suspended content wraps in
  `display: 'none'` (React tree preserved) + `pointerEvents="none"` +
  `accessibilityElementsHidden` (iOS) +
  `importantForAccessibility="no-hide-descendants"` (Android); focus
  transfers to the active entry on every swap.
- **Nested group ownership (round 2)**: child popup bridges are owned at
  the LIST-MODEL scope (registered by ListPicker against its ListModel's
  action set), keyed independently of virtualized cells â€” FlatList
  recycling may unmount a group ROW while its child popup is topmost,
  and that must not trigger the unregister-cancel rule. A group row's
  press calls `item.showPopup()` ONLY (no `onItemClick`).
- One RN `Modal` total (Android nested-Modal unreliability); the Modal
  stays mounted across top-entry swaps; `visible` prop reflects stack
  non-emptiness.

## D4 â€” PopupBridge lifecycle (round-1 major #3)

`registerPopup(popupModel, host)`:

- Subscribe via `popupModel.onVisibilityChanged.add(cb)` (remove on
  unregister); on registration, reconcile an ALREADY-visible model
  immediately (upstream does â€” popup-view-model.ts:139-150).
- **Show sequence**: `model.onShow()` once â†’ host presents entry.
- **Hide sequence** (any path â€” model-driven, backdrop, Android back,
  Cancel action): `isVisible = false` fires ONLY `onVisibilityChanged`
  (popup.ts:129-145 â€” round-2 correction: `onHide` does NOT fire there;
  it runs INSIDE `onHiding()`, popup.ts:226-229). Once the presenter has
  acknowledged visual dismissal for that entry generation, call
  `model.onHiding()` EXACTLY ONCE â€” that runs `onHide` and the ListModel
  refresh (focusedItem reset; dropdown filterString clear). Oracle:
  popuptests.ts:1818-1875.
- **Cancel sequence**: `model.onCancel()` THEN `model.hide()` (upstream
  order; the v1 draft had it reversed).
- Backdrop tap (sheet) and Android back (`Modal.onRequestClose`) run the
  CANCEL sequence (documented+tested: tagbox reverts its pre-open value
  through its own onCancel wiring â€” this is the mobile-parity behavior
  the `_setIsTouch(true)` adapter re-enables).
- Unregister-while-visible = SEMANTIC CLOSE (cancel sequence) then
  removal â€” pinned by test (unmount-with-open, same discipline as 1.1's
  two-phase dispose).
- Content dispatch: `RNElementFactory.createElement(contentComponentName,
  contentComponentData)`. **Factory miss** (round-1 minor): the factory
  itself stays silent-null; the BRIDGE reports the existing
  `element-wrapper-missing` diagnostic (deduped per componentName) and
  presents a fallback panel whose single **Close** action runs the HIDE
  sequence (not cancel â€” nothing to revert).

## D5 â€” Footer actions (round-1 major #4)

Every entry gets a footer slot. **Construction order (round-2 critical
fix â€” upstream popup-modal-view-model.ts:27-45):**

1. Build a RAW `IAction[]` seeded per shape â€” sheet: `cancel`; dialog:
   `cancel` + `apply` (locale via `getLocaleString`).
2. `const footer = popupModel.updateFooterActions(raw)` â€” consumers
   (tagbox Done) push RAW IActions into this array; nothing is
   observable yet, so raw pushes are safe.
3. `container.setItems(footer)` on a fresh core `ActionContainer` â€”
   setItems is where Action wrapping + `onPropertyChanged` observation
   attach. (Seeding the container FIRST would hand tagbox an observed
   array whose raw-push throws â€” the v2 bug.)

Apply runs `onApply()`; **`false` keeps the dialog open**. Explicit
Cancel runs the cancel sequence. Container disposed with the entry.
Regression test uses the REAL tagbox footer listener.

## D7 â€” Presenter injection seam (round-1 major #7)

The HOST owns stack state. Injection replaces the visual container only:

```ts
interface OverlayPresenterProps {
  entry: OverlayEntry;        // shape, title, content, footer
  visible: boolean;
  requestHide(): void;        // routes the HIDE sequence via the model
  requestCancel(): void;      // routes the CANCEL sequence via the model
  onDidShow(): void;          // presenter acknowledgment
  onDidDismiss(): void;       // gates onHiding()
}
type OverlayPresenter = React.ComponentType<OverlayPresenterProps>;
```

Provided via `OverlayPresenterContext` (a NEW context â€” never the base
class's inherited `contextType`, which is the theme context). A custom
presenter reports user-initiated closes through `requestCancel` /
`requestHide` ONLY â€” presenters never mutate PopupModel directly.
`onDidDismiss` is what schedules `model.onHiding()`.
