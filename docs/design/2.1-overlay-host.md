# 2.1 — Overlay host primitive + list picker (CORE, A9) — v2

Status: v2 after llm-pair round 1 (sol@max — 2 critical + 7 major + 1
minor, all folded in below). Owner: orchestrator.

## Problem (unchanged)

M2 popup consumers (dropdown, tagbox, rating-dropdown, buttongroup
overflow, paneldynamic tab overflow, delete confirmations) route through
survey-core's popup stack. The DOM view-model layer (`PopupBaseViewModel`
family) mounts DOM containers (`initializePopupContainer` →
`settings.environment.popupMountContainer` — RN tripwire) and does
coordinate positioning (won't-support). Everything model-level is
consumed unmodified.

**Round-1 verified:** the ONLY model-relevant behaviors living in the
skipped view-model layer are (a) lifecycle sequencing (`onShow` before
present; `model.onHiding()` after hidden — which refreshes the inner
ListModel: resets `focusedItem`, and action-dropdown `onHide` clears
`filterString`), and (b) FOOTER construction (Cancel/Apply actions
seeded through `popupModel.updateFooterActions()` — the seam tagbox uses
to inject its Done action and revert-on-cancel). Both are ported;
`recalculatePosition` is coordinate-only and safely omitted
(popup.ts:146-148, popup-dropdown-view-model.ts:273-320).

## D2 — OverlayHost: stack WITH suspended parents (round-1 critical #2)

`src/overlay/OverlayHost.tsx`. The host owns an ordered **entry stack**;
every entry stays REGISTERED and its content stays MOUNTED while
suspended — only the TOP entry is presented inside the single RN `Modal`
(suspended entries render `display:none`-equivalent: kept in the tree,
zero-size wrapper — unmounting them would tear down child popup bridges
and ListModel subscriptions).

- Present(child) over an open parent → parent suspends (still mounted),
  child presents. Hiding the TOP restores the parent. Hiding a NON-top
  entry removes just that entry from the stack.
- Nested subitem rows (`action.hasSubItems`): the row's press affordance
  calls `item.showPopup()`; the child `popupModel` registers through the
  same bridge (hoisted registration — child registration is owned by the
  ROW's bridge hookup, not by the parent list's mounted subtree… but the
  parent stays mounted anyway per above; both guards apply).
- One RN `Modal` total (Android nested-Modal unreliability); the Modal
  stays mounted across top-entry swaps to avoid re-animation churn;
  `visible` prop reflects stack non-emptiness.

## D3 — display shape + native device mode (round-1 major #6)

Shape: `popupModel.isModal ? 'dialog' : 'sheet'` (unchanged), no
coordinate anchoring ever.

**Native device adapter (NEW):** core computes `IsTouch` from DOM
touch/matchMedia — RN classifies as DESKTOP, which flips
DropdownListModel into desktop popup behavior (suppresses search-input
availability rules, disables tagbox mobile cancel-rollback). The facade
applies an idempotent `_setIsTouch(true)` (utils/devices.ts:43-48) at
shim time — pinned 2.5.33 private seam, explicitly WATCHLISTED and
behaviorally tested (short vs >10-item search rules, tagbox
Done/Cancel). Additionally the Survey root subscribes
`survey.onOpenDropdownMenu` and populates the options (`menuType`,
device type, RN `Dimensions`) — core copies the mutated `menuType` back
(survey.ts:8247-8250).

## D4 — PopupBridge lifecycle (round-1 major #3)

`registerPopup(popupModel, host)`:

- Subscribe via `popupModel.onVisibilityChanged.add(cb)` (remove on
  unregister); on registration, reconcile an ALREADY-visible model
  immediately (upstream does — popup-view-model.ts:139-150).
- **Show sequence**: `model.onShow()` once → host presents entry.
- **Hide sequence** (any path — model-driven, backdrop, Android back,
  Cancel action): the model transition runs first (`isVisible=false`
  fires `onHide` internally per popup.ts:129-145 — verify exact order in
  popuptests.ts:1818-1875 while implementing); once the presenter has
  visually dismissed, call `model.onHiding()` EXACTLY ONCE (guarded
  against re-show racing an animation callback) — this is the ListModel
  refresh (focusedItem reset; dropdown filterString clear).
- **Cancel sequence**: `model.onCancel()` THEN `model.hide()` (upstream
  order; the v1 draft had it reversed).
- Backdrop tap (sheet) and Android back (`Modal.onRequestClose`) run the
  CANCEL sequence (documented+tested: tagbox reverts its pre-open value
  through its own onCancel wiring — this is the mobile-parity behavior
  the `_setIsTouch(true)` adapter re-enables).
- Unregister-while-visible = SEMANTIC CLOSE (cancel sequence) then
  removal — pinned by test (unmount-with-open, same discipline as 1.1's
  two-phase dispose).
- Content dispatch: `RNElementFactory.createElement(contentComponentName,
  contentComponentData)`. **Factory miss** (round-1 minor): the factory
  itself stays silent-null; the BRIDGE reports the existing
  `element-wrapper-missing` diagnostic (deduped per componentName) and
  presents a fallback panel whose single **Close** action runs the HIDE
  sequence (not cancel — nothing to revert).

## D5 — Footer actions (round-1 major #4)

Every entry gets a footer slot. On registration the bridge builds a core
`ActionContainer` seeded per shape:

- sheet (non-modal): `cancel` action (locale via `getLocaleString`).
- dialog (modal): `cancel` + `apply`.

…then calls `popupModel.updateFooterActions(container.actions)` and
renders THE RETURNED ARRAY (that seam is where tagbox injects Done and
consumers mutate). Apply runs `onApply()`; **`false` keeps the dialog
open**. Explicit Cancel action runs the cancel sequence. The container
is disposed with the entry.

## D6 — RNListPicker = `"sv-list"` (rewritten per round-1 critical #1)

Props `{ model: ListModel }` (registered as `sv-list`).

- Rows: `model.renderedActions.filter((i) => model.isItemVisible(i))`.
- Row COMPONENT dispatch: `item.component || model.itemComponent` through
  `RNElementFactory` when set (subitem groups arrive as
  `sv-list-item-group` — v1 registers a group row that renders the
  label + chevron and calls `item.showPopup()` on press); default row
  otherwise.
- Row state: `model.isItemDisabled(item)` / `isItemSelected(item)` /
  `isItemFocused(item)` → recipe variant + a11y state. Press →
  `model.onItemClick(item)`.
- Row subscribes BOTH the ListModel and its item (0.4 base,
  getStateElements → [model, item]).
- Search box: rendered iff `model.showFilter`; TextInput bound to
  `model.filterString` (immediate, not draft/commit).
- Lazy load (round-1 major #5): **no `loadMore` (does not exist)**.
  `onEndReached` → `model.loadingIndicatorVisibilityObserver?.(true)`
  where the OWNER (DropdownListModel, wired in 2.3) registered its
  `updateQuestionChoices` callback; gated on `!model.isAllDataLoaded`
  plus an in-flight flag (owner has no concurrent-request guard).
  Loading row rendered while `!isAllDataLoaded` (upstream's indicator
  row analog).
- Empty state: `model.emptyMessage` in a plain `Text`.
- a11y: container `accessibilityRole="menu"`? — NO: core's `listbox`
  has no RN role; map container → `role="list"`, rows →
  `accessibilityState {selected, disabled}` with
  `accessibilityRole="menuitem"`… **decision**: container `"list"`,
  row `"menuitem"` + selected/disabled state (closest RN vocabulary;
  documented in DIFFERENCES). `keyboardShouldPersistTaps="handled"` on
  the FlatList (first-tap-while-keyboard-open swallowing).

## D7 — Presenter injection seam (round-1 major #7)

The HOST owns stack state. Injection replaces the visual container only:

```ts
interface OverlayPresenterProps {
  entry: OverlayEntry;        // shape, title, content, footer
  visible: boolean;
  requestHide(): void;        // routes the HIDE sequence via the model
  requestCancel(): void;      // routes the CANCEL sequence via the model
  onDidShow(): void;          // presenter acknowledgment
  onDidDismiss(): void;       // gates onHiding()
}
type OverlayPresenter = React.ComponentType<OverlayPresenterProps>;
```

Provided via `OverlayPresenterContext` (a NEW context — never the base
class's inherited `contextType`, which is the theme context). A custom
presenter reports user-initiated closes through `requestCancel` /
`requestHide` ONLY — presenters never mutate PopupModel directly.
`onDidDismiss` is what schedules `model.onHiding()`.

## D8 — Keyboard / back / a11y (round-1 major #9)

- `KeyboardAvoidingView` (padding iOS / height Android) + safe-area
  insets on the sheet; keyboard offset documented.
- Focus ownership: Modal `onShow` → focus search input if `showFilter`,
  else the panel; dismissal → RN gives focus back naturally, plus we
  re-focus the OPENER's ref when the consumer provides one (2.3 passes
  the dropdown control ref).
- iOS `onAccessibilityEscape` → cancel sequence. `onDismiss` is iOS-only
  — never used for lifecycle (the presenter acknowledgment protocol
  covers it).
- Android: back → cancel; underlying content hidden by the Modal itself.
- Roles per D6; dialog panel `accessibilityViewIsModal` (iOS) +
  `aria-modal`.
- REAL-DEVICE tests (example app, agent-device): keyboard selection,
  hardware back, VoiceOver escape, focus restoration — jest cannot see
  these (repo jest ignores the example app).

## D9 — 2.2 preview (round-1 major #8; full contract in 2.2's design)

Survey's host wins; refcounted; unmount restores the previous), ONE-TIME
installation that preserves pre-existing consumer-assigned seams
(`confirmActionFunc` precedence honored — it wins over
