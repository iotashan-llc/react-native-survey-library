# Task 0.4 design — reactive base classes port (A3)

Status: **final** — two llm-pair rounds (codex gpt-5.6-sol@max; Gemini quota-unavailable). Round 1 `revise` (2 critical + 8 major + 1 minor): all accepted except the render→commit gap, which was held with mitigations. Round 2: codex **empirically reproduced** the gap as a deterministic committed-stale-UI tear on installed React 19.2.3 (sibling `componentDidMount` mutates the model after render, before Survey's subscribe; RN 0.86 always requests a concurrent root — RendererImplementation.js:28). Held position withdrawn — closed by the mount reconcile pass (below). Round-2 tightenings (functional `__svRev` updater, partial-overlap registry diff test, spy-based test 9, captured-pair hook ordering) all incorporated. Owner: orchestrator. Implements A3. Source refs: survey-react-ui v2.5.33 `src/reactquestion_element.tsx`, survey-core `src/base.ts`.

## What is being ported

survey-react-ui binds React to survey-core via one class-component base, `SurveyElementBase<P,S>` (reactquestion_element.tsx:7-178): each component instance subscribes to one or more survey-core `Base` models; model mutations synchronously fire callbacks that trigger re-render. No hooks, no store (A3). We port the *mechanism* — with three deliberate, documented divergences where upstream's implementation violates React 19 contracts (render-phase side effects). Behavior-fidelity, not bug-for-bug fidelity.

### The correct survey-core API (three lookalikes — use the third)

| API | What it is | Use? |
|---|---|---|
| `onPropertyChanged` EventBase (base.ts:335) | public documented event; SKIPPED while `isLoadingFromJson` | no |
| `onPropertyValueChangedCallback` instance method (base.ts:893) | single-slot core-internal extension point — core itself assigns it (composite wrappers question_custom.ts:819, grid columns panel.ts:315). UI layers must never override it (would clobber core's own wiring) | no |
| `addOnPropertyValueChangedCallback`/`removeOnPropertyValueChangedCallback` → private `onPropertyValueCoreChanged` EventBase (base.ts:1452-1466) + `addOnArrayChangedCallback`/`removeOnArrayChangedCallback` (base.ts:1435-1448) | lazily-created multi-subscriber events, fired on raw hash writes regardless of load state. **What survey-react-ui actually uses** (:165-177) | **yes** |

Properties that shape the design: these two events are NOT in `eventList`, so `Base.dispose()` does NOT clear them (base.ts:376-396) — unmount unsubscribe discipline is the only leak prevention. `EventBase.add/remove` dedupe via `hasFunc` → remount idempotent. The array callback does NOT pass through `canUsePropInState` upstream (:156-164) — kept as-is, stated explicitly.

## Deliberate divergences from upstream (React 19 safety)

**D1 — committed subscription registry; `shouldComponentUpdate` is side-effect-free.** Upstream unsubscribes inside SCU and resubscribes in componentDidUpdate (:52-58, :34-42). SCU is a render-phase method: React 19 may invoke it speculatively, double-invoke it under StrictMode, or discard the render without a matching commit — leaving a committed component unsubscribed or events missed. Port instead: an instance `subscribedElements: Base[]` registry mutated ONLY in commit-phase lifecycles — componentDidMount subscribes `getStateElements()`; componentDidUpdate diffs registry vs fresh `getStateElements()` (new props → model-swap retarget), unsubscribing dropped elements and subscribing added ones; componentWillUnmount unsubscribes all. SCU returns `true` (plain; see D3 for the omitted freeze/thaw). Model-swap semantics preserved (upstream's implicit swap-by-resubscribe becomes an explicit diff).

**D2 — render guard is exception-safe.** Upstream increments the model-scoped `reactRendering` counter before `renderElement()` and decrements after, no try/finally, re-reading `getRenderedElements()` for the decrement (:59-94). A throw/suspend strands the shared counter positive → every observer of that model silently stops re-rendering; membership change during render corrupts counts. Port: capture the rendered-element list ONCE, increment, `try { renderElement() } finally { decrement same captured list }`. Guard placement stays ON THE MODEL (`stateEl.reactRendering`, shared across observers — moving it to the component reintroduces update loops). Mutation-during-render remains *defensive* (guard prevents loops) — it is NOT sanctioned component behavior; renderer components must not mutate the model from render or mount lifecycles (renderer invariant, see C2 note).

**D3 — dead/web-only surface not ported.**
- `allowComponentUpdate`/`denyComponentUpdate` freeze/thaw (:43-58): zero call sites in all of survey-react-ui (verified by grep — definition only). Omitted (YAGNI). If a later port needs it, it must retarget the D1 registry on thaw (upstream's version leaks a stale subscription if props swap models while frozen) — recorded here so it isn't reintroduced naively.
- `customWidget.isNeedRender/isDefaultRender` SCU branches (:238-247): reading `question.customWidget` triggers global DOM-widget discovery (question.ts:1274) and `isNeedRender` is web-renderer state written by `SurveyCustomWidget` (custom-widget.tsx:10). A host app that registered a DOM widget could suppress RN updates for a widget RN never renders. Not ported; RN custom rendering goes through the 0.5 factory registry with its own contract.
- The rerendered-event dance (`enableOnElementRerenderedEvent`/`disable…`/`afterRerender`, :107,:32-41): activates core's DOM-oriented animation handshake. The gate is a single boolean, not a refcount (base.ts:1675) — with two observers of one model, one unmounting disables the survivor's events (inherited upstream bug); `afterRerender()` can synchronously advance animation state during commit (animation.ts:239). **RN policy: core animation handshake stays OFF in v1** — no enable/disable calls; `animationAllowed` reads false; core animation paths inert. RN animations are owned by per-component style recipes (A7, reanimated). A native animation bridge (refcounted ownership) is a deferred task if core-driven transitions are ever wanted. Ledger entry added.

**State shape (collision-proofing, from review):** `propertyValueChangedHandler` does NOT write model property names into component state (upstream `setState({[key]: value})` invites collisions with subclass state — e.g. a draft/commit adapter's `value` field being clobbered by a model notification, worse under RN New-Arch automatic batching). Port: gate on `canUsePropInState(key)` + `!isRendering` as upstream, then bump a reserved internal counter via the **functional updater** — `this.setState(s => ({ __svRev: ((s as any)?.__svRev ?? 0) + 1 }))` — object-form would lose increments under batching, and the `?? 0` fallback survives a subclass state initializer that omitted the field. `changedStatePropNameValue = key` recorded as metadata for `renderElement()`. Render logic re-reads live model getters (unchanged). Subclass state is never touched by the base. Tested with two synchronous notifications (both increments must land).

**D4 — mount reconcile pass (closes the render→commit gap).** Render reads live model getters; subscription lands in componentDidMount. A mutation in between is missed — round 2 reproduced this deterministically on React 19.2.3: a preceding sibling's `componentDidMount`/`useLayoutEffect` writes the model after the tree rendered but before Survey's cDM subscribes → committed stale UI, in both sync and concurrent roots. App code can legally do this; a consumer-facing "don't mutate on mount" invariant is unenforceable. Fix: in componentDidMount, immediately after subscribing, bump `__svRev` once (and again in componentDidUpdate for registry-diff-ADDED elements only — the swap case). Cost analysis: all mount cDMs run in one commit phase, so their setStates batch into ONE extra render pass for the whole tree, flushed before paint — no flicker, no per-component 2x commits; the reconcile render re-reads live getters, so any pre-subscribe mutation is picked up. Steady-state updates have no gap (subscription persists across re-renders). This supersedes round 1's rejected per-component blanket refresh (the objection was per-component cost; batching makes it one pass). A3's `useSyncExternalStore` escape hatch remains for any tear test 11 reproduces AFTER this fix.

## Port map

### `src/reactivity/SurveyElementBase.tsx`

- `getStateElements()`/`getStateElement()` override points (default `[]`).
- Lifecycle per **D1** (commit-phase registry). Callbacks per **State shape** above; array callback identical minus the (upstream-absent) prop filter.
- Render pipeline: `canRender()` → guard per **D2** → `renderElement()` → `wrapElement()`; `changedStatePropNameValue` cleared at render end.
- Duck-type guard for subscribable elements checks the APIs actually called (`typeof el.addOnPropertyValueChangedCallback === 'function'`), not upstream's `iteratePropertiesHash` probe (:144).
- No rerendered-event calls (**D3**).
- `renderLocString()` base helper: abstract seam in 0.4 — returns the fallback `<Text>{locStr.renderedHtml}</Text>` until M1's LocalizableString renderer (task 1.3) plugs in; every later component port calls through this seam. Explicit deferral, not an omission.

**afterRender-family ban, precise rationale (corrected in review):** two distinct paths. `afterRenderQuestionElement` (question.ts:1308) only forwards `onAfterRenderQuestionInput` — skipping it loses that event, nothing else (no generic rendering/lazy-render state machine depends on it; lazy rendering is `QuestionRowModel`'s, panel.ts:40). `Question.afterRender` (question.ts:1312) is a separate path that reaches responsiveness/ResizeObserver — banned by the 0.3 ledger. Question SUBTYPES hang real setup on these hooks (text masks question_text.ts:830, signature pads, image maps, rating colors, file video state question_file.ts:1196) — each gets a native replacement in its own question task, tracked in the runtime ledger as a per-subtype `native lifecycle` row. Consumer events `onAfterRenderQuestion`/`onAfterRenderQuestionInput`/`onAfterRenderElement` get an RN parity contract in M1 that must NOT pass fabricated `HTMLElement`s (ledger row; DIFFERENCES.md).

## The render→commit gap — resolved by D4

Round 1 held this as accepted-risk; round 2's deterministic repro (see Status) withdrew that. D4's mount reconcile pass closes it. The renderer-internal invariant stays regardless: **renderer components never mutate the model from render/mount lifecycles** (dev-mode warning when `propertyValueChangedHandler` fires during a mount commit) — D4 covers app code; the invariant keeps our own house clean. Test 11 remains instrumented-and-non-probative for residual concurrent interleavings; a reproduced tear post-D4 reopens the `useSyncExternalStore` decision per A3.

## Known sharp edges (encoded as tests)

1. **Leak-by-design**: `dispose()` doesn't clear these subscriptions → unmount MUST unsubscribe. `hasActiveUISubscribers` (base.ts:1467) observes only the property event — leak tests must exercise BOTH scalar and array paths (array cleanup asserted via no-callback-after-unmount, not via that getter).
2. **Post-dispose**: `setPropertyValueCore` still writes the hash post-dispose with the fire skipped (base.ts:794-807); the public scalar `setPropertyValue` ignores post-dispose writes entirely (base.ts:847). Base never assumes dispose = inert; wording scoped to the core setter path.
3. **Array identity**: `ArrayChanges` relies on shadowed mutators on the model's own array instance (base.ts:1493-1589). Base never clones model arrays into state.
4. **StrictMode**: double mount/unmount must not double-subscribe nor leave dangling subscriptions.
5. **Mutation-during-render**: defensive guard, not sanctioned behavior (D2).

## Test plan (TDD; jest + RTL-RN, RN preset; shim via facade; assertions per review round 1)

Fixtures: real `SurveyModel`s via facade + probe subclass exposing instrumentation counters (subscription-callback invocations, `renderElement` entries, setState/`__svRev` bumps — assertions count callbacks/revisions, NOT raw render calls, which StrictMode legitimately doubles).

1. **Mount/subscribe/re-render**: mutate model prop → UI updates; `canUsePropInState` deny-list blocks re-render for filtered keys (callback fires, no `__svRev` bump).
2. **Unmount/unsubscribe**: after unmount, scalar mutation → no callback, `hasActiveUISubscribers === false`; array mutation → no callback (edge 1).
3. **StrictMode**: `<StrictMode>` mount → single mutation produces exactly one `__svRev` bump per observing instance; post-unmount state clean.
4. **Resubscribe on update**: parent re-render, same model → registry unchanged (no unsubscribe/resubscribe churn observable — D1 diffs, unlike upstream's unconditional cycle); subscription still live.
5. **Model swap + partial overlap**: (a) full swap: old model inert, new live, registry exact. (b) `[A,B] → [B,C]`: A unsubscribed, C subscribed, **B never removed/re-added** (callback continuity across the diff asserted — full-swap tests alone don't prove partial-overlap diffing). Registry snapshots are strict-identity, deduplicated.
5b. **Mount reconcile (D4)**: sibling mounted before the probe mutates the model in its own componentDidMount → probe's committed output shows the POST-mutation value (locks the round-2 repro); whole-tree mount produces exactly one reconcile commit (Profiler-counted).
6. **Re-entrancy (defensive)**: force a post-mount update that bypasses nothing (registry intact), have `renderElement()` mutate a model prop → handler reached, suppressed by guard (callback-invocation counter increments, `__svRev` does not), no loop, counter restored.
7. **Shared-model coordination**: two probes on one model; instrument callbacks + convergence in BOTH scenarios: parent re-renders both; only-A re-renders while A's render mutates. Assert eventual convergence and zero-counter restoration (mid-pass internals not asserted via committed output).
8. **Render throw restores guard**: probe whose `renderElement` throws inside an error boundary → `reactRendering` back to 0 for every captured element; other observers keep re-rendering (D2 regression).
9. **Animation policy**: spies assert `enableOnElementRerenderedEvent`, `disableOnElementRerenderedEvent`, and `afterRerender` each receive ZERO calls across mount/update/unmount (`animationAllowed === false` alone doesn't prove non-interference); unmount of one of two observers changes nothing (D3 policy lock).
10. **Array changes**: model-array push/splice → re-render; cloned-array mutation → none (edge 3).
11. **Concurrency (non-probative, instrumented)**: `startTransition` burst mutations across two sibling probes; `Profiler` records commits; assert both siblings converge to the same final value and commit count is bounded. Documented as convergence evidence only — NOT tearing-absence proof. A3 rewrite trigger = deterministic reproduced tear.

Upstream unit coverage of all this: none (verified — one markup-snapshot spec, legacy ReactDOM.render, React 17 toolchain). Suite is written fresh.

## Non-goals

- No hooks/`useSyncExternalStore` (A3; escape hatch = proven tear).
- No afterRender-family calls into core; no fabricated HTMLElements ever.
- No core animation handshake in v1 (D3).
- No freeze/thaw API (D3), no uncontrolled-input support, no factories (0.5).
