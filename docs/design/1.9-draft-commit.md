# Task 1.9 design — text draft/commit adapter (A5)

Status: draft for implementation. Owner: 1.9 agent (CORE, self-contained pure logic). Implements A5; consumed by 1.10 (text) and 1.11 (comment) via the 0.4 reactive base. Source refs: survey-core v2.5.33 `src/question.ts`, `src/question_textbase.ts`, `src/question_text.ts`, `src/question_comment.ts`, `src/utils/text-area.ts`, `src/survey.ts`; survey-react-ui v2.5.33 `src/reactquestion_element.tsx` (`SurveyQuestionUncontrolledElement`), `src/reactquestion_text.tsx`.

## What this replaces

Web renders text inputs **uncontrolled**: the DOM input owns the in-progress text; `SurveyQuestionUncontrolledElement.updateDomElement()` pushes model → DOM after each commit-phase render, and per-event handlers push DOM → model per `textUpdateMode`. RN `TextInput` is used **controlled** (invariant 3), so the "DOM input's own buffer" must be replaced by an explicit **draft** owned by this adapter. The adapter is the state machine; components stay dumb wires (`onChangeText`/`onFocus`/`onBlur`/`onSubmitEditing` → adapter, `value={adapter.renderedValue}`).

`DraftCommitAdapter` is a **pure class**: imports only from `../core/facade` (survey-core types + `Helpers`). No react-native, no React.

## Verified upstream semantics (v2.5.33, file:line)

### Mode resolution
- `SurveyModel.textUpdateMode`: `"onBlur"` (default) | `"onTyping"` (survey.ts:8601-8605); `survey.isUpdateValueTextOnTyping` (survey.ts:4079-4081).
- `QuestionTextBase.textUpdateMode`: `"default"` (inherit survey) | `"onBlur"` | `"onTyping"` per question (question_textbase.ts:74-78).
- Effective flag is **`question.isInputTextUpdate`** = `getIsInputTextUpdate() && isTextValue()` (question.ts:2705-2707). `isTextValue()` is `true` for comment/textbase, but for `text` questions only for date inputTypes + `text`/`number`/`password` (question_text.ts:152-154) — e.g. `inputType:"email"` commits onBlur-only even under `textUpdateMode:"onTyping"`. The adapter therefore reads **`question.isInputTextUpdate` live at each event** (never caches, never re-derives).

### Commit triggers (web)
| Event | onBlur mode | onTyping mode |
|---|---|---|
| React `onChange` (per input event) | only if input NOT focused (autofill/programmatic) (question_text.ts:809-820) | commit |
| `keyup` | Enter (13) commits (question_text.ts:749-764) | commit, unless IME keydown 229 seen → wait for Enter |
| `compositionupdate` | — | commit after `setTimeout(…, 1)` (question_text.ts:741-748) |
| `blur` | commit (question_text.ts:821-826; comment via text-area.ts `onTextAreaBlur` → `onTextAreaChange`) | commit |

**No debounce exists in core.** Commits are synchronous per event; the only timer is the web-only 1 ms composition `setTimeout`. (The docs merely *advise* against `onTyping` with many expressions — question_textbase.ts:72.) The adapter adds no debounce either; hosts wanting throttling set `textUpdateMode:"onBlur"` like on web.

RN mapping: `handleChangeText` ⇔ onChange/keyup/composition (RN `onChangeText` fires once per accepted text change, IME-composed included); `handleSubmitEditing` ⇔ Enter keyup (submit-intent; commits in BOTH modes); `handleBlur` ⇔ blur; `handleFocus` ⇔ focus.

### Commit path (what a commit *is*)
- `text` questions: guarded write `if (!base.isTwoValueEquals(q.value, text)) q.inputValue = text` (question_text.ts:722-728). The `inputValue` setter applies core **mask logic** (unmask → value, re-mask → rendered text; question_text.ts:348-367) — masks come free, no DOM adapter (A5).
- `comment`/textbase: guarded write `if (!Helpers.isTwoValueEquals(q.value, text, false, true, false)) q.value = text` (question_comment.ts getTextAreaOptions:29-33).
- Downstream, `question.value = x` → `setNewValue` → `survey.questionValueChanging` hook → equality/correctness gates → `setQuestionValue` (local, fires `"value"` property change — question.ts:1888-1893 backs `questionValue` with `setPropertyValue("value")`) → `data.setValue(name, v, locNotification, …)` with `locNotification = "text"` when `isInputTextUpdate` (question.ts:2703-2704, 2717-2731).
- `locNotification:"text"` in `survey.updateOnSetValue` (survey.ts:7034-7057): triggers/conditions run, `notifyQuestionOnValueChanged` runs (checkErrorsMode validation, `onValueChanged` event, bindings) — **only `tryGoNextPageAutomatic` is skipped**. So in onTyping mode expressions/validation/events fire per keystroke (exactly web), but per-keystroke commits never auto-advance the page. All inherited free via the model write; zero adapter logic.
- Echo suppression is survey-side: `updateQuestionValue` calls `updateValueFromSurvey` only when the question's value differs (survey.ts:6324-6338) — the committing question already holds the value, so no bounce-back.

### External model changes DURING editing — verified web behavior, mirrored
`SurveyQuestionUncontrolledElement.updateDomElement()` (reactquestion_element.tsx:299-309):

```ts
if (!Helpers.isTwoValueEquals(newValue /* model */, control.value /* DOM */, false, true, false)) {
  control.value = this.getValue(newValue); // isValueEmpty → ""
}
```

It runs on every commit-phase render and **has no focus check**. Comment's `TextAreaModel.onPropertyChangedCallback` (text-area.ts:69-74) likewise writes `element.value` on any `"value"` property change, focused or not. So on web:

- **External write wins ALWAYS — including while the user is mid-typing.** The uncontrolled DOM buffer is overwritten whenever the model-side text no longer loosely-equals it.
- **Self-echo keeps the user's buffer**: after the user's own commit, model text == DOM text under `isTwoValueEquals(…, false, true, false)` (case-sensitive, no trim, numbers↔numeric-strings equal — helpers.ts:188-196), so no overwrite (and no caret jump).
- **Core-transformed commits DO rewrite the buffer** (mask re-format, `questionValueChanging` rewrites, `convertToCorrectValue`): the transformed text differs from what was typed, so web replaces the DOM text — intentional, we keep it.

**Adapter policy (the mirror):** on any subscribed model notification, run exactly that predicate against the draft:

```
if (!Helpers.isTwoValueEquals(modelText, draft, false, true, false)) draft = format(modelText)
```

focused or not. This is `updateDomElement` with `draft ≡ control.value`. No "external wins only when blurred" softening — that would diverge from web (and from the paneldynamic/trigger flows that legitimately rewrite values mid-edit). `format(v)`: `Helpers.isValueEmpty(v) → ""`, else `String(v)` (mirrors `getValue` + DOM string coercion).

Subscription seam: `question.registerFunctionOnPropertiesValueChanged(names, handler, KEY)` — the same public seam core's own `TextAreaModel` uses (text-area.ts:76-78; base.ts:1224). Names: `["value", "_inputValue"]` for text kind (`_inputValue` is a `@property`, question_text.ts:340; it updates *after* the `"value"` notification inside the same synchronous call — question.ts:2834-2843 — so the second firing corrects the masked rendering; handler is idempotent), `["value"]` for comment kind. `dispose()` unregisters by KEY (mirrors `TextAreaModel.dispose`).

### Model-side text read
- text kind: `question.inputValue` (masked rendering — what web's `SurveyQuestionText.getValueCore` override reads, reactquestion_text.tsx:74-77).
- comment kind: `question.value` (what `TextAreaModel.getTextValue` reads).

**Masked questions: the draft lives in MASKED space.** On web, `InputElementAdapter` (question_text.ts:30-36, mask/input_element_adapter.ts) formats the DOM buffer per keystroke, so the text reaching `updateValueOnEvent` is always mask-shaped — raw digits (`"123456"` for pattern `999-999`) are a state web cannot produce (`getUnmaskedValue("123456") === ""`, empirically verified). The adapter therefore commits the draft as-is through the `inputValue` pipeline and expects mask-shaped drafts; **per-keystroke mask formatting of `TextInput` text is 1.10's component concern** (its analog of `InputElementAdapter`, via `maskInstance.processInput`), not 1.9's.

Kind detection: `question.isDescendantOf("text")` → text kind; else (textbase/comment) → value kind. Constructor accepts an explicit override for future wrappers (multipletext items, 2.6).

### Focus/blur side effects (not just commits)
`onFocusCore` arms empty-field validation (`isFocusEmpty = isValidateVisitedEmptyFields`); `onBlurCore` runs `validate(true)` when armed and still empty (question.ts:1639-1652) — plus, for text, date-validation message + counter + the commit itself (question_text.ts:821-829). The adapter calls the **public core handlers** `question.onFocus(e)` / `question.onBlur(e)` with a synthetic `{ target: { value: draft } }` event, committing the draft FIRST so the handler's internal `updateValueOnEvent` no-ops on its equality guard. This keeps validation timing identical to web without duplicating core logic (invariant: never re-implement model-state logic).

Character counter: web updates `updateRemainingCharacterCounter(text)` on every keystroke in both modes (question_text.ts:748,763; question_comment.ts onInput). `handleChangeText` mirrors this (public API on `QuestionTextBase`; guarded `typeof` for safety).

## State machine

```
        construct(question)
              │  draft = format(modelText); subscribe(value[, _inputValue])
              ▼
   ┌──────── IDLE ────────┐   handleFocus: isEditing=true; q.onFocus({target:{value:draft}})
   │ renderedValue=draft  │──────────────────────────────►┌────── EDITING ──────┐
   │ model change → sync  │                               │ renderedValue=draft │
   └──────────▲───────────┘   handleBlur: commit(draft);  │ model change → sync │
              └───────────────q.onBlur({…}); isEditing=false◄────────────────────┘
                                                           │
   EDITING.handleChangeText(t): draft=t; counter(t);       │
       if q.isInputTextUpdate → commit(t)                  │
   EDITING.handleSubmitEditing(): commit(draft)   [both modes — Enter parity]
   ANY.modelNotification: if !eq(modelText, draft) → draft=format(modelText)  [external wins, focused or not]
   dispose(): unregister by KEY
```

- `commit(text)`: per-kind guarded write (see "Commit path"). Re-entrancy-safe: the write synchronously fires the subscription handler; equality/idempotence makes it settle in one pass (a `isCommitting` flag suppresses draft-overwrite only for the *equal* echo — the transformed-echo must still overwrite, so the flag is NOT used to skip sync; sync stays unconditional and predicate-guarded).
- `renderedValue` is always the draft (draft is kept synced while idle, so there is no two-source branch).
- Draft change notifications: constructor accepts `onRenderedValueChange?: () => void`; fired whenever `renderedValue` changes for any reason (typing or sync) so the host component can `setState`. (The 0.4 base's `__svRev` bump also fires on model property changes, but draft-only changes — onBlur-mode typing — are invisible to the model by design.)

### Deliberate divergences from web (documented, intentional)
1. **No `inputType:"color"` `_isValueChanged` guard** (question_text.ts:715-717): a DOM-picker quirk workaround; RN color inputs are a custom fallback (1.10) and must not silently drop commits.
2. **No `_isWaitingForEnter` keyup dance** (IME keyCode 229): web still commits per composition via React `onChange`, so the dance only suppresses the *keyup* duplicate. RN has no per-key events on `TextInput`; `onChangeText` is the single canonical signal.
3. **No 1 ms composition timer**: same reason — `onChangeText` already fires post-composition-update.

### IME/composition safety note
RN `TextInput` composes IME text natively; `onChangeText` delivers composed updates. Two protections keep composition stable under the controlled pattern: (a) self-echo equality means a commit never feeds a *different* `value` prop back mid-composition in onTyping mode (no re-render loop clobbering the composition), and (b) in onBlur mode the draft is never rewritten at all while typing unless a genuine external write arrives — in which case the composition is intentionally abandoned, exactly as web abandons the DOM buffer. Known platform caveat (documented for 1.10, not adapter logic): rapidly changing the `value` prop of a focused Android `TextInput` can reset composition state; since the value prop only changes on commit-equal (no-op) or genuine external/transformed writes, this is the minimal achievable surface and matches web semantics.

## Public shape

```ts
export type DraftCommitKind = 'inputValue' | 'value';
export interface DraftCommitAdapterOptions {
  question: Question;                       // QuestionTextBase-compatible
  kind?: DraftCommitKind;                   // default: isDescendantOf('text') ? 'inputValue' : 'value'
  onRenderedValueChange?: () => void;
}
export class DraftCommitAdapter {
  constructor(options: DraftCommitAdapterOptions);
  get renderedValue(): string;
  get isEditing(): boolean;
  handleChangeText(text: string): void;
  handleFocus(): void;
  handleBlur(): void;
  handleSubmitEditing(): void;
  dispose(): void;
}
```

One adapter per (component instance, question); components recreate on question swap (0.4 registry diff handles the model-swap re-render; adapter `dispose()` in the old pair's cleanup).

## Test plan (TDD — real `Model` + text/comment questions via facade)

1. onBlur default: N× `handleChangeText` → `question.value` unchanged, `survey.onValueChanged` count 0; `renderedValue` tracks draft.
2. `handleBlur` commits: value set, `onValueChanged` count 1; empty-visited validation fires when armed (validateVisitedEmptyFields + checkErrorsMode onValueChanged).
3. `handleSubmitEditing` commits in onBlur mode (Enter parity) — count 1 without blur.
4. onTyping (survey-level): commit per `handleChangeText` — count N; no commit events for equal text (guard).
5. Question-level `textUpdateMode:"onTyping"` overrides survey onBlur; `"onBlur"` overrides survey onTyping.
6. `isTextValue` gate: `inputType:"email"` + onTyping → still blur-commit only.
7. External `survey.setValue`/`question.value=` while NOT editing → draft syncs, `onRenderedValueChange` fired.
8. External write while EDITING (focused, dirty draft) → draft replaced (external wins; the web mirror).
9. Self-echo: onTyping commit does NOT clobber draft (no spurious `onRenderedValueChange` with different text); transformed commit (mask or `onValueChanging` rewrite) DOES update draft to the transformed text.
10. Expressions fire at commit time: dependent `visibleIf`/expression question updates on blur-commit, not on draft keystrokes (onBlur mode).
11. Comment kind: value-write path, carriage-return stripping inherited from model, counter updates.
12. Masked text question: typed digits → committed unmasked value + masked `renderedValue`.
13. `dispose()` unregisters (later external writes don't touch a disposed adapter / no leak via KEY).
14. Empty formatting: undefined/null model → `renderedValue === ""`; numeric value 5 vs draft "5" → no overwrite (Helpers numeric-string equality).
