# Task 2.11 — custom + composite question adapters (ComponentCollection)

Status: **RECONCILED** (three-way: codex sol@max `revise` — "no major
architectural defect", 4 minor + 1 nit; gemini Pro High `approve`. Findings
folded in; see "Three-way reconciliation"). CORE task (A4). Independent of the
in-flight PRs — builds off `main` (depends only on the merged factory/dispatch +
`SurveyPanel` composition + `QuestionChrome`). Weight M (plan row 2.11).

## What these are (verified survey-core 2.5.33, `question_custom.ts`)

`ComponentCollection.Instance.add(json)` registers a runtime component as a REAL
serializer class under its lowercase `name`. Two shapes:
- **CUSTOM** (`questionJSON`/`createQuestion`) → `QuestionCustomModel` — wraps ONE
  inner question.
- **COMPOSITE** (`elementsJSON`/`createElements`) → `QuestionCompositeModel` —
  wraps a PanelModel of inner elements.

Critical dispatch fact: **`getType()` returns the registered NAME** (e.g.
`"full-name"`), but **`getTemplate()` returns `"custom"`/`"composite"`**
(`:797`, `:1112`). Both have no `RendererFactory` renderer, so
`isDefaultRendering()` is true → the renderer dispatches on `getTemplate()`. The
existing `SurveyRowElement` ALREADY computes
`isDefaultRendering() ? getTemplate() : getComponentName()`, so a registered
custom/composite row element already routes to keys `"custom"`/`"composite"` —
this task only needs to REGISTER those two keys (flip the planned descriptor rows
to supported with real components).

## Render strategy — reuse existing infra (mirrors web)

- **Custom** → render the LIVE `question.contentQuestion` (a real `Question`
  built in the model ctor) as an INPUT-ONLY body: dispatch it through
  `RNQuestionFactory.createQuestion(innerKey, { question: contentQuestion,
  creator })` where `innerKey = contentQuestion.isDefaultRendering() ?
  contentQuestion.getTemplate() : contentQuestion.getComponentName()`; fall back
  to `createUnsupportedQuestion` on a miss. NO `QuestionChrome` here — the OUTER
  custom question's row wrapper already renders title/description/errors
  (`SurveyRowElement` → `QuestionChrome`), exactly as web does
  `createQuestionElement(contentQuestion)` (input only).
- **Composite** → render the LIVE `question.contentPanel` (a real `PanelModel`)
  via the EXISTING `SurveyPanel` composition:
  `<SurveyPanel element={contentPanel} survey={question.survey} creator={creator} />`.
  Each inner question then gets its own `QuestionChrome` (title) through the
  normal panel → row → `SurveyRowElement` path; the composite's OWN title comes
  from the outer row wrapper (`showQuestionNumbers="off"` is already set on the
  content panel). Same reuse pattern as paneldynamic (2.8a).

Both render the LIVE instances the model already built — NEVER re-parse
`questionJSON`/`elementsJSON` (the model created `contentQuestion`/`contentPanel`
in its ctor; re-instantiating breaks value proxying + lifecycle).

## Value + lifecycle — drive through the OUTER model (do NOT re-wire inner)

The inner elements' `ISurveyImpl`/`ISurveyData` is the OUTER model
(`setSurveyImpl(this)`), so all inner reads/writes flow through the outer's
`getValue`/`setValue`/converters. The RN components must NOT wire inner inputs
straight to the survey — they already route through the outer because the inner
questions ARE bound to the outer model. So the renderer just renders; the model
owns value aggregation (custom value = inner scalar; composite value = object
keyed by inner names), `onValueChanging` redirects, `runCondition`, readOnly,
visibility, and error cascades. Nothing for the renderer to hand-roll.

## Reactivity

- **CustomQuestion** (`extends QuestionElementBase`): subscribe ONLY the outer
  question (`getStateElement() → questionBase`). The dispatched inner component
  (e.g. `TextQuestion`) subscribes `contentQuestion` through its OWN
  `QuestionElementBase` lineage, so also subscribing it in the wrapper is
  redundant double-rendering (gemini). Value proxies to the outer anyway, so an
  inner change surfaces on the outer's own subscription. (This also sidesteps a
  null `contentQuestion` in `getStateElements()` — see the malformed-content
  guard.)
- **CompositeQuestion**: `SurveyPanel` subscribes the panel + its rows/questions
  itself — the outer needs only its own question subscription.

## Component shape

- `CustomQuestion extends QuestionElementBase`:
  - `getStateElement()` → `questionBase` (outer only, per Reactivity).
  - `renderElement()`: a malformed custom can leave `contentQuestion` null (the
    `createQuestion` CALLBACK form may return null; a bad `questionJSON.type`
    throws inside core before we render). Guard: `contentQuestion == null` →
    render a `custom-question-malformed` fallback View + a distinct diagnostic
    (NOT `createUnsupportedQuestion`, which requires a concrete inner Question).
    Else dispatch the inner input body via the shared
    `resolveQuestionDispatchKey(contentQuestion)` helper (below), falling back to
    `createUnsupportedQuestion({question: contentQuestion, …})` on a factory
    miss. testID `custom-question-<name>`.
- `CompositeQuestion extends QuestionElementBase`:
  - `getStateElement()` → the question (panel handled by SurveyPanel).
  - `renderElement()`: `<SurveyPanel element={contentPanel} …/>` inside a View
    `testID="composite-question-<name>"`, **`role="group"`** (RN 0.86:
    `accessibilityRole` does NOT accept `'group'` — the separate `role` prop maps
    ARIA roles; codex #4), with an `aria-label`/accessibility label from the
    outer `question.processedTitle`. The group owner is the composite wrapper
    ONLY — SurveyPanel does not apply a role, so no duplicate group.

## Shared helper (both peers — avoid drift)

Extract `resolveQuestionDispatchKey(question)` =
`question.isDefaultRendering() ? question.getTemplate() : question.getComponentName()`
into a shared util; use it in BOTH the existing `SurveyRowElement` (which inlines
this today) AND `CustomQuestion`, so the dispatch rule can never drift between
them.

## Registration + wiring

- `descriptors.ts`: flip `custom` + `composite` from planned → supported
  (`route:'template'`, dispatchKey `custom`/`composite`, components
  `() => CustomQuestion` / `() => CompositeQuestion`, milestone M2).
- `manifest.ts`: **NOTHING to flip** (codex #2). custom/composite are NOT in
  `MODEL_TYPE_CLASSIFICATION` — they are `RUNTIME_TEMPLATE_TYPES` exemptions
  (`getTemplate()` runtime templates, not serializer class names), already
  exempt from the class-name gate; keep that set unchanged. Support comes from
  the descriptor rows + the ComponentCollection fixture tests.
- The getAllTypes enumeration lists (register-all/index/import-side-effect-free)
  gain `custom` + `composite` once the descriptor rows register; descriptors.test
  already lists them in the dispatchKey set (they're descriptor rows).
- `index.tsx` export; `api-surface.ts` watchlist rows for the production-read
  accessors: `QuestionCustomModel.contentQuestion`,
  `QuestionCompositeModel.contentPanel` (NOT `ComponentCollection.add/remove` —
  already deliberately excluded; NOT subclass `getTemplate` — the walker accepts
  inherited methods so it wouldn't verify the override semantics; that stays
  BEHAVIORALLY tested).
- kitchen-sink: register a custom (e.g. a wrapped text) + a composite (e.g.
  first/last name panel) via ComponentCollection in the example, render both;
  parity regen. DIFFERENCES: custom renders the inner input only (outer chrome);
  composite renders the inner panel; value shapes documented.

## Test plan (TDD, red first)

Every test that registers a type does so in a `try` and REMOVES it in `finally`
(`ComponentCollection.Instance.remove(name)`), disposing the model too — the
Instance is a GLOBAL singleton and leaks across the suite otherwise (gemini
major + codex #3). Mirror the existing `manifest.test` pattern.

- register a CUSTOM (questionJSON: a text) → renders the inner input (fire row
  layout); typing commits to the OUTER `question.value` (scalar proxy); an
  OUTER-set value flows into the controlled inner input.
- **custom wrapping a renderer-route inner** (questionJSON: `boolean`,
  `renderAs:"checkbox"`) → dispatches via `getComponentName()`, NOT
  `getTemplate()` — proves the full `resolveQuestionDispatchKey` rule (a
  text-only fixture cannot catch a `getTemplate()`-always bug; codex #3).
- register a COMPOSITE (elementsJSON: two texts) → renders both inner inputs via
  SurveyPanel; editing writes the keyed object value `value[innerName]`,
  slot-isolated; an outer-set object distributes into the inner inputs.
- a `createQuestion`/`createElements` CALLBACK fixture renders (not just the
  JSON form).
- converter coverage: `valueToQuestion`/`valueFromQuestion` (custom) applied.
- malformed custom: a `createQuestion` returning null → the
  `custom-question-malformed` fallback + diagnostic, outer survives (no crash).
- dispatch: dispatches on `getTemplate()` (`custom`/`composite`), NOT `getType()`
  (the registered name).
- chrome/numbering: the composite's own title comes from the outer wrapper;
  inner questions render their own titles; `role="group"` present.
- cleanup/leak: unmount detaches subscriptions; `remove()`/`dispose()` in
  `finally` leaves the global registry clean (assert the type is gone after).
- registration; enumeration lists; api-surface (contentQuestion/contentPanel).

## Three-way reconciliation

- **Architecture APPROVED** (codex "no major defect" + gemini approve): dispatch
  on `custom`/`composite`, bare-body custom render, live-instance reuse,
  SurveyPanel for composite, drive value/lifecycle through the outer model.
- **Reactivity (gemini):** CustomQuestion subscribes the outer ONLY (inner
  self-subscribes) — dropped the `[outer, inner]` proposal.
- **Manifest (codex #2):** flip ONLY descriptor rows; custom/composite stay
  `RUNTIME_TEMPLATE_TYPES` exemptions — nothing in `MODEL_TYPE_CLASSIFICATION`.
- **Malformed content (codex #1):** distinct `custom-question-malformed`
  fallback + diagnostic for a null `contentQuestion`.
- **a11y (codex #4, corrects gemini):** RN 0.86 uses `role="group"` (NOT
  `accessibilityRole="group"`) on the composite wrapper only.
- **Shared helper (both):** extract `resolveQuestionDispatchKey`, used by
  SurveyRowElement + CustomQuestion.
- **afterRender limitation (codex nit #5):** "the renderer just renders" covers
  MODEL lifecycle only — ComponentCollection's `onAfterRender`/
  `onAfterRenderContentElement` remain unsupported under the repo-wide
  afterRender ban (document in DIFFERENCES).
